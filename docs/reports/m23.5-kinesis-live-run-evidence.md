## Honest scope notes

- **DescribeStreamSummary read-lag.** Kinesis retention/mode changes propagate to the summary
  read with a few-seconds lag. During test 3 the first revert `apply` read a stale 24h (the
  Increase to 48 had not yet surfaced) and correctly planned `no-op`; once the summary settled at
  48, the revert re-run issued the Decrease and reached 24. Not a handler bug — the handler acts
  faithfully on what it reads; the driver simply polls the summary to a stable value before
  asserting. Recorded so future waves budget for eventual consistency on Kinesis reads.
- **ON_DEMAND posture.** No customer shard management, no CMK (encryption left unpinned = AWS
  default `NONE`, which the desired-gated projection correctly ignores). PROVISIONED shard-count
  scaling was exercised only via mock; the live PROVISIONED touch was the mode conversion, whose
  conversion inherits the on-demand shard count (a follow-up converge to shardCount 1 was not run
  — no new signal for the budget).
- **No mapping emission** for the Stream kind — the run drove the executor directly with a
  hand-built ProviderPlan (same gap as M22.1/M22.2).
- **Parallel wave:** 2 sibling M23.5 service runs (Firehose, OpenSearch) were live in the same
  account/region concurrently, so the region-global `live:sweep` was **NOT** run here — cleanup
  verification was **run-scoped by run id** (`1784351929`); the coordinator runs the global sweep
  after the wave.

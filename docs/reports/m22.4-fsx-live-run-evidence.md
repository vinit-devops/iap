## Honest scope notes

- **OpenZFS-only scope.** The handler implements FSx for OpenZFS single-AZ (SINGLE_AZ_1), the
  cheapest FSx family; LUSTRE / ONTAP / WINDOWS fail closed with an honest unsupported-value error
  (mock-tested). This run exercised only the OPENZFS path.
- **Lean run — replacement execution deferred (test 4).** Classification + closed-gate refusal were
  proven live; the destructive delete+create leg was justified-deferred (cost, ~10-min recreate,
  token-expiry exposure). The full gated replace leg is already proven live in the sibling
  M22.4-EFS evidence.
- **Safe-update choice (test 3).** Backup retention (0↔1) was used rather than throughput, which
  parks the file system optimizing for minutes — recorded as a deliberate lean-run choice, not an
  omission.
- **Default VPC + default SG** (ADR-0005) — first default subnet only (single-AZ); no dedicated
  network. M23.4 replaces this.
- **No mapping emission** for the OpenZFS kind — the run drove the executor directly with a
  hand-built ProviderPlan (same gap as M22.1/M22.2/M22.4-EFS).
- **Fresh clean re-run** after two token-expiry interruptions (`jarvis-1784307761` and an earlier
  attempt); both prior file systems were torn down, so this run started from an empty account.
- **Parallel wave:** five sibling M22.4/M22.x runs were live in the same account/region, so the
  region-global `live:sweep` was NOT run here — cleanup verification was run-scoped by run id

## Honest scope notes

- **Same target type as redis (engine discriminator):** memcached and redis share
  `aws:elasticache:ReplicationGroup`; the desired `engine` attribute selects the API family. A live
  memcached cluster where redis is desired (or vice versa) surfaces as immutable engine drift → gated
  replace, never a blind duplicate create. Delete dispatches on the LIVE engine so a replace tears down
  the correct family.
- **Tag-reconcile-on-update (honest gap):** the handler reconciles the resource's own attributes but does
  not re-assert `iap:*` tags on update. Test 7 restored the stripped tag out-of-band; a converge run does
  not re-add a manually removed managed tag. Same posture as the sibling services.
- **Default VPC (ADR-0005):** the handler placed the cluster in the account default VPC via its own
  subnet group `jarvis-1784338126-mc-subnets` over 3 AZ subnets — the M21.3 ElastiCache live-fix pattern.
  M23.4 Network handlers replace this pragmatism.
- **No mapping emission** for the cache kind — the run drove the executor directly with a hand-built
  ProviderPlan (same gap as M22.1 apigatewayv2 / M22.2 DynamoDB / the DocDB sibling).
- **Cut-order-lean:** single `cache.t4g.micro` node; no multi-node fan-out beyond the test-3 scale.
  Replacement (test 4) executed as refusal-only, justified above.
- **Parallel wave:** 5 sibling M22.3 service runs were live in the same account/region concurrently, so
  the region-global `live:sweep` was NOT run here — cleanup verification was **run-scoped by run id**
  (`1784338126`); the coordinator runs the global sweep after the wave.
- **Clean re-run:** two prior attempts died on token expiry; their resources were torn down before this
  run, which started from zero.

# M23.5 — MSK Serverless (kafka:Cluster): LIVE EVIDENCE — JUSTIFIED DEFERRAL

**Date:** 2026-07-18 · **Status:** DEFERRED at the MSK sub-gate — handler ships **mock-proven**, live
evidence intentionally not executed this wave. **Account:** REDACTED · **Region:** eu-central-1
· **Profile:** `REDACTED_AdministratorAccess`.

This is a deliberate, roadmap-sanctioned deferral, not a failure. ROADMAP-V4 M23.5 states:
*"MSK live evidence is separately gated (long spin-up, cost); deferring it does not block the
milestone if the handler is mock-proven."* MSK Serverless also sits on the roadmap **cut order**
(`SES → AppSync → EKS → Neptune → FSx → MSK live evidence`) as an explicit human-decision cut
under cost/time pressure.

## Why deferred

- **Cost:** MSK Serverless bills ~$0.75/hr per cluster plus ~$0.10/GB-partition-hr and per-GB
  throughput — materially pricier than the other M23.5 targets (Kinesis/Firehose/OpenSearch,
  all run live this wave for cents). A single live pass would be the most expensive resource of
  the entire roadmap to hold.
- **Latency:** cluster create and delete each take ~15–30 minutes to reach a terminal state —
  well beyond the other handlers' live budgets and a poor fit for the parallel-wave protocol
  (a run that cannot finish teardown in-session is a failed run).
- **Budget discipline:** the roadmap running total is ~$4 of the <$25 ceiling. Deferring MSK
  keeps a comfortable margin and honours the requester's cost directive; the live pass can be
  taken at its own sub-gate when explicitly approved.

## What IS proven (mock)

`MskClusterHandler` (`aws:kafka:Cluster`) is fully mock-tested — `packages/deploy-aws/test/m23.5-msk.test.ts`
(10 tests, part of the green deploy-aws suite):

- Serverless-type create (`CreateClusterV2` with a `Serverless` block only — no broker sizing),
  IAM SASL client auth, default-VPC ≥2-distinct-AZ subnets + default SG (ADR-0005), tags.
- Name→generated-ARN resolution via paginated `ListClustersV2` (match on a later page).
- No-op convergence; `clusterType`/VPC-config drift classifies **replace** (Serverless↔Provisioned
  is a different product; VPC config cannot re-home in place).
- `DELETING` → absent; managed-only destroy refusal; destroy by resolved ARN; tag reconcile.
- ≥2-subnet requirement **fails closed** with a clear error when the account default VPC offers
  fewer distinct-AZ subnets.

## How to close this deferral

At the MSK sub-gate, run a single per-service live pass (budget ~15–30 min each way):
create the Serverless cluster on the default VPC → poll to ACTIVE → verify IAM-auth bootstrap
brokers (`GetBootstrapBrokers`) → destroy → poll to gone → run-scoped sweep. Capture the results
in this doc (replacing this deferral section). The handler is ready as-is.

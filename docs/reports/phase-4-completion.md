# Phase 4 Completion Report — Infrastructure Language Server

**Date:** 2026-07-10 · **Milestones:** M4.1–M4.4 (delivered together as `@iap/language-server` 0.1.0; reviewable doc: `docs/milestones/M4-language-server.md`)

## Exit criteria verification

| Exit criterion                                             | Status   | Evidence                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ---------------------------------------------------------- | -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Authoring works in a generic LSP client                    | **Pass** | The server announces only standard LSP 3.17 capabilities (incremental sync, completion, hover, definition, references, rename, documentSymbol, codeAction) — no editor-specific logic exists anywhere in the package. Verified against a raw JSON-RPC stdio client (no editor, no vscode-languageclient): initialize handshake, didOpen → `textDocument/publishDiagnostics` with IAP102 at the correct position, completion over the wire, `iap/preview` + `iap/canonical`, clean shutdown (exit 0). Custom requests are `iap/`-namespaced and simply unused by clients that don't know them. |
| Renames safely update references                           | **Pass** | `computeRename` rewrites the `/resources` key plus every document-derived reference site — relationship `target`s (inline and rule edges), Application `spec.components`, `outputs.*.resource`, `tls.certificate`, and profile `overrides.resources` keys (without which a rename silently breaks the merged document). Tested round-trip: renaming `orders-db` → `orders-database` in `basic-webapp` touches 6 sites and the result re-validates with zero errors; grammar-invalid names and collisions are rejected before any edit is produced.                                            |
| Diagnostics match CLI and SDK                              | **Pass** | By construction, not by comparison: `computeDiagnostics` calls the same `IaPWorkspaceResult.validate()` (phases 1–4 via `@iap/validator`) and `policies()` (phase 5 via `@iap/policy`) that `iap validate` consumes — the server adds source-map positioning only. There is no second validation engine to drift. Tested: conformance case 01 surfaces its registry code (IAP102) as a diagnostic at the documented position; `basic-webapp` is clean in both.                                                                                                                                |
| Large documents remain responsive within documented limits | **Pass** | Measured (5 cold runs each, cache defeated per run): full diagnostics on the largest official example (`enterprise-pci`) **avg 16.6 ms** (max 19.0); on a synthetic **110-resource** document **avg 31.0 ms** (max 33.0) — ~6× inside the ch. 23 §23.4 target of 200 ms for documents under 100 resources. Debounce 150 ms; stale results are dropped when a newer document version supersedes a pass before or during computation. Caching model: single-entry per-version memoized SDK workspace (documented in M4).                                                                        |
| No cloud credentials required for baseline editing         | **Pass** | The package performs no network I/O whatsoever — its runtime inputs are document text over stdio and the schema embedded in `@iap/model`. Dependencies are the workspace engines plus `vscode-languageserver`/`-textdocument` (protocol plumbing). Nothing reads environment credentials, config files, or endpoints; grep-clean of `fetch`/`http`/`net` usage.                                                                                                                                                                                                                               |

## Deliverables checklist (roadmap phase 4)

- **LSP server** ✓ — `@iap/language-server` with the `iap-language-server` stdio bin; pure provider core + thin protocol binding.
- **Protocol extension specification** ✓ — `iap/preview` `{uri, view, application?} → {mermaid}` and `iap/canonical` `{uri} → {canonicalJson, hash}`, specified with error semantics in `docs/milestones/M4-language-server.md`; the `iap/` namespace is reserved per ch. 23 §23.3. Plan preview joins in Phase 7 when a plan format exists (IEP-0011).
- **Editor-neutral integration tests** ✓ — 31 provider-core tests run every capability as pure functions with zero protocol involvement; a raw JSON-RPC smoke test covers the wire.
- **Performance benchmarks** ✓ — two measured, logged benchmarks in the suite (soft-asserted against 1000 ms for CI stability; actuals recorded above).
- **Caching model** ✓ — per-version memoized SDK analysis (single entry, keyed on exact text) over the SDK's own memoized `validate`/`canonical`/`policies`.

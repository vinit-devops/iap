## Real bugs caught by live execution (mock tests could not)

1. **NAT teardown released the EIP while the gateway was still `deleting`.** During the live
   destroy, `NatGatewayHandler.waitForNatDeleted` treated the NAT as done as soon as it left
   `pending`/`available` (i.e. entered `deleting`), then called `ReleaseAddress`. A NAT in
   `deleting` still holds its EIP, so AWS rejected the release — verbatim:
   `You do not have permission to access the specified resource.` — which cascaded and blocked the
   pub-subnet / IGW / VPC deletes (all depend on the NAT being fully gone). Mock tests could not
   catch this: they returned a single post-delete `deleted` describe, never the real
   `available → deleting → deleted` transition.
   **Cause:** the waiter's terminal condition used `LIVE_NAT_STATES = ['pending','available']`, so
   `deleting` counted as "done".
   **Fix:** added `NAT_EIP_BOUND_STATES = ['pending','available','deleting']` and made the waiter
   poll until none of those remain (only `deleted`/`failed`/absent frees the EIP).
   File: `packages/deploy-aws/src/network-routing.ts`.
   **Regression test:** `packages/deploy-aws/test/m23.4-network-routing.test.ts` — "waiter treats
   `deleting` as EIP-bound: releases the EIP only after `deleted`" (fake-timer driven; asserts the
   waiter re-describes through `deleting`→`deleted` before the single `ReleaseAddress`). Rebuilt;
   `pnpm vitest run packages/deploy-aws` → **416 passed**.

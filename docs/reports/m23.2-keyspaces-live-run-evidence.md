## Real bugs caught by live execution (mock tests could not)

Both are genuine live handler bugs; fixed under the live-bug protocol (two files only:
`packages/deploy-aws/src/keyspaces.ts` + `packages/deploy-aws/test/m23.2-keyspaces.test.ts`), rebuilt,
full package suite green (**364 tests, 38 files**), regression tests added.

1. **`UpdateTable` sent more than one custom property → hard rejection.** Enabling PITR alone failed
   with the verbatim error:

   > `Changing more than one custom property is not supported. Tried changing 2`

   Cause: the handler's `update()` always sent BOTH `capacitySpecification` AND `pointInTimeRecovery`
   on every call; Amazon Keyspaces counts each present property as a change and allows only ONE per
   `UpdateTable`. The mock could not catch it — the mocked `UpdateTableCommand` resolves regardless of
   input. **Fix:** compare desired vs. live projection and send only the properties that actually
   drifted, one `UpdateTable` per property. Regression tests: a pitr-only change now issues exactly one
   call carrying only `pointInTimeRecovery` (capacity NOT resent), and a capacity+pitr change issues two
   calls, each with exactly one custom property.

2. **Trailing `TagResource` raced the just-issued `UpdateTable`.** After an `UpdateTable` the table
   enters `UPDATING`, and Keyspaces then rejects the immediate tag reconcile:

   > `infraasprompt_1784345753_ks.infraasprompt_1784345753_tbl is currently being created, altered or deleted.`

   Timing-dependent (test 3 happened to pass; the test-5 reconcile failed). **Fix:** reconcile tags
   FIRST, while the table is still `ACTIVE`, then issue the property `UpdateTable`(s).

## Honest scope notes

- **Async create/delete, operator polls (findings A & B).** Keyspaces `CreateTable`/`DeleteTable`/
  `DeleteKeyspace` are asynchronous and the handler keeps the repo's documented "no waiter" idiom
  (the driver verifies readiness). Two consequences surfaced live and are **orchestration**, not
  handler bugs (the executor's replace/destroy do delete-then-next in one pass; `executor.ts` is
  outside the two-file bug scope, and the runbook's destroy test prescribes operator polling — the
  M22.3 DocDB precedent):
  - **A (replace):** `CreateTable` immediately after `DeleteTable` hit
    `Cannot add already existing table infraasprompt_1784345753_tbl to keyspace infraasprompt_1784345753_ks`; a
    second apply pass (after the delete settled) completed the recreate.
  - **B (destroy):** `DeleteKeyspace` immediately after `DeleteTable` hit
    `infraasprompt_1784345753_ks or a table under it is currently being created or deleted.`; polling the
    table gone, then a second destroy pass, deleted the keyspace cleanly.
    A future improvement would add a bounded ACTIVE/absent waiter between sequential mutations (also
    needed for multi-property `UpdateTable`, where consecutive calls must each wait for `ACTIVE`).
- **CQL underscore naming.** Keyspaces/table names must be valid CQL identifiers, so hyphens are
  disallowed — the run id is `infraasprompt_1784345753_ks` / `_tbl`. Run-scoping is therefore by the
  `1784345753` substring in `iap:*` tag values, verified present.
- **On-demand only.** `PAY_PER_REQUEST` throughput; no provisioned capacity, no customer KMS key
  (AWS-owned encryption), no data rows written (DDL/control-plane only).
- **No mapping emission** for the wide-column-Database kind — executor driven directly with a
  hand-built ProviderPlan (same gap as M22.1/M22.2/M22.3).
- **Parallel wave.** 5 sibling M23.2 service runs were live in the same account/region concurrently, so
  the region-global `live:sweep` was NOT run here — cleanup verification was **run-scoped by run id**
  (`1784345753`); the coordinator runs the global sweep at wave closeout.

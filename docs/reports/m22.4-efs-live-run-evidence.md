## Real bugs caught by live execution (mock tests could not)

1. **Mount-target lifecycle race with a non-convergeable partial create (FIXED).** The handler
   issued CreateMountTarget immediately after CreateFileSystem; live, a fresh file system is
   `creating` for a few seconds, so the call always failed — surfaced verbatim as
   `"infraasprompt-1784289034-fs.aws:efs:FileSystem: UnknownError"` (EFS's rejection deserializes
   poorly in the SDK). Worse, the partial create could NOT converge: the file system existed
   and matched its projection, so re-plan classified `no-op` ("present and converged") with
   zero mount targets, forever. Proof of cause: the identical CreateMountTarget parameters
   succeeded once the file system reached `available`. Fix: `create()` now runs a bounded
   fail-closed waiter (2s × 60 = 2-min budget) for LifeCycleState `available` before
   CreateMountTarget — `packages/deploy-aws/src/efs.ts`; regression test asserting the
   mount-target call lands strictly after a describe that saw `available` —
   `packages/deploy-aws/test/m22.4-efs.test.ts`. Package suite green (298 tests) after the fix;
   the post-fix live create applied cleanly in ~8s.
2. **elastic+maxIO is an invalid AWS combination — and replace deletes before it finds out
   (recorded).** CreateFileSystem rejected the requested replacement verbatim: "Elastic
   throughput is not available for file systems using MaxIO performance mode." The replace
   path (delete+create in one pass) had already deleted the old file system, leaving the
   resource absent — fail-closed (recorded in `errors[]`, exit 3, no silent success) and
   convergeable by re-apply with a valid combination, but pre-create cross-attribute
   validation in the planner would have refused before destroying. Left as a recorded planner
   improvement, not patched here (not a handler defect; the executor behaved to spec).
3. **CreationToken replacement race (not a bug — the M22.2 DynamoDB idiom).** EFS holds the
   CreationToken while the old file system is `deleting`, so the reverse replacement's create
   leg failed with "File system '…' already exists with creation token 'infraasprompt-1784289034-fs'".
   The token released in under 15 seconds, describe read the dying fs as absent, and re-apply
   converged as `create` — exactly the delete-then-create retry behaviour the handlers document.

## Honest scope notes

- **One mount target only** — the handler owns a single mount target in the FIRST default
  subnet, so NFS access is single-AZ; multi-AZ mount targets are out of scope for M22.4.
- **Default VPC + default SG** (ADR-0005) — no dedicated network; M23.4 replaces this.
- **No mapping emission** for the elastic-NFS kind yet — this run drove the executor directly
  with a hand-built ProviderPlan (same gap as M22.1/M22.2 runs).
- **Parallel wave:** sibling M22.4/M22.x runs were live in the same account/region, so the
  region-global `live:sweep` was NOT run here — cleanup verification was run-scoped by run id;
  the coordinator runs the global sweep after the wave.
- The documented 24h throughput-mode cooldown never fired across three throughput changes on
  the same run (two on one fs, minutes apart) — recorded as observed, not assumed absent.
- Out-of-band UpdateFileSystem is asynchronous (`updating` for a few seconds) — a drift plan
  run inside that window truthfully reads the pre-change value.

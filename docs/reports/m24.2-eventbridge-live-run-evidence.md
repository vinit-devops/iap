## Real bugs caught by live execution (mock tests could not)

No code defect — the handler behaved to spec on real AWS (bus-before-rule ordering, empty-bus
convergence, in-place state/pattern updates both directions, drift read-only + reconcile, managed-only
destroy refusal with exit 3, targets-before-DeleteRule teardown reversed to rule-before-bus). One
**test-fidelity finding** the mock could not surface, recorded honestly (no fix applied — the code is
correct defensive design):

1. **`eventBusName`-drift `replace` is structurally unobservable against real AWS.** The handler's
   `read()` is bus-scoped — it calls `DescribeRule({ Name, EventBusName: <desired bus> })`. AWS rules
   are physically partitioned per bus, so a rule "moved" to another bus is simply **absent** on the
   queried target bus (`ResourceNotFoundException`) and classifies `create`, while the original rule is
   orphaned on the source bus. AWS never returns a rule whose `EventBusName` differs from the bus you
   queried. The mock test (`test/m24.2-eventbridge.test.ts:139`) constructs exactly that
   physically-impossible response (DescribeRule on `app-bus` returning a rule reporting `other-bus`) to
   drive the `replace` classification — a shape real AWS cannot produce. The immutable-key safety net +
   the gate-closed refusal are still correct and valuable defensive design; they are just unreachable
   via live reads for this resource type. Verbatim refusals the `replace` path WOULD emit (from
   `packages/deploy-aws/src/executor.ts`): gate-closed —
   `refusing to replace <logicalId>: immutable attribute drift requires delete+create (destructive); re-run with the replacement gate open`;
   managed-only —
   `refusing to replace <logicalId>: not tagged iap:managed=true (managed-only replace)`.

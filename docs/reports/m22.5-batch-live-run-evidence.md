## Real bugs caught by live execution (mock tests could not)

1. **Disable-then-delete self-race (JQ + CE).** First destroy pass, verbatim on both:
   `Cannot delete, resource is being modified.` Cause: the handlers issued
   UpdateJobQueue/UpdateComputeEnvironment `State=DISABLED` and the Delete immediately after;
   live Batch holds the resource in status UPDATING while the disable lands and rejects a
   delete mid-modification — so a single destroy pass could **never** delete an ENABLED JQ/CE.
   Mocks resolve instantly and cannot surface this. Fix (`packages/deploy-aws/src/batch.ts`):
   disable-**settle**-delete — after the disable, poll Describe\* until state DISABLED and
   status out of UPDATING (120 s budget, fail-closed timeout; already-DELETING skips the
   delete), then delete. Regression tests:
   `packages/deploy-aws/test/m22.5-batch.test.ts` ("delete WAITS for the disable to settle…",
   CE + JQ). Package suite green after fix: 32 files / 301 tests.

Not bugs, recorded honestly (both are the DynamoDB-race idiom — async AWS transitions
surfacing as recorded errors that a re-apply converges): (a) JQ create while the CE was
CREATING (test 1); (b) CE delete while its JQ was async-DELETING (test 8) — the executor
orders JQ before CE but does not wait between resources; a cross-resource drain waiter is a
possible future ergonomics improvement.

## KMS-special cleanup semantics (coordinator: read before the global sweep)

A KMS key **cannot disappear**: AWS forbids immediate deletion. "Deleted" for KMS means
`KeyState=PendingDeletion` with a DeletionDate ≈ now+7d **and** the alias (our identity) gone
from `list-aliases`. Both keys from the replacement — old symmetric `3102e57b-…` and final RSA
`da114047-…` — are PendingDeletion; the alias is gone; the handler reads the resource as absent
and never resurrects a pending-deletion key (no CancelKeyDeletion, ever).

**Tagging-index caveat:** the run-scoped
`resourcegroupstaggingapi get-resources --tag-filters Key=iap:managed,Values=true` check still
returns the two pending-deletion key ARNs (filtered to `iap:planId=infraasprompt-1784279606`). This is
the AWS-mandated terminal state, **NOT an orphan condition** — the wave's global sweep must
treat `KeyState=PendingDeletion` KMS entries as deleted, or KMS runs will false-positive
forever (well, for 7 days). `pnpm run live:sweep` was NOT run by this service run: it is
region-global and sibling live runs (DynamoDB, Timestream, Backup) were executing in parallel;
the coordinator sweeps after the wave.

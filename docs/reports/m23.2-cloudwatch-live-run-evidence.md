## Workload

`aws:cloudwatch:Alarm` + `aws:cloudwatch:Dashboard` applied via the executor with a hand-built
ProviderPlan (`--confirm` equivalent = `apply: true`). The alarm is created with `PutMetricAlarm` (a
full idempotent upsert — create and update issue the same call) and tagged
`iap:managed=true` / `iap:planId` / `iap:resourceId` via `TagResource`. The dashboard is created with
`PutDashboard`; because classic dashboards support no tags, the handler injects a text marker widget
(`iap:managed=true iap:resourceId=jarvis-1784345752-dash`) as the first widget in the body and derives
managed-ness from that marker on read. The marker is stripped before drift comparison so it never
registers as body drift. The two resources are independent; the executor orders them alphabetically and
reverses on destroy.

## Honest scope notes

- **Dashboard managed gate is a marker widget, not a tag.** Classic CloudWatch dashboards support no
  tags on `PutDashboard`, so ownership is asserted by an embedded `iap:managed=true` text widget parsed
  on read. This is honest but weaker than a tag: a human editing the dashboard body in the console could
  strip the marker (exactly the tamper simulated in test 6) and thereby orphan it from the runtime. The
  cleanup name-prefix `list-dashboards` sweep is the backstop for that untaggable surface.
- **Placeholder alarm metric.** The abstract Alert kind carries no metric vocabulary yet, so the alarm
  fires on a synthetic `IaP/Placeholder Heartbeat` metric with default thresholds. A valid, harmless
  alarm — but it monitors nothing real. Real metric wiring is deferred to the Alert grammar work.
- **No mapping emission** for the Alert/Dashboard kinds — the run drove the executor directly with a
  hand-built ProviderPlan (same gap as M22.1/M22.2/M22.3).
- **Free-tier resources.** Alarms (first 10/month free) and classic dashboards (≤3 free) sat well inside
  the free tier; effective cost ~$0.
- **Parallel wave:** 5 sibling M23.2 service runs were live in the same account/region concurrently, so
  the region-global `live:sweep` was NOT run here — cleanup was **run-scoped by run id**
  (`jarvis-1784345752`) via tag filter + dashboard name prefix. The coordinator runs the global sweep at
  wave closeout.

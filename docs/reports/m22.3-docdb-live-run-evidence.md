## Honest scope notes

- **Default VPC (ADR-0005):** the handler placed the cluster in the account default VPC via its own
  subnet group `jarvis-1784307756-db-subnets` over 3 AZ subnets — the M21.3 ElastiCache live-fix pattern.
  M23.4 Network handlers replace this pragmatism.
- **No mapping emission** for the document-Database kind — the run drove the executor directly with a
  hand-built ProviderPlan (same gap as M22.1 apigatewayv2 / M22.2 DynamoDB).
- **Cut-order-lean:** single cluster + single `db.t4g.medium` instance; no read replicas, no multi-AZ
  fan-out, no customer-KMS key (StorageEncrypted uses the AWS-owned default docdb key). Replacement
  (test 4) executed as refusal-only, justified above.
- **Master password:** locally generated, write-only. Honestly noted: it is unrecoverable by design —
  never surfaced in any report, projection, log, or AWS read. A production posture would rotate it into
  Secrets Manager; this run does not, matching the ssm-parameter write-only idiom.
- **Parallel wave:** 7 sibling M22.3 service runs were live in the same account/region concurrently, so
  the region-global `live:sweep` was NOT run here — cleanup verification was **run-scoped by run id**
  (`1784307756`); the coordinator runs the global sweep after the wave.
- **Clean re-run:** a prior attempt was interrupted by a session limit; its resources were torn down
  before this run, which started from zero.

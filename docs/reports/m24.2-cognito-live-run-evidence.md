## Workload

`aws:cognito-idp:UserPool` + `aws:cognito-idp:UserPoolClient` applied via the executor with a
hand-built ProviderPlan (`--confirm` equivalent = `apply: true`). The pool carries
`iap:managed=true` / `iap:planId` / `iap:resourceId`; **clients are untaggable**, so the client's
managed-ness is inherited from the parent pool (route53 record-set idiom). The executor orders
topologically (pool before client on create; reversed — client before pool — on destroy). No mapping
emission for the Identity user-directory kind yet (grammar gap, deferred like M22.1/M22.2/M22.3).

## Honest scope notes

- **Identity kind = user-directory (spec 1.3.0):** a user pool + one app client; no hosted UI,
  identity providers, resource servers, user import, Lambda triggers, or a Cognito identity pool
  (federated identities) — those are out of scope for this run.
- **Clients are untaggable → parent-pool managed gate:** a client carries no `iap:*` tags of its own;
  ownership is inherited from the pool's tags (test 6 proves the inheritance in both the refusal and
  the "still alive" checks). This is the route53 record-set idiom.
- **Write-only client secret:** honestly noted — when `generateSecret=true` the secret is generated by
  Cognito and is unrecoverable through this runtime by design; it is never projected, logged, stored,
  or returned as an identifier (the ClientId is). A production posture would rotate it into Secrets
  Manager; this run does not, matching the docdb/ssm-parameter write-only idiom.
- **No mapping emission** for the Identity user-directory kind — the run drove the executor directly
  with a hand-built ProviderPlan (same gap as M22.1 apigatewayv2 / M22.2 DynamoDB / M22.3 DocumentDB).
- **Safe-update field choice:** `passwordMinimumLength` was used for the in-place update (not
  `mfaConfiguration`), because flipping MFA to ON/OPTIONAL requires an SMS/TOTP second-factor config
  the pool does not have — a password-policy change is the clean, config-free in-place field.
- **Free tier:** 0 monthly active users, so the pool + client are $0 (Cognito bills per MAU).
- **Parallel wave:** 2 sibling M24.2 service runs (CloudFront, EventBridge) were live in the same
  account/region concurrently, so the region-global `live:sweep` was NOT run here — cleanup
  verification was **run-scoped by run id** (`1784356902`); the coordinator runs the global sweep
  after the wave.

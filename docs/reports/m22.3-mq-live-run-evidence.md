## Real bugs caught by live execution (mock tests could not)

Two real handler bugs — both idempotency failures the mock hid because it echoed idealised
DescribeBroker fields. Both fixed under the live bug protocol (own files: `src/mq.ts` +
`test/m22.3-mq.test.ts`), rebuilt, full deploy-aws suite green (303 tests).

1. **engineType case mismatch → spurious destructive `replace`.** CreateBroker takes the SDK enum
   `ACTIVEMQ`, but DescribeBroker returns `ActiveMQ`. The handler compared the desired `ACTIVEMQ`
   against the live `ActiveMQ` on the **immutable** `engineType` key, so a freshly converged broker
   classified `replace` — meaning every re-run proposed a destructive delete+create (test 2 failed
   before the fix). Verbatim first observation: plan `replace` / `refusing to replace … immutable
attribute drift requires delete+create`. **Fix:** upper-case `engineType` on both sides
   (`desiredProjection` + `read`). Regression test:
   `packages/deploy-aws/test/m22.3-mq.test.ts` ("engine casing is normalised … → no-op, not a spurious
   replace").

2. **Accepted instance-type update never converged (plan never went quiet).** An accepted UpdateBroker
   `HostInstanceType` change lands in `PendingHostInstanceType`; DescribeBroker keeps the OLD
   `HostInstanceType` until the maintenance window. The handler's `read()` ignored the pending field,
   so after an accepted resize the plan re-classified `update` on every run and re-issued UpdateBroker
   indefinitely — contradicting the handler's documented contract ("the projection reads the accepted
   desired values, so the plan goes quiet after the call"). **Fix:** project
   `PendingHostInstanceType ?? HostInstanceType`. Regression test: same file ("an accepted instance-type
   update lands in PendingHostInstanceType → plan goes quiet").

## Honest scope notes

- **Maintenance-window update semantics.** Mutable changes (`instanceType`,
  `autoMinorVersionUpgrade`) are accepted by UpdateBroker immediately but the live broker converges only
  at the next maintenance window (or reboot). For `instanceType` AWS exposes the accepted value under
  `PendingHostInstanceType`; the handler now reads it so the plan goes quiet (bug #2). Every accepted
  resize in this run was reverted before any window, so the broker ran mq.t3.micro end to end.
- **autoMinorVersionUpgrade true→false is not achievable on this engine.** AWS auto-selected ActiveMQ
  **5.19** at create, and that version rejects `autoMinorVersionUpgrade=false`:
  `Brokers on [ActiveMQ] version [5.19] must have [autoMinorVersionUpgrade] set to [true].` The handler
  correctly issued UpdateBroker and **failed closed** (recorded in `errors[]`, `applied:false`, exit 3,
  broker untouched) — a valid live finding, not a handler bug. Tests 3 and 5 therefore used
  `instanceType` (the other mutable) instead of the runbook's suggested autoMinorVersionUpgrade flip.
- **Replacement (test 4) executed as refusal-only,** justified above (DynamoDB carried the live
  destructive-replacement mandate in M22.2).
- **Default VPC (ADR-0005):** SINGLE_INSTANCE broker placed on the first default-VPC subnet + default
  SG, `PubliclyAccessible false`. M23.4 Network handlers replace this pragmatism.
- **No mapping emission** for the mq kind — the run drove the executor directly with a hand-built
  ProviderPlan (same gap as M22.1/M22.2).
- **Write-only admin credential:** locally generated (base64url, ≥12 chars), passed once to CreateBroker,
  unrecoverable by design — never surfaced in any report, projection, log, or AWS read. A production
  posture would rotate it into Secrets Manager; this run does not, matching the ssm-parameter idiom.
- **Parallel wave:** 5 sibling M22.3 service runs were live concurrently, so the region-global
  `live:sweep` was NOT run here — cleanup verification was **run-scoped by run id** (`1784338125`); the
  coordinator runs the global sweep at wave closeout.
- **Clean re-run:** two prior attempts died on token expiry; the earlier broker was torn down before
  this run, which started from zero.

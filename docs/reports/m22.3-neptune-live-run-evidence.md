## Real bugs caught by live execution (mock tests could not)

None. The handler behaved to spec end to end on real AWS: subnet-group-before-cluster ordering, the
deliberate absence of master credentials (Neptune IAM/none auth — `MasterUserSecret: null`,
`IAMDatabaseAuthenticationEnabled: false`), in-place retention updates, the immutable-engine replace
classification + gate-closed refusal, drift read-only + reconcile, managed-only destroy refusal, and
the bounded instance→cluster→subnet-group teardown. One honest observation, not a bug:
`describe-db-clusters` reports `MasterUsername: admin` even though the handler sends no `MasterUsername`
— this is a Neptune service-side default surfaced in the describe payload; no password or
`MasterUserSecret` exists (both null/absent), so no credential material is generated, projected, or
stored by IaP. The 13 mock tests already covered these paths; the live run confirmed the same
behaviour against the service, including the multi-minute create/delete waiters the mock cannot
exercise.

## Honest scope notes

- **Default VPC (ADR-0005):** the handler placed the cluster in the account default VPC
  (`vpc-033b2dc4f769d7f77`) via its own subnet group `jarvis-1784338123-db-subnets` over 3 AZ subnets —
  the M21.3 ElastiCache live-fix pattern. M23.4 Network handlers replace this pragmatism.
- **No mapping emission** for the graph-Database kind — the run drove the executor directly with a
  hand-built ProviderPlan (same gap as M22.1 apigatewayv2 / M22.2 DynamoDB / M22.3 DocumentDB).
- **Cut-order-lean:** single cluster + single `db.t4g.medium` instance; no read replicas, no multi-AZ
  fan-out, no customer-KMS key (StorageEncrypted uses the AWS-owned default key). Replacement (test 4)
  executed as refusal-only, justified above.
- **No credential material:** Neptune authenticates via IAM/none; the handler deliberately omits
  MasterUsername/MasterUserPassword. Nothing to rotate, store, or leak — the strongest posture of the
  M22.3 database trio (Aurora/DocDB carry generated master passwords; Neptune carries none).
- **Two-invocation destroy:** the cluster+instance deleters exceed the 10-min foreground shell ceiling,
  so destroy ran across an interrupted first pass + an idempotent background re-run. This is the
  handler's designed convergence behaviour (a deleting/absent resource reads as absent → the next run
  continues the reversed teardown), not a failure — end state is fully torn down.
- **Wall-clock:** ~44 min cluster lifetime (01:31:24Z create → 02:15:20Z gone), longer than the lean
  ideal because of the ~9-min instance-provision wait and an inter-turn gap before teardown began; the
  live cost driver (the single instance-hour) is unchanged.
- **Parallel wave:** 5 sibling M22.3 service runs were live in the same account/region concurrently, so
  the region-global `live:sweep` was NOT run here — cleanup verification was **run-scoped by run id**
  (`1784338123`); the coordinator runs the global sweep after the wave.
- **Clean re-run:** the two prior attempts were interrupted by token expiry; their resources were torn
  down before this run, which started from a verified-empty baseline.

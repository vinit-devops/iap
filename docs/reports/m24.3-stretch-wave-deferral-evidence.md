# M24.3 — Stretch wave (EKS · AppSync · SES): DEFERRAL EVIDENCE

**Date:** 2026-07-18 · **Status:** ALL THREE DEFERRED at their per-item gates — recorded with
rationale, per ROADMAP-V4 M24.3. **Account:** REDACTED · **Region:** eu-central-1
· **Profile:** `REDACTED_AdministratorAccess`.

M24.3 is explicitly a per-item, individually-gated stretch wave: *"EKS, AppSync, SES each get
their own go/defer decision. Pre-declared defer candidates; deferring any or all does not block
M24.4."* All three are the first entries in the roadmap **cut order**
(`SES → AppSync → EKS → Neptune → FSx → MSK live evidence`). Under the standing cost directive
(running total ~$4 of the <$25 ceiling) and the autonomous-completion guardrails, each is
deferred here with rationale; none blocks the M24.4 closeout.

## SES — DEFERRED (kind rejected upstream)

**Decision: N/A — no handler, no live run.** M24.1 / IEP-0017 formally **rejected** the Email
kind (email sending is a messaging-verb/integration concern, not an infrastructure resource kind
with a stable desired-state surface — model it over `Function`/`Service`/`EventBus`/`Topic`/`Queue`
plus provider extensions). With no Email kind in the specification, there is no abstract kind for
SES to realize, so there is deliberately no `aws:ses:*` handler and no live evidence to produce.
This is a closed decision (revisitable only by a future spec minor that accepts an Email kind).

## EKS — DEFERRED (cost + wall-clock; cut order)

**Decision: defer live evidence.** EKS realizes `Service.runtime = kubernetes` (the enum value
added in 1.3.0). Deferred because:
- **Cost:** an EKS control plane bills **~$0.10/hr** regardless of workload — the most expensive
  standing control plane in the candidate set — plus any node group.
- **Wall-clock:** cluster create ~10–15 min and delete ~10–15 min; a managed node group adds
  more. A lean live pass is ~30 min of the priciest resource.
- **Pull-through:** low for the IaP runtime-infra target (Kubernetes is an orchestration layer,
  not a primitive resource); it sits high on the cut order.
The handler is **not implemented** (deferred item, not undertaken). When the EKS gate opens, a
`aws:eks:Cluster` handler (control plane on the default-VPC subnets, IAM cluster role, optional
node group) would be added mock-tested first, then a single gated live pass (budget ~30 min +
the ~$0.10/hr control-plane cost).

## AppSync — DEFERRED (gated; low pull-through)

**Decision: defer live evidence.** AppSync realizes `Gateway.protocol = graphql` (the new
optional Gateway field added in 1.3.0). Deferred because it is an individually-gated stretch item
on the cut order with low runtime-infra pull-through, and closing it adds a GraphQL API + schema +
data-source/resolver wiring that is out of proportion to the remaining budget/time. AppSync itself
is near-free at idle (pay-per-request), so the deferral is about scope/priority, not cost. The
`aws:appsync:GraphqlApi` handler is **not implemented** (deferred item). When its gate opens, the
handler + a lean live pass (create API → verify endpoint → destroy) would be added.

## Net

Zero resources created; **$0.00**; nothing to tear down. All three deferrals are recorded at their
gates with rationale, satisfying M24.3's exit criterion for deferred items. Running roadmap total
unchanged (~$4 / $25). These do not block M24.4.

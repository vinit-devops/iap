**Real bug caught by live execution (mock tests could not):**

1. **RunInstances pinned the first default-subnet AZ and failed outright on capacity.** First
   apply, verbatim: "We currently do not have sufficient t4g.nano capacity in the Availability
   Zone you requested (eu-central-1a). Our system will be working on provisioning additional
   capacity. You can currently get t4g.nano capacity by not specifying an Availability Zone in
   your request or choosing eu-central-1b, eu-central-1c." Cause: `create` took
   `defaultSubnetIds(ec2, 1)` — always the alphabetically-first AZ — with no fallback, so a
   transient single-AZ capacity shortage (real in eu-central-1a for t4g.nano on run day) made
   creation impossible even though sibling default subnets had capacity. Fix
   (`packages/deploy-aws/src/ec2-instance.ts`): try each default subnet's AZ in deterministic
   order, advancing ONLY on capacity errors (`InsufficientInstanceCapacity`/`Unsupported`);
   any other launch error still fails closed at once, and exhaustion of all AZs surfaces the
   real AWS error. Regression tests:
   `packages/deploy-aws/test/m22.5-ec2-instance.test.ts` ("AZ capacity failover" — failover,
   all-AZs-exhausted fail-closed, non-capacity errors do not failover). Package suite green
   after the fix (297 tests / 32 files). Test 1's instance landing in eu-central-1b is the
   live proof.

**Honest scope notes:** `instanceType` replace is a deliberate **fail-closed v1 posture** —
AWS allows an in-place ModifyInstanceAttribute while the instance is stopped, but v1 does not
orchestrate stop→modify→start, so the drift classifies as gated replace (an explicit gated
delete+create instead of an implicit outage). Terminated instances **linger in
DescribeInstances** as `terminated` for up to ~an hour — the handler's live-state filter
excludes them (terminated = deleted, honestly); the run-scoped tagging-API check happened to
return 0 ARNs at verification time, but that index can also lag and listing a terminated
instance ARN there would not be an orphan. Default VPC/SG placement per ADR-0005 (M23.4
replaces this). No mapping emission for the Compute kinds yet — the run drove the executor
directly with a hand-built ProviderPlan (same gap as M22.1/M22.2). Sibling M22.5 runs were
live in the same account/region, so the region-global `live:sweep` was NOT run here — cleanup
verification was run-scoped by run id; the coordinator sweeps after the wave.

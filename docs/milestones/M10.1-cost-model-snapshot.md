# Milestone M10.1 — Cost Model API and Price Snapshot Format

**Phase:** 10 — Cost Engine
**Milestone:** M10.1 (cost model API + pricing snapshot format)
**Status:** Completed
**Date:** 2026-07-11

## Implemented

`@iap/cost` 0.1.0 — the reference cost model and estimation engine per spec ch. 16 and
IEP-0005. Cost is an **annotation layer computed by tooling, never document content**: the
core document describes intent; this package projects prices onto a canonical model. A cost
report is a **pure function of three versioned inputs** — the canonical profile-merged
document, a mapping cost model, and a content-addressed price snapshot — with no clock and
no network, so identical inputs yield a byte-identical report (§16.9). The package boundary
is `@iap/model` + `@iap/parser` + `ajv`.

### Price snapshot format (`price-snapshot-v1`)

The pricing input's implementation, left open by IEP-0005 and decided here. A snapshot
(`spec/schema/price-snapshot-v1.schema.json`) is a versioned capture of provider list
prices: `{ formatVersion, id, provider, currency, region, asOf, source, prices }`, where
`prices` maps opaque SKU keys → `{ unit, amount }`. It is the ONLY pricing input at
computation time — never a live lookup — so every amount is attributable. `snapshotContentAddress`
derives `<id>#sha256:<hex>` over the snapshot's canonical JSON; a report records this so a
price change is always detectable. `validateSnapshot`/`loadSnapshot` enforce the schema.
A bundled **reference (illustrative) snapshot** (`snapshots/reference-cloud.snapshot.json`,
USD) exercises the engine.

### Cost model API (`CostModel`)

A cost model DECOMPOSES a canonical resource into billable line items that reference
snapshot SKUs, plus the confidence and the usage assumptions behind an estimate — it holds
no prices and reads no clock/network (pure). Normatively a cost model ships WITH a provider
mapping (ch. 12); the bundled **`referenceCostModel` (`reference-abstract@1.0.0`)** prices
abstract kinds directly with transparent formulas so the engine and report format can be
exercised before provider-distributed models land (M10.2): compute kinds by size × mean
replicas (from the scaling range) × availability redundancy; databases by instance +
storage-GiB + throughput with a class multiplier; caches by memory-GiB; object stores and
messaging from stated usage assumptions; volumes/gateways exactly; logical kinds
(Application, Identity, Secret, …) at $0 exact; and everything it does not cover (reserved
kinds like Stream) as `unknown`.

### Estimation engine (`estimateCost`)

`estimateCost(model, { costModel, snapshot }) → cost-report/v1`. Per resource: decompose,
look each SKU up in the snapshot, sum (hourly items → `estimatedHourly`, ×730 → monthly;
monthly items the reverse), round deterministically. **Honest uncertainty (§16.2):** a
resource the model does not cover — OR one whose SKU is absent from the snapshot — is
reported `unknown` with **no numbers** and a stated reason (missing price data is visible,
never guessed). **Roll-ups (§16.3):** per Application (over `components`) and per label,
each carrying the sum of priced members, the **weakest member confidence**, and a
`lowerBound` flag set whenever an unknown member was omitted; `totals` aggregates the whole
document. The report records `modelHash` (the CIM hash), `priceSnapshot` (content address),
and `costModel` (`id@version`) so any change is attributable to exactly one input.

### Optimization suggestions (`suggestOptimizations`)

Deterministic rules over the model (§16.4), never model inference: **excess-availability**
(a `high`/`maximum` resource, savings = current − re-priced-at-standard) and
**orphaned-resource** (a stateful/messaging kind with no inbound workload edge, savings =
its full monthly). Each carries the rule id, the resource, a detail, and the projected
delta; output is sorted by (rule, resource). Oversizing is deferred until a versioned
observed-utilization input exists.

### Companion schemas

`spec/schema/price-snapshot-v1.schema.json` and `spec/schema/cost-report-v1.schema.json`
(both compiled by the spec harness under strict ajv), embedded copies synced into
`packages/cost/schemas/` (added to `tools/schema-generation/sync-schemas.mjs`) and
byte-drift-tested — the spec copy is the single source of truth (ADR-0002).

## Design decisions taken (spec ambiguities resolved)

1. **Snapshot content address = `id#sha256:<canonical-json-hash>`.** IEP-0005 left the
   snapshot format and content-addressing open; this mirrors IEP-0011's versioned-input
   treatment and ch. 16.8's `…#sha256:…` sketch. The hash is over `canonicalJsonStringify`
   of the whole snapshot, so id collisions cannot mask a price change.
2. **SKUs are opaque to the engine; prices live only in the snapshot; multipliers live only
   in the model.** The engine multiplies quantities by amounts and never knows what a SKU
   means; availability/class multipliers are model logic (not "list prices"), keeping the
   snapshot a pure price table and the model a pure cost function.
3. **Missing price = `unknown`, not zero.** A covered resource whose SKU is absent from the
   snapshot degrades to `unknown` with the missing SKU named — satisfying "missing price
   data is visible" without inventing a number. This reuses the same honest-uncertainty
   path as an uncovered kind.
4. **Roll-ups keep a number even when confidence is `unknown`.** Unlike a per-resource
   `unknown` (which carries no number), an aggregate reports the sum of its priced members
   with `lowerBound: true` and weakest confidence — the honest "at least $X" reading (§16.2
   lower-bound rule), distinct from a resource's no-number rule.
5. **The reference cost model prices abstract kinds.** Ch. 16 distributes cost models with
   provider mappings; a reference model over abstract kinds is a legitimate stand-in that
   makes the engine and report format testable now and gives provider-neutral estimates.
   Provider-distributed models replace it per target in M10.2; the report names the model
   `id@version` so the two are never confused.
6. **Money is float; the hashed path is untouched.** Cost is a projection, never part of the
   CIM or its hash (IEP-0008), so IEEE-754 money arithmetic with fixed deterministic
   rounding (monthly 2dp, hourly 4dp) is correct and byte-reproducible — no float ever
   enters a canonical-model hash.

## Tests added

`packages/cost/test/cost.test.ts` (20 tests): the reference snapshot is schema-valid and a
malformed one is rejected; the content address is deterministic and price-sensitive; both
embedded schemas are byte-identical to the spec copies (drift guard); every official example
(6) produces a schema-valid report with a confidence on every resource; a service is priced
by mean replicas across its scaling range; logical resources cost $0 exact and gateways are
flat/exact; an uncovered kind and a snapshot-missing SKU both report `unknown` with no
numbers and a reason, tainting the total as a lower bound; application and label roll-ups
aggregate with weakest-member confidence and lower-bound flagging; identical inputs yield
byte-identical reports and reload reproducibly; a different profile yields a different
report; excess-availability and orphaned-resource suggestions fire with correct deltas and
deterministic ordering; and the quantity/throughput helpers convert exactly.

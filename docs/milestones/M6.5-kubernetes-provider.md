## Implemented

**`@iap/provider-kubernetes` 0.1.0** (`providers/kubernetes/`, depends on `@iap/model` + `@iap/parser` + `@iap/provider-sdk` + `ajv` + `yaml` — no new dependencies): a complete, signed provider package claiming certification level **core** (mapping-only; execution handlers arrive with Phase 14, PC-5 no-partial-claims). Together with `providers/aws` it proves the phase-6 exit criterion: the canonical `spec/examples/basic-webapp.iap.yaml` (production profile) maps through both packages — same intent, two substrates.

### Package contents

- `manifest.json` — `plugin.iap.dev/v1`, name `iap-provider-kubernetes`, namespace `kubernetes`, version 0.1.0, `specCompat ">=1.0.0 <2.0.0"`, `sdkCompat ">=0.1.0 <1.0.0"`, `certificationLevel: core`, `capabilities.kinds` exactly the eight covered kinds, 11 pinned `sha256:` artifact digests, ed25519 signature (`keyId: kubernetes-test-2026`) produced by `tools/provider-packaging/sign-manifest.mjs`.
- `keys/` — committed **test-only** ed25519 keypair + README marking it as test material; the public key filename stem is the keyId, matching how the shared runner builds trust stores from `providers/<name>/keys/*.public.pem`.
- `mappings/core.iap-map.yaml` — one core mapping, eight kinds, 13 target types, all rules first-match-wins, every derive map total, every abstract output attribute bound (verified statically by the loader at every package load).
- `schema/extension.schema.json` — validates `extensions.kubernetes`: optional `version` plus per-resource refinement hints (`namespace` DNS-label pattern, `storageClassHint`, `replicasHint ≥ 1`), `additionalProperties: false` at every level; hints never alter core semantics (ch. 11) — the schema text says so explicitly (the encryption demand always prevails over `storageClassHint`).
- `conformance/` — four cases over five corpus documents (all conforming IaP documents), including a verbatim copy of `basic-webapp.iap.yaml` as the shared cross-provider corpus.
- `src/index.ts` — the package's `AttestationRegistry` factory plus name/namespace/keyId/target constants.

### Realization choices per kind (and what each can honestly attest)

| Kind          | Targets                                                                                                                                             | Realization notes                                                                                                                                                                                                                                                                                                              |
| ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `Service`     | `kubernetes:apps:Deployment` + `kubernetes:core:Service` + `kubernetes:autoscaling:HorizontalPodAutoscaler` + `kubernetes:networking:NetworkPolicy` | Ch. 12 §12.7's canonical decomposition. `size` → cpu/memory requests; `scaling` → replicas + HPA bounds/target; `availability: high` → zone-spread demand; `exposure` → NetworkPolicy posture (`private` = default-deny).                                                                                                      |
| `Gateway`     | `kubernetes:gateway:Gateway` + `kubernetes:gateway:HTTPRoute`                                                                                       | Gateway API per §12.7. `exposure` selects the gateway class (`public` → external, `internal` → internal); `tls.minimumVersion` → listener TLS floor; routes come from `routesTo` edges (edge-realized, not field-derived).                                                                                                     |
| `Database`    | `kubernetes:postgres-operator:PostgresCluster` + `kubernetes:core:Secret` + `kubernetes:networking:NetworkPolicy`                                   | Operator-managed PostgreSQL (CloudNativePG-style). `availability: high` → 2 instances with synchronous replication; `encryption.atRest` → encrypted-StorageClass demand; `inTransit` → operator-enforced TLS; `resilience` → backups + RPO/RTO carried onto the operator target; credentials are an operator-generated Secret. |
| `Cache`       | `kubernetes:redis-operator:RedisFailover` + `kubernetes:core:Secret` + `kubernetes:networking:NetworkPolicy`                                        | Operator-managed redis-compatible failover cluster; persistence deliberately disabled (cache contents are loss-tolerant by definition, §3.10 — with no durable data the at-rest surface is empty); `availability: high` → 3 replicas under sentinel.                                                                           |
| `ObjectStore` | `kubernetes:objectstorage:Bucket` + `kubernetes:networking:NetworkPolicy`                                                                           | COSI-style bucket claim on an in-cluster object store; `versioning`, at-rest encryption, TLS-only access, and backup posture derived.                                                                                                                                                                                          |
| `Queue`       | `kubernetes:messaging:Queue`                                                                                                                        | Operator-managed broker queue per §12.7 (RabbitMQ-style). Two realize rules demonstrate first-match-wins: `ordering: fifo` → quorum queue with single active consumer; default → classic queue.                                                                                                                                |
| `Identity`    | `kubernetes:core:ServiceAccount`                                                                                                                    | Workload identity as a ServiceAccount; permissions remain derived from the relationship graph (ch. 15), never enumerated.                                                                                                                                                                                                      |
| `Application` | `kubernetes:core:Namespace`                                                                                                                         | The ch. 3 §3.4 informative mapping ("Namespace + common labels"): a grouping realization that provisions scope, not workload.                                                                                                                                                                                                  |

**Eight kinds, not six.** `basic-webapp.iap.yaml` declares `Identity` and `Application` resources alongside the six production-slice kinds, and the acceptance criterion is zero diagnostics — an uncovered kind is a loud `unsupported-kind` rejection (ch. 12 §12.3), so both wave-2 packages must cover all eight. The loader enforces `capabilities.kinds` = covered kinds exactly (PC-5), so the manifest claims all eight.

### Rejection surface (deliberate, fail-closed)

Everything below is **rejected with a diagnostic naming the field/value**, never silently downgraded:

- `Database`: `class` other than `relational`; `engine` other than `postgresql` (no mysql/mariadb operator realization in this package — the AWS package accepts mysql, which is exactly the substitutability the supports matrix is for); `exposure: internal`.
- `availability: maximum` everywhere — a single cluster cannot attest a ≥99.99% multi-region floor.
- `Service`: `artifact.type` other than `container-image`, `runtime` other than `container`, `exposure: public` (public entry belongs to `Gateway`).
- `Queue`: `delivery: exactly-once` (the broker attests at-least-once).
- `Cache`: `memcached-compatible`; `ObjectStore`: `exposure: public` and `lifecycle` rules; `Gateway`: `tls.certificate` (no `Certificate` realization).
- `observability.traces: required` on every kind — this realization ships no tracing pipeline it could attest (`none`/`preferred` accepted).

### Attestation semantics vs. AWS (IEP-0012 "Provider impact")

Same abstract capabilities, structurally different predicates — the point of the two-target exercise:

| Capability                                   | kubernetes attestation (this package)                                                                                          | aws attestation                         |
| -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------- |
| `exposure.private`                           | `kubernetes:networking:NetworkPolicy` with `defaultDenyIngress` and connections allowed only from declared `connectsTo` edges  | security-group non-reachability         |
| `encryption.atRest`                          | encrypted-StorageClass demand on the operator target (PostgresCluster / broker Queue) or `encryptionAtRest` on the COSI Bucket | engine-level storage encryption flag    |
| `encryption.inTransit`                       | operator-enforced TLS on client connections                                                                                    | TLS enforcement on the managed endpoint |
| `availability.zonesMinimum` (params `{min}`) | operator `instances ≥ min` **and** synchronous replication on the HA path                                                      | multi-AZ placement                      |

Registry: six predicates across four capabilities and four target types, registered per `(capability, targetType)` via the SDK's `AttestationRegistry` — pure functions over plan attributes only. `resilience.backup` is deliberately **not** attested: the webapp case asserts it with `expect: unsupported`, exercising the third verdict honestly.

### Conformance cases

- `webapp-core.case.yaml` (shared corpus doc, production profile) — `encryption.atRest`, `encryption.inTransit`, `exposure.private` (on both `orders-db` and `assets` — same NetworkPolicy predicate across kinds), `availability.zonesMinimum {min: 2}` all `satisfied`; `resilience.backup` `unsupported`.
- `queue-fifo.case.yaml` — FIFO queue on the quorum/single-active-consumer variant, at-rest posture `satisfied`.
- `fail-closed.case.yaml` — `engine: mysql` and `delivery: exactly-once` both `rejected` with diagnostics naming the field.
- `function-uncovered.case.yaml` — a `Function` document `rejected` via `unsupported-kind`. **Note:** under the locked evaluator semantics an uncovered kind can never yield the `unsupported` verdict — any mapping failure is `rejected`; `unsupported` is reserved for unattested capabilities on successfully mapped plans (see `backup-unattested` above). The case file documents this in its header.

## Tests added

38 unit tests in `providers/kubernetes/test/` (+ `helpers.ts` building the trust store from `keys/*.public.pem`, keyId = filename stem, exactly like the shared runner):

- `package.test.ts` (11) — signed package loads via `loadProviderPackage` (PC-1 gauntlet passed by construction); core level claimed exactly once; `capabilities.kinds` ≡ covered kinds ≡ the eight documented; refusals on missing allowlist entry and empty trust store; extension schema accepts a well-formed refinement and rejects unknown properties, malformed namespaces, and bad resource keys; target vocabulary all `kubernetes:`-namespaced and in lockstep with the exported constants.
- `webapp-acceptance.test.ts` (8) — **the mandatory acceptance surface**: canonical basic-webapp (production) maps with zero diagnostics into 16 plan resources with the documented decomposition; every `desiredAttributes` entry has a provenance record; production-profile values derived (web `cpuRequest: "2"`, `replicas: 2`, HPA 2–6; orders-db `instances: 2` + synchronous replication; `defaultDenyIngress` true for private orders-db, false for internal web); every abstract output attribute of every kind bound to a produced plan resource; `dependsOn` derived from canonical edges; double-run byte-identical plans + equal `planHash` (PC-3); non-interference — canonical hash unchanged, frozen model throws on mutation (CM-6).
- `conformance.test.ts` (12) — all four cases pass through `evaluateConformanceCase` (PC-2) with per-assertion verdicts pinned; rejected assertions carry `unsupported-value`/field detail; the uncovered-kind case carries `unsupported-kind` detail; **PC-4**: three tampered-mapping fixtures (unencrypted storage demand, non-denying NetworkPolicy, under-replicated HA cluster) each flip exactly their assertion to `violated` while untampered assertions stay `satisfied`; duplicate attestation registration throws.
- `fail-closed.test.ts` (7) — direct `applyMapping` diagnostics: mysql → `unsupported-value` naming `spec.engine`; Function → `unsupported-kind`; `exactly-once` → `spec.delivery`; `availability: maximum` → `spec.availability`; `traces: required` → `spec.observability.traces`; no plan ever emitted alongside diagnostics; first-match-wins proven both ways (fifo → quorum rule 0, default → classic rule 1, provenance `ruleIndex` asserted).

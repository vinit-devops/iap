# Milestone M1.4 — Additional Official Examples

**Phase:** 1 — IaP v1 Stabilization
**Milestone:** M1.4
**Status:** Completed
**Date:** 2026-07-10

## Implemented

Five new official examples in `spec/examples/`, each complete, realistic, and schema-valid, following the established header-comment convention (`# <Title>: ...` + `# Demonstrates: ...`):

- `serverless-api.iap.yaml` — public Gateway routing to two Functions on distinct paths; key-value Database class (no engine — the class has no v1 engine enum value); a Topic with a Function `consumesFrom` subscriber; `storesDataIn` to an ObjectStore; workload Identity.
- `private-internal-service.iap.yaml` — zero public surface: no Gateway, one `internal` Service and one `private` Service connected service-to-service, private Database and Cache, Secret with required rotation, and a `deny` policy on `spec.exposure: public` targeting all kinds.
- `data-processing.iap.yaml` — scheduled Job (5-field cron) with read-only `storesDataIn` from an ObjectStore, write access to a timeseries-class Database (engine omitted — no timeseries dialect exists in the v1 engine enum), Queue-fed worker Service, Volume scratch space, and a reserved `Stream` kind (comment notes the expected IAP801 warning).
- `hybrid-environment.iap.yaml` — extension namespaces `aws` and `onprem` registered at document level with versions and refining different resources; `cloud`/`onprem` profiles as environments; cross-site Database `replicatesTo` with `site: datacenter` labels; comment restates the Extension Non-Interference Rule (all extensions blocks deletable without changing core semantics; provider nouns appear only inside extensions).
- `import-intent.iap.yaml` — brownfield adoption modeled entirely through the `x-*` passthrough convention (`x-iap-import`, `x-iap-import-source` on adopted resources) plus descriptive `metadata.annotations`; header explains that v1 models import intent via tool annotations and the state engine's import handlers (roadmap Phase 14), never via core fields — core stays intent-only.

Code and docs kept in lockstep: the harness's example count check (`tests/conformance/run.mjs`) and the parser test (`packages/parser/test/parser.test.ts`) now expect nine official examples; the README artifact table and Examples table list all nine.

## Conformance status

Green. All nine official examples validate against `spec/schema/iap-v1.schema.json` (ajv draft 2020-12, strict mode with the x-iap vocabulary registered).

## Architecture decisions

- Classes without a matching v1 engine enum value (key-value, timeseries) are exemplified with `engine` omitted — the class carries the intent and the mapping picks the store, which is the pattern authors should copy.
- Import intent is demonstrated strictly through `x-*` passthrough (plus non-semantic annotations) rather than proposing core fields, preserving the intent-only invariant until Phase 14 specifies import handlers.

## Specification gaps

None found while authoring; every intended construct was expressible in the v1 schema without workarounds (the reserved `Stream` kind's loose spec is by design, IAP801).

## Security findings

None. No credentials or provider identifiers appear anywhere; provider nouns occur only inside `extensions` blocks in `hybrid-environment.iap.yaml`.

## Known limitations

`import-intent.iap.yaml` documents a convention, not a behavior — nothing enforces `x-iap-import` semantics until the state engine (roadmap Phase 14) lands; until then validators simply pass the annotations through.

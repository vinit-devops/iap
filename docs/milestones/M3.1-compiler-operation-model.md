# Milestone M3.1 ‚Äî Compiler Operation Model: Vocabulary, Envelope Schema, Operation Gate, Replayable Log

**Phase:** 3 ‚Äî Intent Authoring Engine and Intent Compiler
**Milestone:** M3.1 (compiler operation model per IEP-0009 ‚Äî the foundation of `@iap/intent-compiler`)
**Status:** Completed
**Date:** 2026-07-11

## Implemented

**`@iap/intent-compiler` 0.1.0** (`packages/intent-compiler/`, depends on `@iap/model` + `@iap/parser` + `@iap/sdk` + `ajv` ‚Äî no new runtime dependencies): the closed operation model through which ALL authoring surfaces mutate an IaP document (IEP-0009). Proposals are data; `apply` is the gate; only the committed result serializes bytes. The package's dependency set structurally excludes every execution surface (phase-3 design decisions 4 and 10).

### Companion schema (IEP-0009 schema impact)

- `spec/schema/compiler-operations-v1.schema.json` ‚Äî `operations.iap.dev/v1`: a proposal BATCH (`apiVersion` + ordered `operations`, minItems 1 ‚Äî an empty proposal is meaningless). The envelope carries `operationId` (caller-supplied, pattern-checked, unique per batch), the closed twelve-type `type` enum, per-type `target`/`change` shapes enforced by twelve if/then branches, `sourceSpan`, `confidence` (0‚Äì1), `assumptions[]` (field/assumed/reason), `requiredClarifications[]` (id/question/field), `provenance` (closed 7-value `source` vocabulary per design decision 6 + closed 5-value authoring-surface `channel` + audit-only `modelId`/`promptVersion`), and optional `validationResult`/`previewDiff`. Update-shaped operations carry explicit `change.set` (dot path ‚Üí value) / `change.unset` (dot-path list) ‚Äî design decision 2, resolving IEP-0009 open question 1; no merge-patch semantics exist. `additionalProperties: false` throughout, every property described, draft 2020-12, compiles under strict ajv with the x-iap vocabulary registered.
- Embedded copy under `packages/intent-compiler/schemas/`, synced by the extended `tools/schema-generation/sync-schemas.mjs` and byte-equality drift-tested (ADR-0002); compile check added to `tests/conformance/run.mjs`.

### `operations.ts` ‚Äî closed vocabularies and envelope types

`OPERATION_TYPES` (the twelve: Create/Update/RemoveResource, Create/Update/RemoveRelationship, Apply/RemoveProfile, AddPolicy, ChangeConstraint, SetMetadata, SetExtensionValue), `DEFAULT_CONFIDENCE_THRESHOLD` = 0.8 (design decision 2, resolving IEP-0009 open question 3; configurable per apply session), `CONFIRMATION_CHANNELS` (`user-input`/`confirmed-clarification`/`accepted-recommendation`, IEP-0009 rule 3), `PROPOSAL_CHANNELS`, `STATEFUL_KINDS` (ch. 14 ¬ß14.2 six), typed envelope/batch/target/change shapes mirroring the schema (drift-tested), and `emptyDocument(name)` ‚Äî a skeleton to author into, deliberately not valid on its own (only the gate's dry run judges the RESULT).

### `gate.ts` + `apply.ts` ‚Äî the operation gate and transactional apply

`apply(document, batch, options) ‚Üí ApplyResult`, async and pure: **(1)** structural validation against the embedded schema (strict ajv; all findings collected); **(2)** batch-conflict check (duplicate operationIds) and confirmation-record validation; **(3)** copy-on-write application ‚Äî `structuredClone` once per batch, then sequential per-operation editing of the DOCUMENT object (design decision 1: the source of truth, never the derived CIM), each operation resolving targets against the state its predecessors produced (dangling targets, duplicate creates, ambiguous relationship references, grammar violations refuse; first application failure aborts, since later state is undefined); **(4)** transactional dry run ‚Äî the mutated document is fed as canonical JSON text to `@iap/sdk`'s `load` and the full ch. 8 pipeline runs on the result: parse + phases 1‚Äì4 via `validate()`, phase 5 via `policies()`, the phase-8 extension subset via `validateExtensions`; error-severity findings refuse as `validation-failed` with the IaP findings passed through untouched; **(5)** confirmation gate (OP-3): confidence strictly below the threshold, non-empty assumptions, or non-empty requiredClarifications each refuse without a caller-supplied confirmation (actor + closed channel + injected RFC 3339 timestamp ‚Äî the gate never reads a clock); destructive operations additionally require `acknowledgeDestructive: true`; **(6)** commit: `{ document', logEntries, canonicalHash, previewDiff, provenance, findings, serialize }` ‚Äî atomic per batch, input untouched on any refusal (IEP-0009 rule 1).

**OP-1 by construction:** the only exported byte-producing path is the `serialize` closure ON the committed result, created inside the gate after every stage passed. No exported function takes a raw document or proposal and returns serialized text.

### `errors.ts` ‚Äî closed operation-error taxonomy (design decision 3)

Fifteen exported constants: `invalid-operation-type`, `schema-violation`, `id-grammar`, `dangling-target`, `ambiguous-target`, `duplicate-create`, `batch-conflict`, `invalid-change-path`, `extension-namespace-violation`, `validation-failed`, `below-confidence-threshold`, `unconfirmed-assumptions`, `unconfirmed-clarifications`, `unacknowledged-destructive`, `invalid-confirmation`. Refusals are machine-readable (`code`, `operationId`, `path`, `message`, `findings` on `validation-failed` only). These are package-level codes, same precedent as mapping diagnostics (phase 6) and plan refusals (phase 7); registry registration goes to the standing candidate-diagnostics IEP. IaP codes raised by the dry run are never rewritten.

### `log.ts` ‚Äî append-only operation log with byte-identical replay (OP-2)

`OperationLogEntry` = `{ sequence, envelope, confirmation?, resultingHash }` where `resultingHash` is the canonical document hash after operations 0..sequence applied (computed per step during application). `replay(baseDocument, logEntries, options?) ‚Üí CommittedBatch` re-runs the identical gate, carries recorded confirmations through, and fails closed with `TypeError` on a non-contiguous log, a refusal (wrong base), or any recorded-vs-recomputed hash mismatch.

### `preview.ts` ‚Äî semantic diff + destructive flagging (design decision 8)

`iap-semantic-diff/v1`: adds/removes/changes as document dot paths computed over the CANONICAL projections of base and result (authoring noise never reports; a wholly added subtree reports its root path only, matching the IEP envelope example), plus `destructive` and per-operation `destructiveOperations` with closed reasons `stateful-remove` / `replace-eligible-update`. Classification: `RemoveResource` on a stateful kind, or an update touching a replacement-eligible path per `REPLACE_ELIGIBLE_PATHS` (Database class/engine, Cache engine, Volume accessMode, Queue/Topic delivery/ordering, Secret source) with directional rules for `Database.spec.engineVersion` and `Volume.spec.capacity.storage`: only decreases are destructive, compared via BigInt dotted-version segments and `@iap/model`'s exact-rational quantity parsing ‚Äî no floating point anywhere. Undecidable comparisons (unparsable operands, unset of a directional path, a wholesale `spec` set overlapping an eligible path) fail toward the destructive extreme, mirroring the planner's M7.1 posture.

### `provenance.ts` ‚Äî per-field records citing operation ids (OP-4)

`FieldProvenanceRecord` = `{ path, operationId, source }` for EVERY leaf an operation writes (creates walk to leaves; set values walk to leaves), with the closed source vocabulary of design decision 6 (`explicit-user`, `confirmed-clarification`, `organization-profile`, `policy`, `iap-default`, `mapping-default` reserved, `accepted-recommendation`). A confirmation through the `confirmed-clarification` or `accepted-recommendation` channel upgrades the recorded source (IEP-0009 rule 3). Records are last-writer-wins per path and sorted by path; defaults/profile materialization keep their existing canonicalization provenance. M3.3 extends this model.

### `schema.ts` + `index.ts`

`compilerOperationsSchema()` (embedded copy) and `validateBatchStructure()` ‚Äî the structural stage as data, reusable for structured-output enforcement against model adapters (M3.4). ajv errors fold into the taxonomy: out-of-enum `type` ‚Üí `invalid-operation-type`, identifier pattern failures ‚Üí `id-grammar`, everything else ‚Üí `schema-violation`, each attributed to its `operationId`. The public surface is pinned by test (OP-1).

## Design decisions taken (spec ambiguities resolved)

1. **`apiVersion` is `operations.iap.dev/v1`.** IEP-0009 names the artifact but not its apiVersion; the spelling follows the `plan.iap.dev/v1` companion-artifact precedent.
2. **Echoed outputs live in the input schema, optional and untrusted.** The IEP's envelope example carries `validationResult`/`previewDiff`; keeping them (optional) in the one schema means stored/replayed envelopes revalidate, while the gate ALWAYS recomputes both ‚Äî a supplied value can never influence application (OP-1 preserved).
3. **Dependency set: `@iap/model` + `@iap/parser` + `@iap/sdk` + `ajv`.** `@iap/validator` and `@iap/policy` are engaged through the sdk facade (both remain permitted by decision 10); `yaml` is unneeded at runtime ‚Äî the dry run feeds `load()` JSON text (a YAML subset) and output bytes come from the sdk's round-trip serializer, so the compiler adds zero serialization code of its own.
4. **`ApplyProfile` is an upsert of the complete definition.** Profiles are overlays; "patching a patch" via set/unset would be unreviewable. `RemoveProfile` deletes. There is deliberately no policy REMOVAL in the closed twelve ‚Äî `AddPolicy` creates, `ChangeConstraint` edits (set/unset over the policy body, id immutable); deleting governance rules is not an authoring-surface operation in v1.
5. **`UpdateResource` path guards.** `kind` is immutable (a kind change is remove + create ‚Äî the same reasoning as resolved open question 2's "no rename"); `relationships` paths refuse (edges are edited by the relationship operations, which enforce duplicate/ambiguity semantics); `extensions` paths refuse as `extension-namespace-violation` (ch. 11: SetExtensionValue is the sole path).
6. **Relationship addressing is by (verb, target).** Zero matches ‚Üí `dangling-target`; more than one ‚Üí `ambiguous-target` (two `connectsTo` edges to the same target with different ports are legal, so ambiguity is real); verb/target changes refuse ‚Äî retargeting is remove + create. Identical-edge creation refuses as `duplicate-create`.
7. **`lifecycle.replaceOn` stand-in.** Documents declare no `lifecycle.replaceOn` ‚Äî that channel is mapping/plan-side (`iap-mapping` carries none in v1 either, M6.1). The closed `REPLACE_ELIGIBLE_PATHS` table is derived from the normative ch. 3 per-kind Lifecycle clauses, with directional decrease rules for `engineVersion`/Volume capacity ‚Äî the deterministic reconstruction posture of the planner's statefulness derivation (M7.1 decision 3), failing toward the destructive extreme.
8. **Sequential target resolution.** Each operation resolves against the document state its predecessors produced (create-then-connect works in one batch); references embedded in created entries (`change.relationships[].target` of `CreateResource`) are content and surface as dry-run IAP201 findings ‚Äî except `CreateRelationship.change.target`, which IS the operation's essence and refuses as `dangling-target`.
9. **Collect-all where stateless, first-failure where stateful.** The structural stage and the confirmation gate report every problem at once; the application stage aborts at the first refusal because later operations' targets are undefined after a failed edit.
10. **Confirmation records are validated data.** Closed channel enum, non-empty actor, injected RFC 3339 timestamp (the package contains no `Date.now()`, no `Math.random()`, no ambient time), at most one per operation, and a confirmation naming an operation outside the batch refuses (`invalid-confirmation`, fail closed ‚Äî a stale record must not silently ride along).
11. **Document hashes are bare hex** (document-side convention, matching `@iap/model`'s `canonicalize`); the `sha256:<hex>` spelling remains plan-side (IEP-0011).
12. **`AddPolicy` requires an explicit `target`.** The gate materializes no implicit targeting ‚Äî `{}` (targets everything) must be authored deliberately.
13. **The companion schema embeds the closed kind/verb enums** (drift-tested against `@iap/model`'s `KINDS`/`RELATIONSHIP_TYPES`): a hallucinated kind dies structurally at the gate, before any YAML exists (IEP-0009 security impact), rather than deep in the dry run.
14. **Resource-level `SetExtensionValue` refuses a `version` path** (ch. 11 ¬ß11.2: the document-level registration governs), and unset-emptied namespace blocks are dropped so authored absence round-trips.
15. **Set semantics are strict**: missing intermediate containers are created (arrays when the next segment is an index), writing through an existing non-container refuses, and array writes may extend by exactly one index (no sparse arrays). Unset of an absent path is an idempotent no-op.

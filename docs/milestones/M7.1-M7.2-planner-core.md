**Phase:** 7 — Deterministic Planner
**Milestones:** M7.1 (plan schema + planner API) + M7.2 (diff engine, lifecycle determination, scheduler) — combined per the phase-7 execution order (wave 1: one coherent unit)
**Status:** Completed

## Implemented

**`@iap/planner` 0.1.0** (`packages/planner/`, depends on `@iap/model` + `@iap/parser` + `@iap/provider-sdk` + `ajv` + `yaml` — no new runtime dependencies): the deterministic planner core of IEP-0011. The planner consumes provider plans, never documents (phase-7 design decision 2): document → `@iap/model` `canonicalize` → `@iap/provider-sdk` `applyMapping` → **`@iap/planner` `plan`** → `plan.iap.dev/v1` artifact.

### Companion schema (normative-adjacent, per IEP-0011 schema impact)

- `spec/schema/plan-v1.schema.json` — `plan.iap.dev/v1` exactly per the IEP-0011 artifact sketch: `apiVersion`, `planId` (`sha256:<hex>` over canonical `content`), optional `envelope` (`createdAt`/`expiresAt` RFC 3339 + ed25519 `signature` — validated in shape, **never hashed**, created by M7.3), and `content` with `inputs` (all nine identities + `inputsHash`), `waves` (arrays of action entries: `resource`, closed `action` enum `create|update-in-place|replace|delete|import`, `fields`, `provenance {changedBy, fieldSources}`, `destructive`, `reversibility`), `destructiveActions` (always present, PL-3), `unknownValues`, `risk` (`score`/`class`/`factors`), `deltas` (cost/security/compliance with explicit unavailable/deferred markers), `rollback`, `verification`, `approvalsRequired`. `additionalProperties: false` everywhere, every property described. **Deliberately no `^x-` passthrough anywhere** (unlike the document-side schemas): any additional content would silently change `planId` and break approval binding — plan bytes are a closed contract (CP-2).
- Compiles under ajv 2020-12 **strict mode ON** (`@iap/parser` `createValidator`); embedded copy under `packages/planner/schemas/` synced by the extended `tools/schema-generation/sync-schemas.mjs` and byte-equality drift-tested (ADR-0002). Chapter 14 §14.5 gained the one-line normative reference per the IEP's schema-impact clause.

### `inputs.ts` — the determinism input vector (IEP-0011)

`DeterminismInputs`: the closed nine-element set — `documentHash` + `target {provider, profile}` (1), `profileHashes` (2), `policyBundles` (3), `extensionVersions` (4), `mappingVersions` (5), `discoverySnapshot` (6), `pricingSnapshot` (7), `stateRevision` + `stateIntegrity` (8), `plannerVersion` (9). `computeInputsHash` rebuilds the identity object member-by-member (extraneous properties can never leak into the hash) and folds it through `@iap/model`'s `canonicalJsonStringify` + SHA-256 — one canonical byte form in the whole toolchain, no re-implementation. `StateSnapshot` is the minimal IEP-0010 subset of design decision 3 (`{revision, integrity, objects: {logicalId: {type, attributes, managed}}}`) with `emptySnapshot()` (revision 0, the golden-plan baseline) and `computeStateIntegrity` (canonical serialization hash of `objects`).

### `diff.ts` — semantic diff

`diffAttributes`/`diffResources`: desired (`ProviderPlan.resources`) vs actual (`StateSnapshot.objects`) per attribute path, paired by logical id over the union of both sides, in lexicographic order. Comparison is **exact equality on canonical values** — both sides are already canonical, so no tolerance bands and no floating-point arithmetic exist anywhere in the hashed path. A desired/actual provider-`type` mismatch at one logical id sets `typeChanged` (impossible from a correct pipeline since the logical id embeds the type; treated as an implicit replace trigger, fail closed).

### `lifecycle.ts` — closed rule order (design decision 4) + reversibility (ch. 14 §14.6)

First match wins, per logical id: **(1)** desired, absent → `create`; **(2)** present but unmanaged → `import` (never a blind update; mirrors the mock executor's M6.2 semantics); **(3)** managed, a `lifecycle.replaceOn` field (or the provider type) differs → `replace` (destructive); **(4)** managed, any other attribute differs → `update-in-place`; **(5)** no difference → no-op, excluded from the plan (§14.2); **(6)** in state, managed, not desired → `delete` (destructive); **(7)** undesired unmanaged objects are invisible (import material, M6.2 decision 1).

Each entry carries `fields` (create: every desired attribute; update/replace: changed+added+removed paths, plus the `type` pseudo-field on a type change; import: drifted attribute names; delete: empty — the action is total; **names only, never values**), `provenance.changedBy` (`documentHash` for intent-driven actions, `stateRevision` for import), `provenance.fieldSources` (per field, encoded from the provider plan's per-attribute provenance: `constant`, `from:<path>`, `map:<path>`, or `state` for state-only paths), `destructive` (exactly replace/delete), and a reversibility class.

**Statefulness derivation** (`deriveStatefulness`): mapping v1 plans carry no kind (M6.1 decision 4), so the originating kind is reconstructed from the plan's two kind-correlated channels — provenance `source` paths (declared per kind by the normative schema, via `resolveKindField`) and the bound abstract-output set (which equals `abstractOutputsForKind` of the true kind exactly, CM-4). All candidate kinds stateful → stateful; none → stateless; **mixed, empty, or state-only (deletes) → stateful**, failing toward the destructive extreme. `STATEFUL_KINDS` = Database, Volume, ObjectStore, Queue, Topic, Secret (ch. 14 §14.2).

**Reversibility table** (roadmap rollback classes, ch. 14 §14.6): create/update-in-place/import → `fully-reversible`; replace → `reversible-with-data-risk` (stateful) / `replacement-based-recovery` (stateless); delete → `irreversible` (stateful — no restore source visible to the plan; M7.3 surfaces this in `rollback.limitations`) / `replacement-based-recovery` (stateless).

### `scheduler.ts` — waves (ch. 14 §14.3–§14.4)

Forward actions layer by longest dependency path (Kahn layering) over the desired plan's `dependsOn`, restricted to scheduled nodes with **transitive ordering edges inserted through no-op nodes** (§14.3). A `replace` occupies a single atomic node at its DAG position (§14.2's create-successor/rebind/delete-predecessor is one scheduled step); changed dependents order behind it via their edge, unchanged dependents surface in `verification` only (documented narrowing of §14.3 verify nodes). **Deletes schedule after all forward waves in reverse dependency order** (§14.3; design decision 4), with chains through retained state objects preserved transitively. Entries within every wave sort lexicographically by resource id (§14.4, CP-2). Dependency cycles throw (ordering cycles must have failed validation — §14.1; a cycle here is a malformed input, refused rather than mis-scheduled).

### `plan.ts` + `validate.ts` — assembly, canonical serialization, facade

`buildPlan({desired, state, inputs, risk?})` → `content`; `plan(providerPlan, stateSnapshot, identities?, options?)` → full artifact with `planId = sha256:<hex>` over `canonicalPlanSerialization(content)` (compact UTF-8 JSON, keys sorted by Unicode code point — `@iap/model`'s `canonicalJsonStringify`; waves by dependency rank; every collection in documented sorted order; CP-2/§14.5). The facade derives everything derivable (documentHash/target/mappingVersions from the provider plan, stateRevision/stateIntegrity from the snapshot, plannerVersion from the package) so it cannot be handed a contradictory vector; `buildPlan` **fail-closed verifies** the rest: provider-plan `planHash` recomputed, snapshot `integrity` recomputed (ch. 13 §13.1 — mismatch is corruption), and every identity cross-checked against the artifacts it names (TypeError on any mismatch). Content members: `unknownValues` (output-bound attributes of scheduled resources + `sensitiveFields` markers), `destructiveActions` (always present), honest `deltas` (cost `unavailable`/`deferred`, security derived from `spec.exposure`/`spec.encryption`-sourced scheduled fields, compliance `deferred: phase-11`), `rollback` skeleton (`re-plan-to-revision`, empty limitations), `verification` (one per scheduled resource: `created`/`attributes-applied`/`successor-ready`/`adopted`/`removed`), `approvalsRequired: []`, and `risk` from the injected `RiskAnnotator` seam (default `stubRiskAnnotator`: `{score: 0, class: low, factors: []}`, marked for M7.3 replacement). `validatePlanArtifact` validates against the embedded schema (strict ajv); a plan failing its own schema is unexecutable.

## Design decisions taken (spec ambiguities resolved)

1. **Digest spellings.** Plan artifacts use IEP-0011's explicit `sha256:<hex>` spelling throughout (`planId`, `inputsHash`, `documentHash`, `profileHashes`, `stateIntegrity`); the facade converts the provider plan's bare-hex `documentHash` (M6.1 decision 7) at the boundary.
2. **State snapshot carries optional deployed-time `dependsOn`.** Design decision 3's minimal shape (`type`/`attributes`/`managed`) suffices for action _classification_, but §14.3's reverse delete ordering — "a deleted resource is removed only after every deleted resource that depends on it" — is uncomputable without edges, because deleted objects no longer appear in the desired plan (the mock executor solved this identically by storing edges at apply time, M6.2 decision 7; the full state document records the deployed edge set, ch. 13 §13.3). `StateObject.dependsOn` is therefore an optional, additive member; absent ⇒ deletes are mutually independent (single trailing wave).
3. **Kind statefulness is derived, fail-closed.** The reconstruction rule above; undecidable resources (mixed candidates, reserved kinds, state-only deletes) classify stateful → destructive extreme. IEP-0011's provider impact assigns real per-action reversibility metadata to mappings; this derivation is the deterministic stand-in until a mapping-schema minor carries it (M7.3 may refine).
4. **Deletes never enumerate fields.** A delete's `fields` is empty (the action is total) and its `fieldSources` is `{}`; state attribute names beyond the diff never leak into plan content.
5. **`changedBy` is a two-value rule for now.** Intent-driven actions (create/update/replace/delete) → `documentHash`; import → `stateRevision` (the state snapshot, not the document, causes the classification). The schema's `changedBy` enum admits all eleven identity-element names so richer attribution (profile-caused, policy-caused) needs no schema change.
6. **Security deltas include creates.** Design decision 5 says "derived from exposure/encryption-relevant attribute changes present in the plan itself"; a create _establishes_ security posture, so any scheduled field whose provenance source lies under `spec.exposure`/`spec.encryption` is listed — including on creates. Deletes contribute none (no desired-side provenance).
7. **Cost reason tokens.** `unavailable`/`no-pricing-snapshot` without a pricing snapshot id; `deferred`/`pricing-deferred-phase-10` (echoing the pinned id) with one — machine-readable constants, never fabricated numbers (IEP-0011 cost impact).
8. **Risk rule-table version folds into `plannerVersion`** (phase-7 design decision 8, resolving IEP-0011 open question 2): swapping `stubRiskAnnotator` for the M7.3 rule table bumps `PLANNER_VERSION` and therefore every `planId` and `inputsHash`.
9. **Redaction rule (PL-5).** Plan content carries attribute _names_ only — no `desiredAttributes` values anywhere in the artifact; attributes named in `sensitiveFields` are additionally marked in `unknownValues` with reason `sensitive` so reviewers see a secret-bearing attribute participates without any material (CE-6). `unknownValues` covers scheduled (non-delete) resources only: a no-op resource's outputs are already resolved.
10. **The planner verifies its inputs before planning.** Recomputed `planHash`/`integrity` plus identity cross-checks make "plan over inputs you cannot vouch for" impossible by construction — the plan-side counterpart of CP-4, and the foundation M7.3's `verifyPlan`/`refuseIfInvalid` (PL-2) builds on.
11. **Closed plan bytes.** No `x-*` passthrough in `plan-v1.schema.json` (unlike document-side schemas): extensions would change hashed bytes invisibly to approvals.
12. **Provider-type change ⇒ replace.** `typeChanged` acts as an implicit `replaceOn` trigger with the `type` pseudo-field (source `state`), fail closed.

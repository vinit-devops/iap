# Milestone M11.1 — Security Rule Engine and Least-Privilege Reports

**Phase:** 11 — Security and Compliance Engines
**Milestone:** M11.1 (security rule engine + derived least-privilege reports)
**Status:** Completed
**Date:** 2026-07-11

## Implemented

`@iap/security` 0.1.0 — the security analysis engine per spec ch. 15. Its central rule is
ch. 15's: **security is derived from the canonical document, never annotated onto it**. The
engine reads exactly the three sanctioned sources (intent fields, the relationship graph,
policies) and produces the §15.9 review surface — no separate security description exists to
drift. Pure: no clock, no network, no provider input. Depends only on `@iap/model`.

- **Least-privilege grants (`deriveGrants`, §15.3)** — the sole source of permissions is the
  `access` attribute on edges. One grant per workload edge that declares `access`, attached
  to the workload's identity (its `authenticatedBy` target, or an implicit per-workload
  identity). No edge → no grant; an edge without `access` derives connectivity but no
  data-plane permission. Sorted deterministically.
- **Zero-trust reachability (`deriveReachability`, §15.4)** — per resource, who may initiate
  traffic: declared inbound `connectsTo`/`routesTo` sources with port/protocol, plus the
  `exposure` field (public/internal widen ingress only). Anything undeclared is denied.
- **Encryption posture (`deriveEncryption`, §15.6)** — both dimensions default `required`;
  explicit `preferred` downgrades are surfaced.
- **Findings IAP601–IAP603 (`securityFindings`)** — **IAP601** public exposure on a data
  kind (contextual: error when it is a `storesDataIn` target, warning otherwise); **IAP602**
  plaintext secret material in `configuration`/`x-*` fields (well-known token shapes —
  AWS/GitHub/Slack/PEM/JWT — plus secret-named keys with non-trivial values, §15.5);
  **IAP603** an encryption downgrade to `preferred` while `pci-dss-4.0`/`soc2` is active
  (§15.6). IAP604 (isolation unenforceable) is a plan-time, mapping-specific code and is not
  derivable from the document alone — it is emitted by the planner/mapping, not here.
- **Risk scoring (`scoreRisk`)** — a deterministic `none`…`critical` level from the finding
  set and public-data-kind count (secret material or public data + errors → critical).
- **`securityReport`** — assembles grants + reachability + encryption + findings + risk into
  a `reportVersion: 1` artifact; `hasBlockingFindings` reports whether any error is present.

### `iap security` (CLI)

Flips the Phase-11 stub live: load + canonicalize (respecting `--profile`), derive the
report, and render — the grant table, the zero-trust reachability graph, and findings
(human) or the full report (`-o json`, `formatVersion: 1`). Exit 0 when clean; **1 on any
error-severity finding** (critical findings can block per policy). `compliance` remains a
Phase-11 stub until M11.2.

## Design decisions taken

1. **Everything derives; nothing is annotated.** The engine takes only the canonical model;
   there is no security-config input that could widen posture (§15.1). This makes the report
   exactly reproducible and the document diff _the_ security review.
2. **IAP604 stays plan-time.** Isolation-unenforceable depends on the selected mapping's
   substrate capability, not the document; the validation-time engine cannot decide it, so it
   is left to the planner (Phase 14 execution surface). The engine provides the reachability
   graph that plan-time enforcement checks against.
3. **IAP602 is a conservative heuristic.** Well-known token shapes are flagged anywhere;
   secret-named keys are flagged only with non-trivial string values — favoring precision to
   avoid false positives on ordinary config, while catching the obvious leaks §15.5 targets.
4. **Risk is deterministic and coarse.** A stable `none`…`critical` scale from finding
   composition; no scoring model or inference (ch. 19).

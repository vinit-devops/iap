## Implemented

**`@iap/provider-sdk` 0.1.0** (`packages/provider-sdk/`, depends on `@iap/model` + `@iap/parser` + `ajv` + `yaml` — no new runtime dependencies; all crypto via `node:crypto`):

### Companion schemas (normative-adjacent, per IEP-0012 schema impact)

- `spec/schema/plugin-manifest-v1.schema.json` — `plugin.iap.dev/v1`: name/namespace, semver `version`, `specCompat`/`sdkCompat` comparator-list ranges, single `certificationLevel` (core | execution | drift, PC-5), `artifacts` (mappings[] + extensionSchema + conformanceCases required; icons/docs optional), `capabilities` (kinds/hooks/handlers), optional `attestations` path, `integrity.digests` (path → `sha256:<hex>`), ed25519 `signature`. `additionalProperties: false` with `^x-` passthrough at every object, descriptions on every property.
- `spec/schema/conformance-case-v1.schema.json` — `conformance.iap.dev/v1` per the IEP-0012 assertion example: `case`, `document`, `profile?`, `mappingInputs?` (scalar map), `assertions[]` with `id`, `select {resource, kind?}` (optional — `expect: rejected` cases evaluate the whole document), `capability` (ch. 5 dot vocabulary), `params?`, `expect` (satisfied | rejected | unsupported), per-assertion `document` override.
- Both compile under ajv 2020-12 **strict mode ON** with the `x-iap-*` vocabulary registered (`@iap/parser`'s `createValidator`); embedded copies under `packages/provider-sdk/schemas/` are synced by the extended `tools/schema-generation/sync-schemas.mjs` and byte-equality drift-tested (ADR-0002 no-second-source, same as `@iap/model`).

### `manifest.ts` — types + validation

`PluginManifest`/`UnsignedPluginManifest` and friends mirroring the schema; `validateManifest()` against the embedded copy; `IAP_SPEC_VERSION = '1.0.0'` and `PROVIDER_SDK_VERSION = '0.1.0'` constants that `specCompat`/`sdkCompat` are checked against (ch. 10: a tool refuses artifacts whose range excludes the version it implements).

### `signing.ts` — canonical signing form + digests (design decision 2)

The signing input is the manifest with `signature` removed, serialized by `@iap/model`'s `canonicalJsonStringify` (sorted keys, compact UTF-8) — one canonical byte form in the whole toolchain, not a re-implementation. `signManifest(manifest, privateKeyPem, keyId)` (ed25519 only; TypeError otherwise), `verifyManifestSignature(manifest, trustStore)` (fail-closed on unknown keyId, wrong alg, unparseable key, bad signature), `computeArtifactDigest(bytes)` → `sha256:<hex>` over exact file bytes. Ed25519 is deterministic, so re-signing the committed fixture reproduces its signature byte-for-byte (tested — doubles as a fixture-drift guard).

### `semver.ts` — minimal range checker (no new deps)

`parseSemver`/`compareSemver` (full prerelease precedence per semver.org §11, integer arithmetic only) and `satisfiesRange(version, range)` over space-separated comparator lists (`>=`, `>`, `<=`, `<`, `=`, bare = exact) — exactly the grammar the manifest schema's `semverRange` admits. Malformed input never satisfies (fail closed).

### `loader.ts` + `verify-mapping.ts` — refuse-at-install (PC-1, design decision 6)

`loadProviderPackage(dir, { trustStore, allowlist, specVersion?, sdkVersion? })` verifies in stages, collecting every failure within a stage, refusing the whole package on any (`{ ok: false, refusals: [{code, message, path?}] }` with the closed code set `manifest-missing | manifest-invalid | allowlist | signature | integrity | spec-compat | sdk-compat | artifact-invalid | coverage-tiling`):

1. manifest presence/JSON/schema; 2. publisher allowlist; 3. ed25519 signature against the trust store; 4. integrity — path-traversal rejection, every referenced artifact (directories expanded recursively) must have a digest, every digest must match exact file bytes; 5. package `specCompat`/`sdkCompat`; 6. mapping artifacts — YAML + `iap-mapping-v1.schema.json` validity, `provider` = package `namespace`, per-mapping `specCompat`, one owner per kind, `capabilities.kinds` exactly equal to covered kinds (no inflated or partial claims), extension schema compiles under ajv strict; 7. **static coverage tiling** via `verifyMappingArtifact`:

- **(a) total derive maps (CM-3)** — every `from`+`map` entry covers the supported domain of its source field: `supports.values` when constrained, else the specification enum resolved from the embedded IaP schema via `@iap/model` (`$defs/kinds/<Kind>` walk with `$ref` resolution; booleans → {true, false}); a map over an unbounded domain can never be total and refuses. Derive-entry form (`constant` xor `from` xor `from`+`map`) and target-key resolvability are verified here too.
- **(b) supports/realize tiling (ch. 12 §12.4)** — a no-`when` rule tiles everything; otherwise the cartesian product of the `when`-field domains (absence-aware: fields without a schema default and not required by their parent contribute an `(absent)` value; capped at 65536 combinations, documented) must be matched by some rule. `when` fields outside `supports.fields` and `when` values outside the domain (unreachable rules) are defects.
- **(c) output binding (CM-4)** — every abstract output attribute declared for the kind by ch. 3 §3.3 is bound, and every binding names a target type some rule produces.

### `engine.ts` — the pure mapping engine (ch. 12, design decisions 3–5)

`applyMapping(canonicalModel, mapping, { profile?, inputs?, specVersion? })` → `{ ok: true, plan }` | `{ ok: false, diagnostics }`. Purity is structural: inputs only, no clock/network/environment/filesystem; the input model is **deep-frozen** on entry and a before/after canonical-hash test proves non-interference (CM-6). Resources iterate sorted by resource ID; rule selection is first-match-wins in document order with structural equality on canonical field paths; `options.profile` differing from `model.profile` throws TypeError (the model owns profile merging); non-scalar inputs throw TypeError.

Fail-closed diagnostics use the **closed reason taxonomy** of design decision 3 (`unsupported-kind`, `unsupported-field`, `unsupported-value`, `unsupported-relationship`, `no-realize-rule`, `derive-map-gap`, `unbound-output`, `spec-compat`, `newer-minor-construct`) as `{reason, message, kind?, resourceId?, field?, value?, verb?}`, deterministically sorted; every defect in the document is reported in one run, and no plan is ever emitted alongside diagnostics (complete or absent).

Plan shape exactly per design decision 4: `formatVersion: 1`, `provider`, `mappingVersion`, `specVersion`, `profile`, `documentHash` (CIM hash), `inputs` (explicit scalar mapping inputs, part of the hashed identity), `resources[]` — `type`, `logicalId` (`<resourceId>.<targetType>`), `desiredAttributes`, `dependsOn` (sorted logical ids of every plan resource realized from each canonical edge target), `lifecycle` (`createOnly`/`updateInPlace`/`replaceOn`), `sensitiveFields`, `provenance` (per attribute: `constant` | `from` | `map` + source field + rule index) — plus `outputBindings` (resourceId → abstract attribute → `{logicalId, attribute}`) and `planHash` = SHA-256 over the canonical serialization with `planHash` excluded. Double-run hash equality is tested (PC-3/CM procedure input side).

### `conformance.ts` — shared assertion evaluator (design decision 8)

`evaluateConformanceCase(caseDoc, { mapping, attestations, corpusDir, specVersion? })`: validates the case against the embedded schema, loads each referenced corpus document through `@iap/parser` (`loadDocument`, must be conforming) and `@iap/model` (`canonicalize` with the case profile), applies the mapping with the case's `mappingInputs`, then evaluates each assertion through the `AttestationRegistry` — pure predicates registered per `(capability, targetType)`, receiving `{plan, resource, resourceId, capability, params}`. Verdicts: mapping diagnostics → `rejected`; no registered attestation on any produced target → `unsupported`; all matching attestations hold → `satisfied`; any returns false → `violated`, which can never equal an `expect` — the mechanism by which tampered plans prove attestations are not vacuous (PC-4). Harness misuse (invalid case, non-conforming corpus, unknown `select.resource`, kind-guard mismatch) throws; assertion failures are reported, never thrown.

### Signing helper + fixtures

- `tools/provider-packaging/sign-manifest.mjs` — `--generate-key` emits an ed25519 PEM pair; the signing mode recomputes every artifact digest (directories walked recursively, sorted), fills `integrity.digests`, signs via the **built SDK** (single implementation of the signing form), validates, and rewrites `manifest.json`. Used to produce the committed fixture and intended for the wave-2 provider packages.
- `packages/provider-sdk/test/fixtures/` — `keys/` (committed test-only ed25519 keypair, marked as test material), `tiny-provider/` (a complete signed core-level package: Database+Queue mapping with first-match-wins rules, bare and target-qualified derive keys, extension schema, conformance case + corpus, signed manifest with 5 pinned digests), and `webapp.iap-map.yaml` (a seven-kind mapping that tiles canonical `basic-webapp` under the production profile exactly, for the engine tests).

## Design decisions taken (spec ambiguities resolved)

1. **Field-coverage semantics.** `supports.fields` entries cover their entire subtree: a document leaf is in-matrix when its **index-free dot path** (array indices skipped, so `spec.ports[0].port` reads as `spec.ports.port`; empty containers contribute their own path) has any dot-prefix in `fields`. Coverage is scoped to `spec.*` — `labels`, `description`, and `extensions` are non-capability metadata/refinement (ch. 12 §12.1 allows mappings to _read_ extensions; they are not intent fields), and `x-*` keys are non-semantic annotations excluded at every depth (consistent with edge identity in ch. 4 §4.7).
2. **`newer-minor-construct` detection.** The engine has no newer schema to consult, so a spec field the embedded 1.0.0 schema does not declare _is_ the observable form of a newer-minor construct: it is rejected with `newer-minor-construct` (ch. 12 §12.3 — plans are complete or absent, never IAP804-warned), while a schema-known field merely missing from the matrix is `unsupported-field`. Reserved kinds (loose schema) skip the known-check and rely on the matrix alone.
3. **Derive-key grammar.** `<target>.<attribute>` with longest-declared-target prefix matching (target types contain `:` and may contain `.`); a bare key belongs to the rule's first target. Both spellings appear in normative examples (ch. 12 §12.6 uses bare keys; `spec/mappings/aws-reference.iap-map.yaml` uses qualified keys) — the SDK accepts both.
4. **`lifecycle`/`sensitiveFields` are neutral in v1 plans.** `mapping.iap.dev/v1` carries no lifecycle or sensitivity metadata, so every plan resource emits `lifecycle: {createOnly: [], replaceOn: [], updateInPlace: []}` and `sensitiveFields: []` — the shape is fixed now (design decision 4 names it) so a future mapping-schema minor fills it without a plan format change.
5. **Absent optional source fields.** A `derive.from` whose canonical value is absent (possible only for fields without specification defaults) yields an absent attribute — absent intent maps to absent configuration, deterministically; it is not an error and produces no provenance entry.
6. **No cascade diagnostics.** A resource that already failed coverage does not proceed to rule matching/derivation: reporting `derive-map-gap` for a value that was itself just reported as `unsupported-value` would be noise on top of the precise diagnostic.
7. **Hash spellings.** `documentHash`/`planHash` are bare SHA-256 hex, matching the CIM `hash` they interoperate with; `integrity.digests` values keep the explicit `sha256:<hex>` spelling of IEP-0012. Documented here to prevent drift.
8. **Package coherence checks** beyond the IEP letter: each mapping's `provider` must equal the manifest `namespace`, exactly one artifact may own a kind, and `capabilities.kinds` must equal the covered kind set — the PC-5 no-partial-claims rule applied to capability claims.
9. **Attestation delivery.** The manifest declares an `attestations` path, but M6.1 does not dynamically import package JavaScript; the evaluator takes an `AttestationRegistry` supplied programmatically by the provider package's own test suite. Sandboxed module loading is deferred to the wave-2/Phase 14 execution boundary (IEP-0012 open question 1).
10. **Engine takes one mapping artifact per invocation.** Packages may ship several mapping artifacts (the loader verifies disjoint kind ownership); a document must be mapped by the artifact covering its kinds. Multi-artifact composition into a single plan is deferred until a real package needs it (wave-2 packages ship one core mapping).

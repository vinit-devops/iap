# Milestone M3.5 — Evaluation Benchmark and Natural-Language Authoring Prototype

**Phase:** 3 — Intent Authoring Engine and Intent Compiler
**Milestone:** M3.5 (evaluation benchmark + natural-language authoring prototype)
**Status:** Completed
**Date:** 2026-07-11

## Implemented

M3.5 closes Phase 3 by turning the M3.2–M3.4 authoring surface into (a) a single
runnable **natural-language authoring prototype** and (b) a **measured evaluation
benchmark** over the roadmap §3.7 categories and measures. Nothing new touches the
normative boundary: the prototype only composes the pinned M3.1–M3.4 exports, and the
gate (`apply`) remains the sole path to document bytes (OP-1). No new runtime
dependencies; the package boundary stays `@iap/model` + `@iap/parser` + `@iap/sdk` +
`ajv`.

### The authoring prototype — `runAuthoringSession` (`src/author.ts`)

A single deterministic orchestrator that drives one natural-language request the whole
way through the pipeline, stopping honestly at the first thing only a human can decide:

```
adapter session (extract) → compileFacets → clarify → answer resolution
  → applyClarificationAnswers → requiredConfirmations → apply (the gate)
```

- **Closed outcome taxonomy** (`AUTHORING_OUTCOMES`): `committed` · `explained`
  (the §3.5 explain directive narrates, it does not author) · `no-operations` (nothing
  derivable — e.g. a wholly unsupported request) · `needs-input` (a human decision is
  required: an unanswered question or an unacknowledged destructive change) · `refused`
  (an adapter refusal, or the gate rejected the batch for a non-human-gate reason such
  as a validation failure).
- **One structured transcript** (`AuthoringSessionResult`): the full extraction (facets,
  unparsed, unsupported), the compiled proposal, every clarification question with the
  distinct **fired triggers**, the answers the engine resolved (explicit vs recommended
  default), the questions still blocking, the assembled confirmations, a deterministic
  `explainBatch` preview, and the committed result or the refusals — everything a UI or
  the Phase 5 CLI needs to render, decide, or continue.
- **Clock-free determinism** (exit criterion 4): the audit timestamp is INJECTED
  (`options.timestamp`); there is no clock and no randomness anywhere. Identical inputs
  produce byte-identical committed output, verified in both the unit tests and the
  benchmark.
- **The human gates are respected, not bypassed.** `autoAnswerDefaults` chooses a
  closed question's `recommendedOptionId` only when the recommended option needs no
  value; free-form and value-requiring questions still stop the session. Destructive
  operations are **never** auto-acknowledged — `acknowledgeDestructive` is an explicit
  opt-in that answers the destructive clarification with its acknowledge option and adds
  the `acknowledgeDestructive` confirmation; without it, a destructive change stops at
  `needs-input` (§19.6, phase-3 design decision 8). The input document is never mutated.

The Phase 5 `iap create` command (M5.3, now unblocked) will wrap this engine: the CLI
adds argument parsing, interactive prompting, and rendering; the authoring logic and
every normative boundary live here.

### The runnable demo (`tools/authoring-prototype/author.mjs`)

A thin human-facing driver over `runAuthoringSession` that prints the whole transcript —
extracted intent, clarifications with recommended defaults, assumptions, the semantic
preview, and the committed YAML with its canonical hash. `--demo` runs a scripted tour
(clear request → committed; missing artifact → needs-input; budget-vs-HA conflict →
clarification; unsupported products → nothing authored). Flags: `--base <path>`,
`--yes-defaults`, `--acknowledge-destructive`, `--json`. Runs against the built package.

### The evaluation benchmark (`tests/evaluation/`, `pnpm run test:eval`)

A harness plus a curated **oracle** dataset spanning all eleven §3.7 requirement
categories (`dataset.mjs`, 16 cases across clear · ambiguous · conflicting · missing ·
unsupported · provider-specific · incremental · destructive · security-sensitive ·
cost-sensitive · compliance). Each case's expectations are authored from the request's
SEMANTICS — what a correct compiler should produce — not by echoing the implementation.
The harness (`run.mjs`) drives the shipped `runAuthoringSession` over every case and
scores all **eight** §3.7 measures:

| #   | §3.7 measure                    | How it is scored                                                 |
| --- | ------------------------------- | ---------------------------------------------------------------- |
| 1   | Correct resource extraction     | multiset precision/recall/F1 of created resource kinds vs oracle |
| 2   | Correct relationship extraction | set precision/recall/F1 of created edges (`from-type->to`)       |
| 3   | Clarification precision         | fired triggers vs expected triggers (exact set)                  |
| 4   | Unsupported-feature detection   | set precision/recall/F1 of reported capabilities                 |
| 5   | False assumptions               | count of assumptions outside the per-case allowed set (target 0) |
| 6   | IaP validity                    | committed document reloads and re-validates green through ch. 8  |
| 7   | Semantic equivalence            | serialized bytes round-trip to the same canonical hash           |
| 8   | Deterministic serialization     | the same request twice → byte-identical YAML and equal hash      |

The harness is BOTH a measurement instrument — it prints per-category coverage and
aggregate precision/recall/F1 — and a conformance gate wired into `verify`: every case
must meet its oracle exactly, so any regression in extraction, clarification, or
determinism turns the run red. On the reference `rulesAdapter`, the corpus scores **100 %
F1 on all four extraction/clarification/unsupported measures, zero false assumptions**,
and validity/equivalence/determinism hold for every committed case (123/123 checks).

## Design decisions taken

1. **The prototype is an engine, not a command.** `runAuthoringSession` lives in the
   library so the CLI (M5.3), an IDE, and the benchmark all drive the identical logic and
   boundary. The Phase 3 deliverable is the engine + a runnable demo; the productized
   `iap create` is the distinct Phase 5 milestone.
2. **Destructive acknowledgment is explicit, at two layers.** The clarification engine
   raises a destructive question whose SAFE default cancels the change; `acknowledgeDestructive`
   selects the proceed/acknowledge option AND supplies the gate-level `acknowledgeDestructive`
   confirmation. A destructive removal that would still fail closed downstream (e.g. a
   dangling `outputs` reference, IAP203 — design decision 10 from M3.2–M3.4) surfaces as
   `refused` with the finding, never a partial write.
3. **The benchmark oracle is independent of the implementation.** Expectations are
   authored from request semantics; the harness measures how the reference extractor
   scores against them. The curated corpus sits within the extractor's designed
   vocabulary, so the reference implementation is expected to meet every oracle — a real
   conformance gate that is simultaneously an honest precision/recall report. Cases that
   the compiler correctly cannot express (named regions, provider products) are scored as
   unsupported-detection successes, not extraction failures.
4. **False assumptions are measured against a per-case allowed set.** Every legitimately
   defaulted field (e.g. `cache.spec.engine` when a cache's engine is unstated) is listed
   as allowed; any assumption outside that set is a false assumption. On the curated
   corpus the count is zero.
5. **The pinned export surface grew by exactly two names** (`runAuthoringSession`,
   `AUTHORING_OUTCOMES`), extended additively in `op-conformance.test.ts` per the M3.1
   OP-1 mechanism (design decision 11). The orchestrator composes the pinned surface
   above the gate and reaches document bytes only through `apply`; the no-byte-path audit
   still holds (`runAuthoringSession` is exercised in the authoring-e2e and author tests,
   never producing a serialization except via a committed result).

- **`packages/intent-compiler/test/author.test.ts`** (14 tests): the closed outcome
  taxonomy; a clear request committing to a green-revalidating document with OP-4
  provenance; missing-required-field → `needs-input` with the blocking question and no

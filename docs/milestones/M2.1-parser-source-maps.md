# Milestone M2.1 — Full-Fidelity Parser (Source Maps, Safety Limits, File/Stream Input)

**Phase:** 2 — Reference SDK
**Milestone:** M2.1
**Status:** Completed
**Date:** 2026-07-10

## Implemented

`packages/parser/src/source-map.ts` (new module, re-exported from the package root):

- `SourcePosition` (1-based `line`/`col`, 0-based `offset`), `SourceRange`, and `SourceMap` — a `Map` from RFC 6901 JSON Pointers (`''` = root, `~`/`/` escaped as `~0`/`~1` via `escapePointerSegment`) to source ranges, fulfilling the ch. 21 §21.3 Parser contract ("the AST preserves, for every node, its byte offset, line, and column").
- `buildSourceMap(text)` parses and walks the `yaml` AST (`node.range` + `LineCounter`), recording a range for every map entry (key start → value end) and sequence item; returns `{ map, errors }` with best-effort coverage on broken input (partial results, ch. 21 §21.1.3). `sourceMapFromDocument(doc, lineCounter)` exposes the walk for an already-parsed document so `parseText` never parses twice. Aliases are recorded at their use site but never dereferenced — the walk is linear in source length.

`packages/parser/src/index.ts` (non-breaking additions; every existing export and behavior unchanged):

- `ParseOptions.sourceMap?: boolean` → `ParseResult.sourceMap?: SourceMap`, populated by `parseText` and propagated through `loadDocument`.
- Alias/anchor safety: `doc.toJS` now runs with `maxAliasCount: MAX_ALIAS_COUNT` (= 100, exported); a document exceeding it (alias bomb) is rejected with an IAP101 "alias expansion limit" finding instead of an exception, without ever expanding the bomb.
- File and stream input: `loadFile(path, options?)` (`node:fs/promises` + `filename` defaulting to the path) and `loadStream(asyncIterable)` (accumulates string/Buffer chunks — documents are bounded per ch. 2 §2.2 — with multi-byte-safe Buffer concatenation).
- `attachPositions(findings, map, filename?)` — returns copies of findings whose `path` resolves in the map (exact pointer or nearest recorded ancestor) with messages prefixed `file:line:col`, giving schema findings (IAP1xx) authored-source positions.

## Architecture decisions

- Source map as a flat JSON-Pointer map rather than a retained AST: findings already carry JSON Pointers (`instancePath`), so decoration is a lookup; the LSP (ch. 23) gets positions without the SDK exposing `yaml` internals in its public types.
- Map entries span key → value (not value only): "missing/invalid field" findings point where a human looks (the key), while enum/pattern findings on scalars land on the same line either way.
- Alias limit enforced at `toJS` time (the only expansion point) and the source-map walk never dereferences aliases, so both paths are bomb-safe by construction.
- `loadStream` buffers the whole stream before parsing: IaP documents are single bounded files by spec, and YAML cannot be validated incrementally against a document schema anyway.

## Specification gaps

None found; ch. 21 §21.3 was implementable as written.

## Security findings

Alias-expansion ("billion laughs") input previously threw an uncaught `ReferenceError` from `yaml`'s default guard; it is now a structured IAP101 finding, keeping the "engines never throw on document problems" contract (ch. 21 §21.1.3).

## Known limitations

- Comments preservation is deferred to the serializer milestone **M2.6**, with rationale: canonical form discards comments _by definition_ (canonicalization normalizes a document to its semantic content), so the round-trip serializer — not the parser — owns any comment-carrying representation; source-fidelity editing (rename, autofix without reformatting) is an LSP concern that operates on original text via this milestone's source map rather than on a re-serialized tree.
- Pointers inside aliased content resolve only at the anchor site; `attachPositions` falls back to the alias use-site entry (nearest ancestor).
- `loadFile` rejects on I/O errors (unreadable file ≠ document problem); document problems remain findings.

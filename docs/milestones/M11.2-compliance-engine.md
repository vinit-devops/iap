# Milestone M11.2 — Compliance Bundles, Evidence Model, and `iap compliance`

**Phase:** 11 — Security and Compliance Engines
**Milestone:** M11.2 (compliance bundle format + six framework bundles + evidence model)
**Status:** Completed
**Date:** 2026-07-11

## Implemented

`@iap/compliance` 0.1.0 — the compliance engine per spec ch. 17. Compliance is **embedded,
not bolted on**: because posture, data protection, and reachability are all derived from the
canonical document (ch. 15), framework controls are evaluated — and evidenced — entirely at
the intent level, re-verifiable by any party. Pure and deterministic; depends only on
`@iap/model`. Completes Phase 11.

- **Bundle format + the six versioned bundles (§17.1–§17.3).** `FRAMEWORK_BUNDLES` is the v1
  registry — exactly the six schema-enum frameworks (`soc2`, `pci-dss-4.0`, `hipaa`,
  `iso27001-2022`, `nist-800-53-r5`, `cis-8.0`), each a versioned bundle (`version`,
  `specCompat`, optional `scopeLabel`) of controls drawn from ch. 17.2's representative
  catalogs. A control maps a control id + title to a **rule** — either a field rule (a
  dot-path leaf over the resource) or a **derived rule** from a closed set of graph checks
  (`every-data-edge-has-access`, `no-admin-to-data`, `workloads-authenticated`,
  `no-undeclared-reachability`) — plus remediation, technical evidence, and the external
  evidence an auditor still needs.
- **Evaluator + evidence report (§17.4/§17.6).** `evaluateCompliance(model)` evaluates each
  active framework's controls against their in-scope resources, giving every control a
  disposition — **`satisfied` / `violated` / `not-applicable`** — with contributing document
  paths. Violations emit **IAP701** carrying framework, control, bundle version, and resource
  path; a declared framework with nothing in scope emits **IAP702**. The report carries a
  summary, the bundle versions evaluated, and a normative **disclaimer**: it is configuration
  coverage at the intent level, **never a claim of certification**.
- **Label scoping (§17.7).** Scoped controls apply only to resources carrying the bundle's
  scope label (e.g. `pci-scope: "true"`); out-of-scope resources are `not-applicable`.
  Scoping narrows targets, never weakens a rule.
- **`iap compliance` (CLI).** Flips the last Phase-11 stub: load + canonicalize, evaluate,
  and render per-control dispositions with remediation and external-evidence notes (human),
  or the full evidence report (`-o json`, `formatVersion: 1`). Exit 0 when clean; **1 on any
  IAP701/IAP702 error** — so critical findings block through policy.

## Design decisions taken

1. **Bundles are in-code data, versioned like packages.** A bundle is a typed literal with a
   semver `version`; the report records the exact versions evaluated (§17.3), so a control
   addition is an explicit, reviewable version change. (A distributable bundle-file format is
   a follow-on; the data shape is the contract.)
2. **Controls carry rules, not free code.** Field rules + a closed derived-check set keep
   every control a deterministic, reproducible function of the document (§17.2) — no bundle
   can smuggle in nondeterminism.
3. **IAP701 vs IAP702.** IAP701 is a control violation (a resource fails a rule); IAP702 is a
   structural gap (a scoped framework declared with nothing labeled in scope) — the two are
   distinct, matching the registry.
4. **Certification is never claimed.** The evidence report distinguishes automated
   configuration coverage from formal certification and names the external evidence still
   required, satisfying the Phase 11 exit criterion directly.
5. **Overlap with the security engine is intentional.** An encryption downgrade under an
   active framework yields both IAP603 (security, ch. 15 §15.6) and IAP701 (compliance) —
   emitted independently, as §17.4 requires.

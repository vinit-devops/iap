# Milestone M05.3 — Retroactive IEPs 0001–0007

**Phase:** 0.5 — Specification Change Control
**Milestone:** M05.3
**Status:** Completed
**Date:** 2026-07-10

## Implemented

Wrote the seven retroactive IaP Enhancement Proposals required by roadmap Phase 0.5, formalizing the design decisions already shipped in the v1.0.0 draft specification so the existing design carries the same review trail as future changes. Each follows the full IEP template (`spec/ieps/IEP-TEMPLATE.md`), Status: Implemented (retroactive), Target version 1.0.0, Decision: "Adopted in IaP v1.0.0 draft; formalized retroactively per Phase 0.5":

- **IEP-0001 — Resource model.** Flat `resources` map with `kind` discriminator over capability-keyed sections: uniform graph node, zero top-level schema churn, no forced buckets; Field Definition Contract, shared intent vocabulary, 13 full + 9 reserved kinds.
- **IEP-0002 — Relationship semantics.** Canonical edge tuple, closed ten-verb set, inline-plus-rule-edge declaration with deterministic normalization, order derived never declared, `dependsOn`/`replicatesTo` ordering exceptions, DAG requirement.
- **IEP-0003 — Provider mapping contract.** Separate `*.iap-map.yaml` artifacts applied as a pure function with no ambient lookups, fail-closed supports matrix, first-match structural-equality realize rules with total derive maps, mandatory abstract-output binding, the §12.8 equivalence definition.
- **IEP-0004 — Policy language.** Structural condition trees with nine operators and three effects; Rego/CEL deliberately excluded for v1 determinism and audit surface; three-domain ordered comparison; budgets reuse the same machinery.
- **IEP-0005 — Cost model.** Cost as an annotation layer, never document content; pure report function over canonical document + mapping cost model + content-addressed price snapshot; budgets as ordinary deny policies (IAP505); deterministic-rule-only suggestions; carbon parity channel.
- **IEP-0006 — Security model.** Posture derived from exactly three sources; least privilege solely from edge `access` attributes; zero-trust default-deny reachability; secret values never in documents; explicit-only encryption downgrades; three enforcement points.
- **IEP-0007 — Extension framework.** Flat namespaces, document-level registration + resource-level refinement, the Extension Non-Interference Rule with the deletion test (inline provider fields rejected by construction), IAP802/IAP803, `x-*` vs. extensions contract split.

Updated the IEP index (`spec/ieps/README.md`): rows 0001–0007 now link to the new files with Status "Implemented (retroactive)"; rows 0008–0013 untouched.

## Tests added

None (specification documentation). Each IEP's Conformance requirements section points at the existing Chapter 24 classes and suite cases covering its area.

## Conformance status

Unchanged. No normative chapter, schema, or conformance-case content was modified; the retroactive IEPs document shipped design without altering it.

## Architecture decisions

Retroactive IEPs record decisions rather than propose them: every Detailed design section is a concise recap with authoritative pointers to the owning chapters/schemas, and every Alternatives/Rejected section reconstructs the trade-offs honestly (e.g. capability-keyed sections, inline-only edges, Rego/CEL, embedded mappings, inline provider fields). Open questions never dangle: each links to the gap-analysis item (`docs/reports/v1-gap-analysis.md` §4/§5) or to the successor IEP (0008–0013) that owns the area, satisfying the Phase 0.5 exit criterion that every unresolved specification question be linked to an IEP or explicitly deferred.

## Specification gaps

No new gaps. Known gaps are now formally owned: quantity normalization and default materialization (IEP-0008), `replicatesTo` failover non-goal and `require`-autofix scope (1.0.x clarifications flagged in IEP-0002/IEP-0004), `exposure: internal` realization metadata (IEP-0003 open question), and the missing conformance cases from gap analysis §4 (Phase 1 additions).

## Security findings

None new. IEP-0006 consolidates the shipped derived-security rules; IEP-0007 records non-interference as a security boundary (extensions narrow, never widen, derived posture).

## Known limitations

The retroactive IEPs are documentation of the v1 draft, not re-litigations: where the shipped design is underspecified, the IEP records the gap and its owner rather than resolving it here. Suite coverage for several cited error codes (IAP301/302/402/5xx/601/602/803) remains a Phase 1 deliverable.

# Milestones M2.2 + M2.3 — Canonical Infrastructure Model and Canonicalization Engine

**Phase:** 2 — Canonical Model and Reference SDK
**Milestones:** M2.2 (`@iap/model` — CIM types, IEP-0008) and M2.3 (canonicalization engine + stable hashing + determinism suite)
**Status:** Completed
**Date:** 2026-07-10

## Implemented

All in `packages/model` (no new runtime dependencies; `node:crypto` only):

**`src/cim.ts` (M2.2)** — the CIM logical schema per IEP-0008: `CanonicalModel` (apiVersion, metadata, resources, edges, policies, compliance, extensions, outputs, active `profile`, `hash`, `provenance`, `diagnostics`), `CanonicalResource` (kind, labels, defaults-materialized spec, extensions, `x-*` passthrough), `CanonicalEdge` (`source, type, target, attributes` with non-semantic `description`/`x-*` carried but excluded from identity, ch. 4 §4.2), and a total `ProvenanceMap` of JSON pointer → `{source: explicit | default | profile, originId, explanation}` (policy-sourced provenance arrives with the policy engine). Diagnostics and provenance are out-of-band and never affect the hash (I1/I2).

**`src/quantity.ts` (M2.3)** — exact-rational quantity/duration engine per ch. 1 §1.5.2. All arithmetic is BigInt over milli-units (10⁻³); binary floating point is never used. `parseQuantity` returns the exact value or `null` (grammar violation or precision finer than `m` — IAP103); `canonicalQuantity`/`canonicalDuration` return `{ok, value | reason}` so callers own the finding. Canonical spelling: zero → `0`; integers prefer `Ti, Gi, Mi, Ki, T, G, M, k` in that order, else bare; sub-integer → `<milli>m`; durations take the largest unit representing an integer (`ms/s/m/h/d` ladder; zero spells `0d` under the same rule). All seven normative quantity examples and all four duration examples reproduce byte-exactly.

**`src/canonicalize.ts` (M2.3)** — the C1–C6 pipeline (ch. 1 §1.5) as pure functions:

- `mergeProfile` (C2) — RFC 7386 merge (objects deep-merge, arrays replace, `null` deletes), `extends` chain root-first, selected profile last, `profiles` key removed post-merge; unknown profile / `extends` cycle → IAP205 and the merge aborts; overrides carrying a `profiles` key are rejected (ch. 6 §6.4). Optional provenance tracking records every profile-set pointer.
- `flattenEdges` (C3) — the six-step ch. 4 §4.7 algorithm: rule-edge expansion in document order with lexicographically sorted matches (zero matches → IAP402), inline collection in lexicographic resource order, dedupe on `(source, type, target, key-sorted attributes)` ignoring `description`/`x-*` (inline declarations donate non-semantic fields), stable sort by (source, verb enumeration order, target, serialized attributes). The `access: read-write` schema default materializes on `connectsTo`/`storesDataIn` edges only, so explicit and omitted spellings converge. Reference validation (§4.7 step 4) is deferred to M2.4; dangling targets are carried through. Pre-flattened `edges` arrays re-normalize, making the pipeline idempotent.
- `materializeDefaults` (C4a) — schema-driven walk of the embedded normative schema (`$defs/kinds/<Kind>` per resource; reserved kinds hit the default-free `ReservedKind` subschema; `$ref`s resolved into the common `$defs` with sibling keys winning). Implements the seven §1.5.1 rules: scalar defaults when the parent is present; absent `spec` counts as present-empty; recursive object materialization; arrays never; `x-iap-presence-semantic` constructs never synthesized (member defaults still apply when authored); `x-iap-default-when` conditional defaults evaluated against effective sibling values (`deadLetter.maxReceives` only while `enabled: true`); per-kind defaults via the split resilience defs (M1.2); extensions and `x-*` untouched. Every materialized pointer gets a `default` provenance record.
- `normalizeUnits` (C4b) — canonical spellings applied only to fields whose resolved subschema `$ref`s `common/quantity`/`common/duration` (never blind pattern-matching, so image references and configuration values that look like quantities are untouched); grammar/precision violations → IAP103, authored value kept.
- `canonicalize` (C1–C6) — merge → flatten → materialize → normalize → recursive key sort (Unicode code point) → compact JSON → SHA-256. The canonical JSON replaces all relationship spellings with the top-level `edges` array, never contains `profiles`, retains `x-*` keys as data, and serializes numbers via shortest round-trip. Returns `{model, canonicalJson, hash, findings}` with total provenance (explicit/default/profile) over every leaf of the canonical document.

`src/index.ts` re-exports the three new modules; all pre-existing exports are intact (drift tests unchanged). `yaml` added as a devDependency of `@iap/model` for the test suite only.

## Architecture decisions

- Defaults are discovered exclusively by walking the embedded schema (`default`, `x-iap-presence-semantic`, `x-iap-default-when` annotations) — no hand-maintained default tables, so schema evolution cannot drift from the canonicalizer (continues M1.2's def-splitting rationale).
- Quantity/duration fields are identified by `$ref` target, never by value shape, per the §1.5.2 requirement that only schema-typed values normalize.
- With no active profile (`profile: null`) the `profiles` key is still removed from the canonical projection, keeping the CIM shape uniform; the canonical form remains "relative to exactly one active profile" for planning purposes (ch. 6 §6.3 allows validation without one).
- `canonicalize` accepts a canonical document back (top-level `edges`), making canonicalization a fixed point without a second entry point.
- IAP205 is used for the ch. 6 §6.4 "overrides must not contain `profiles`" rejection (profile-mechanism error family); revisit if the error registry later assigns a dedicated code.

## Specification gaps

None found: chs. 1/4/6 plus the M1.1/M1.2 annotations were sufficient to implement C1–C6 without interpretation beyond the decisions above.

## Security findings

None. The CIM stores no secret material (CD-6); hashing uses `node:crypto` SHA-256.

## Known limitations

- Provenance for values that profiles write into `relationships` arrays does not survive C3 flattening (the array is replaced by canonical edges); those edge fields read as `explicit`. Precision here can improve when the relationship engine (M2.5) owns edge provenance.
- `canonicalize` parses JSON text only; YAML (C1 with source spans) is `@iap/parser`'s contract (M2.1) and the SDK facade (M2.6) will compose the two.
- Per-resource `intentHash()` (ch. 13 §13.2) is deferred until the planner needs it (IEP-0008 sketches it; no consumer exists yet).

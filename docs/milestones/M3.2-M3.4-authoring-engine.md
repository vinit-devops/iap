# Milestone M3.2+M3.3+M3.4 — Intent Authoring Engine: Facets, Extraction, Clarification, Provenance Flow, Semantic Explain, Model-Provider Abstraction

**Phase:** 3 — Intent Authoring Engine and Intent Compiler
**Milestones:** M3.2 (intent extraction + clarification engine), M3.3 (confidence/provenance flow + semantic diff), M3.4 (model-provider abstraction)
**Status:** Completed
**Date:** 2026-07-11

## Implemented

**`@iap/intent-compiler` 0.1.0 (extended)** — the layer between natural language and the M3.1 operation gate, entirely inside the existing package (no new runtime dependencies; the boundary stays `@iap/model` + `@iap/parser` + `@iap/sdk` + `ajv`). Everything added emits DATA — facets, proposal batches, questions, recommendations, prose — and the M3.1 gate remains the only path to document bytes (OP-1 untouched).

### M3.2 — facet model, deterministic extraction, facet compiler, clarification engine

- **`src/facets.ts`** — the typed, CLOSED facet vocabulary: the twenty roadmap §3.2 extraction targets (`environment`, `workload`, `application`, `data-service`, `messaging`, `networking`, `exposure`, `identity`, `secret`, `availability`, `scaling`, `region`, `backup`, `recovery-objective`, `security`, `compliance`, `budget`, `operational`, `provider-preference`, `existing-resource`) plus `removal` (the §3.5 removal directive). Every facet carries value(s), a `sourceSpan` (input id + start/end offsets + quoted text), a per-facet confidence, and the extraction channel from the closed three-value set (`exact-keyword` 0.95 / `pattern-match` 0.85 / `inferred-association` 0.7 — `CONFIDENCE_TIERS`, deterministic constants; the inferred tier sits BELOW the 0.8 gate threshold by design, so inferred structure always requires confirmation). `ExtractionResult` is the adapter output contract: facets plus the explicit `unparsed` and `unsupported` reports, the `explain` directive flag, an optional direct `proposal` batch, and integer adapter `usage` counts. The machine-readable contract is the embedded `intent-facets/v1` schema (strict ajv with the x-iap vocabulary via `@iap/parser`; `intentFacetsSchema()` / `validateExtractionStructure()` — the facet half of structured-output enforcement, collect-all issues).
- **`src/compile.ts`** — `compileFacets(facets, currentDocument, options) → { batch | null, unsupported, unresolved }`, a pure deterministic function. Resource and operation ids derive from facet content (`op-create-db`, `op-remove-edge-web-orders-db`, deterministic `-2` suffixes on collision — never random). Per-operation confidence is the MINIMUM of the contributing facets' confidences (documented aggregation rule; `min` is a comparison, never float arithmetic). Every defaulted value becomes an `assumptions` entry (cache engine → redis-compatible, database class → relational, volume size → 10Gi, DR objectives → 1d/4h, unstated scaling range → 1–4, Gateway asked to be private → internal) — never silently confident. Modifier facets fold into batch creations or merge into ONE `UpdateResource` per existing resource (minimal diffs); subject references resolve explicit-id-first then unique-kind, and zero/multiple matches surface as `UnresolvedSubject` records for the clarification engine. Removals emit edge cleanup and Application-membership cleanup before `RemoveResource`. Compliance facets compile to closed per-framework policy sets (all six ch. 17 frameworks; ids slugged to the DNS-label grammar; correct ch. 7 effect semantics — `require` for must-hold rules, `deny` for forbidden states). Budget and provider preference become NON-SEMANTIC annotations (`budget-monthly-usd`, `provider-preference` — ch. 19 §19.5). Out-of-vocabulary requests (named provider regions, impossible per-kind exposures) surface as explicit `UnsupportedFinding`s — never guessed into extensions.
- **`src/extract-rules.ts`** — the real, deterministic NL→facets extractor: ordered vocabulary/pattern tables over the facet space (kinds, engines with class consistency, availability words, exposure words, all six compliance framework spellings, integer budgets, RPO/RTO with duration-unit normalization, region names, scaling ranges, relationship phrasings — "connects to", "behind a gateway", "publishes to", "stores in", "cache for the X"). Each match claims (covers) its span in rule order; later rules never re-interpret covered text; artifact references are claimed FIRST because image URLs contain vocabulary words. Duplicate phrasings of one resource merge ("web app … the storefront"); provider products (DynamoDB, S3, Lambda, …) and out-of-scope capabilities (VPN, Kafka, …) become `unsupported` findings with provider-neutral suggestions; reserved kinds are reported as reserved. Handles all eight roadmap §3.5 incremental-edit commands against an existing document (subject nouns resolve to document resource ids first, then to kinds). PRECISION OVER RECALL: uncovered non-stopword text becomes `UnparsedSpan` reports — unparsed input never silently drops (an unattachable artifact reference is reported explicitly).
- **`src/clarify.ts`** — the §3.3 trigger list as a CLOSED deterministic rule set (`CLARIFICATION_TRIGGERS`, exactly eight) over (facets, proposal batch, current document): required-field-not-defaultable (workload artifact), materially-divergent interpretations (queue-vs-topic; "Reduce expected cost"; wholly unparsed input), cost/availability conflict (the roadmap worked example: budget below the deterministic `HA_DATABASE_BUDGET_FLOOR_USD = 400` against a high-availability database produces the three-option question verbatim — raise the budget / use standard availability / use a lower-cost database class), destructive updates (stateful removes and replace-eligible updates via the M3.1 preview classifiers), compliance-scope gaps, provider selection (more than one named provider), unresolved existing-resource references (ambiguous → retargeting options over the candidates; missing → informational, no operation exists to block), and policy conflicts (speculative application + the document's own policies through the ch. 8 pipeline via `@iap/sdk`). Questions are minimal, logically grouped, deterministic in order, and machine-answerable: closed options each carry a label, an impact explanation, and machine-executable `AnswerEffect`s (`no-change`, `remove-operation`, `acknowledge-destructive`, `retarget`, `amend-create`, `amend-set`, `add-operations`); every closed question names a recommended default. Each blocking question is attached to its operations' `requiredClarifications` (the existing M3.1 `Clarification` shape), so the gate keeps the batch uncommittable until answered (OP-3). **`applyClarificationAnswers`** executes the chosen effects on the PROPOSAL (never the document), clears answered questions, and emits `ConfirmationRecord`s with channel `confirmed-clarification` — only for operations whose questions are ALL answered (partial answers keep blocking). **`requiredConfirmations`** reports (data, not records) which operations still need a human confirmation and why.

### M3.3 — provenance flow and semantic explain

- **§3.4 provenance flow wired end to end** over the closed M3.1 vocabulary: explicit user input compiles with source `explicit-user`; clarification answers confirm on channel `confirmed-clarification` (the gate upgrades the per-field source — M3.1 behavior, now exercised by the full flow); **`src/recommend.ts`** is the `accepted-recommendation` seam — deterministic rules (`explicit-backup-for-stateful`: backup `none` on Database/Volume or non-required ObjectStore backup; `gateway-tls-minimum`: Gateways below TLS 1.3) produce OFFERED `Recommendation`s, and `acceptRecommendations` is the only path into operations: accepted envelopes carry provenance source `accepted-recommendation` plus a confirmation on the same channel. IEP-0013 TB-3 is modeled now: `Recommendation.knowledgeSnapshotIds` with origin `mcp` MUST be non-empty at acceptance (fails closed with `TypeError`); real MCP retrieval arrives Phase 12.
- **`src/explain.ts`** — `explainBatch(document, batch, options)` renders the §3.5 "Explain what changes this request will make" view: a deterministic, human-readable adds/changes/removes narration over the M3.1 canonical-projection diff with old → new values, per-field provenance citing the writing operation and source, destructive classification with the acknowledgment requirement, and the assumption/clarification (OP-3) status of the proposal. Sync and pure — a speculative copy is diffed and discarded; the output is prose, never a document serialization. Unexplainable proposals return the gate's refusal shapes.

### M3.4 — model-provider abstraction

- **`src/adapter.ts`** — the vendor-neutral `ModelAdapter` interface: `{ id, version, residency, prompts?, extract(request, context) → Promise<ExtractionResult> }`. No vendor SDK exists in-tree and the interface makes NO network assumption — an out-of-tree vendor adapter is any package implementing this interface, referencing its prompt artifacts by exact id@version from the registry, declaring its data residency, and returning facets (validated against the facet schema) and optionally a direct proposal batch (validated via the M3.1 `validateBatchStructure`). `createAdapterSession(adapter, config)` is the enforcement middleware every adapter is driven through: **structured-output enforcement with bounded retry/repair** (invalid output goes back to the adapter with the validation issues via `context.repair`, at most `maxAttempts` times; repair means RE-validation, never auto-acceptance; exhaustion refuses), **token/cost limits** (counts supplied by the adapter as integers, limits enforced by the middleware; per-request token ceilings plus a cumulative session cost ceiling; configured limits with missing counts fail closed), **redaction hooks** (caller-registered functions scrub the request before ANY adapter sees it), and **data residency** (a session allowlist refuses non-conforming adapters BEFORE invocation). The closed refusal taxonomy is `ADAPTER_ERROR_CODES` (six codes).
- **`src/prompts.ts` + `prompts/`** — the prompt registry: versioned prompt ARTIFACTS as byte-pinned data files (`extract-facets` v1 — the extraction prompt an out-of-tree LLM adapter would use, encoding the ch. 19 §19.7 never-invent/ask-don't-guess duties and the unparsed/unsupported reporting obligations; `repair-extraction` v1 — the bounded-repair instructions), each pinned by a SHA-256 content hash in the manifest. Lookups are EXACT-version only (`getPrompt(id, version)`; no "latest" exists in any deterministic path) and fail closed on unknown versions and on any hash mismatch. The `prompts/` directory is excluded from reformatting (content-hash pins) and ships in the package files.
- **Two in-repo adapters** (phase-3 design decision 9): `fixtureAdapter(recordings)` replays recorded `ExtractionResult`s keyed by request id (the evaluation/OP-2 workhorse; missing recordings fail closed through the middleware), and `rulesAdapter()` wraps the M3.2 deterministic extractor behind the same interface (`iap-rules@1`, residency `local`).

## Design decisions taken (spec ambiguities resolved)

1. **The facet schema is embedded-only.** Facets are the reference toolchain's internal adapter contract, not a document companion artifact; publishing `intent-facets-v1` under `spec/schema/` would freeze extraction vocabulary as if normative. The schema lives as data in `facets.ts`, compiled under the same strict-ajv-with-x-iap configuration as every other schema. `compiler-operations-v1.schema.json` remains the only compiler companion artifact.
2. **Confidence aggregation is `min` over contributing facets.** A chain of evidence is only as strong as its weakest link, and `min` is a comparison — no float accumulation feeds any deterministic path. Extraction confidences are constants from the closed three-tier table; the `inferred-association` tier (0.7) deliberately sits below the 0.8 gate threshold, so extractor-inferred structure (derived edges, gateway routing, identity attachment) can never commit without a human confirmation (OP-3 by construction).
3. **Derived resource identifiers are addressing, not intent.** Generated ids (`db`, `cache`, `edge`, deterministic collision suffixes) surface in the preview diff and are not recorded as assumptions; assumed VALUES are. This keeps the assumption channel meaningful: every entry is a semantic default a human must accept.
4. **The twenty-facet roadmap list is extended by exactly one facet (`removal`)** to carry the §3.5 removal directive; relationship phrasing rides on `networking` (the roadmap's "networking intent"), and "Reduce expected cost" rides on `budget` (`reduce: true`) — resolved by the clarification engine, never compiled into guessed operations.
5. **Compliance controls are closed per-framework policy sets** derived from the normative ch. 3 defaults (encryption-at-rest/in-transit, no-public-data-stores, backup-required), with policy ids slugged to the DNS-label grammar (`pci-dss-encryption-at-rest` — dots are not label characters) and ch. 7 effect semantics applied correctly: `require` for conditions that must hold, `deny` for rules describing the forbidden state. Existing policy ids are skipped (re-running a compliance request is idempotent).
6. **Budget and provider preference are non-semantic annotations.** Neither is expressible as core intent in v1; both are recorded under `metadata.annotations` (`budget-monthly-usd`, `provider-preference`), which ch. 19 §19.5 guarantees no conforming tool branches on. The budget still drives the deterministic cost/availability clarification (`HA_DATABASE_BUDGET_FLOOR_USD = 400`, an exported integer constant compared, never computed).
7. **Named provider regions and provider products are unsupported findings, not translations.** §19.3 forbids resolving mappings; the extractor reports `dynamodb` with the SUGGESTION `Database (class: key-value)` as prose and never emits the facet itself. Multi-region intent maps to `availability: maximum` (§3.2.1 — "MUST be realizable across regions"), the one geography the core vocabulary expresses.
8. **Questions carry machine-executable effects.** The §3.3 requirement that questions be answerable through UI controls and text alike is met by making each option's consequence data (`AnswerEffect`): retargeting, proposal amendment, operation removal, destructive acknowledgment, or adding pre-built operations (the "Reduce expected cost" options carry deterministic update sets computed from the document). Answer application edits the PROPOSAL only; the document is reachable solely through the gate.
9. **Required-field gaps block via validation, clarification-blocked ops via OP-3 — both fail closed.** A workload without an artifact is uncommittable twice over: the attached clarification refuses via `unconfirmed-clarifications`, and even a confirmation cannot force it because the dry run fails (`validation-failed`) until the answer supplies the artifact. The engine never invents a placeholder reference.
10. **Removal compiles its blast-radius cleanup deterministically** (inbound edges, Application `components` membership — arrays replace wholesale per ch. 6 merge semantics). What it cannot express in the closed vocabulary (e.g. top-level `outputs` referencing the removed resource — no output operation exists in the twelve) is left to the dry run, which refuses fail-closed with the IaP finding.
11. **The OP-1 pinned-surface test was extended ADDITIVELY — documented deviation.** The M3.1 conformance test pins `Object.keys(compiler)` exactly; adding any authoring export requires extending the pinned list. All fifteen M3.1 names remain, the no-byte-path audit now also covers the new functions (facets, batches, questions, recommendations, registry entries, and prose only — `explainBatch` output is asserted non-parseable as a document), and no M3.1 assertion was weakened. Gate semantics (`apply`/`replay`, schema shapes, refusal taxonomy) are byte-untouched.
12. **Residency is declarative and enforced structurally.** Adapters MUST declare `residency`; the middleware refuses a non-allowlisted declaration BEFORE the adapter (or the redacted request) is touched. The in-repo adapters declare `local` — the local-model option is the degenerate case of an interface that never assumed a network.

## Next milestone

M3.5 (+M5.3) consumes this surface directly:

- **Benchmark harness** (§3.7): `createAdapterSession(rulesAdapter())` / `fixtureAdapter(recordings)` for the two evaluation paths; `extractRules` output (`facets` for extraction-correctness scoring, `unparsed` for coverage, `unsupported` for unsupported-feature detection); `compileFacets` (`batch.operations` for resource/relationship extraction scoring, per-op `assumptions` for false-assumption counting); `clarify().questions` with the closed `trigger` field for clarification precision (fire/no-fire per crafted case, as the trigger tests already demonstrate); `apply`/`replay` for validity, semantic equivalence against the official examples, and deterministic re-serialization after confirmation. Recordings for examples beyond the rules extractor's vocabulary are `ExtractionResult` fixtures validated by `validateExtractionStructure`.
- **`iap create` (M5.3)**: the interactive loop is `session.extract → compileFacets → clarify` (render `ClarificationQuestion.options` with recommended defaults and impacts as prompts), `applyClarificationAnswers` for `--answers`/`--yes-defaults` (choose each `recommendedOptionId`), `requiredConfirmations` to build the explicit `--yes` confirmation records, `explainBatch` for the preview step, and `apply` as the only write path. `--adapter` selects `rulesAdapter` (default) or `fixtureAdapter`; `promptRegistry`/`getPrompt` supply the versioned prompt artifacts to out-of-tree adapters.

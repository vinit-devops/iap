# Milestone M5.3 — `iap create` (Natural-Language Authoring at the CLI)

**Phase:** 5 — Reference CLI
**Milestone:** M5.3 (`iap create`, natural-language flow — was gated on Phase 3)
**Status:** Completed
**Date:** 2026-07-11

## Implemented

`iap create` (`packages/cli/src/commands/create.ts`) — the reference CLI's natural-language
authoring command (ch. 22 "Natural-language creation"), unblocked by the Phase 3 authoring
engine. It is a thin, non-interactive shell over `@iap/intent-compiler`'s
`runAuthoringSession` (M3.5): accept a natural-language requirement, run the compiler,
surface any clarifications, show a semantic preview, and — only when the request fully
commits — write `infrastructure.iap.yaml`. The command holds no authoring logic of its
own; the normative boundary is the engine's (an LLM never writes YAML; the gate produces
the document after the whole ch. 8 pipeline passes, OP-1), and the CLI still cannot deploy.

### Request input (§22 automation)

- **positional** (`iap create "a web app running …"`), **`--request`**, or **stdin**
  (piped requirement) — resolved in that order; an empty request is a usage error (exit 2).
  Stdin arrives through an injected `CliIO.readStdin` seam so the command stays testable
  in-process (the bin shim wires it to `process.stdin`).

### Non-interactive answering

- **`--answers '<json>'`** — a JSON array of `{questionId, optionId?, value?}` applied to
  the clarification questions; malformed JSON is a usage error (exit 2).
- **`--yes-defaults`** — resolve clarifications with their recommended defaults (only
  where the recommended option needs no value; free-form questions still stop the run).
- **`--acknowledge-destructive`** — the explicit human "yes" for destructive changes
  (irrelevant for a from-scratch create, wired for symmetry with the engine and future
  `iap edit`).

### Output and writing

- **Human** (default): request → extracted intent → unsupported findings → clarifications
  (each option with its id, impact, and the recommended marker) → the deterministic
  `explainBatch` semantic preview → and, on commit, `created <path> (<n> resources)`.
- **`-o json`**: one deterministic `formatVersion: 1` payload — `outcome`, `intent`,
  `unsupported`, `clarifications`, `answered`, `unanswered`, `preview`, `refusals`,
  `resources`, `canonicalHash`, `provenanceCount`, and `writtenTo`/`document`.
- **`--out <path>`** (default `infrastructure.iap.yaml`), refusing to overwrite an existing
  file without **`--force`** (exit 2, matching `iap init`); **`--stdout`** prints the
  document instead of touching disk; **`--name`** sets `metadata.name`.
- **`--timestamp`**/`--actor` inject the audit identity so machine output is fully
  reproducible; the timestamp never reaches the document (which carries no timestamps),
  so a committed document is byte-identical regardless of when it was created.

### Exit codes (§22.1)

`committed`/`explained` → 0 · `needs-input`/`no-operations` → 1 (the request needs the
author's attention: answer clarifications, or rephrase an unsupported request) · a
validation-refused batch → 1, an adapter/other refusal → 3 · usage errors → 2.

## Design decisions taken

1. **`create` is non-interactive by contract.** The reference CLI is scriptable and
   deterministic: no readline prompts, no TTY assumptions. A request that needs a human
   decision exits 1 and prints exactly what to supply (`--answers`/`--yes-defaults`);
   re-running with the answer commits. Interactive prompting is a UI concern (Phase 13/15),
   not the reference contract.
2. **Only `create` shipped; `edit` stays gated.** M5.3 is `iap create` (from-scratch
   authoring into a new document). Incremental authoring against an existing document
   (`iap edit`) reuses the same engine with a loaded base and is the next CLI step — it
   keeps its phase-3 stub until it lands with its own tests, so the milestone stays
   scoped.
3. **The clock lives at the CLI boundary, not the engine.** `runAuthoringSession` is
   clock-free; `iap create` reads `new Date()` once for the audit timestamp (overridable
   with `--timestamp`). Because the document has no timestamps, this never affects the
   written bytes — determinism of the artifact is preserved regardless.
4. **`CliIO` gained an optional `readStdin`.** A minimal, backward-compatible extension so
   stdin is both wired (bin shim) and injectable (tests); commands without it simply fall
   back to a usage error when no request is supplied on the command line.

## Tests added

**`packages/cli/test/create.test.ts`** (13 tests): a clear request writes a
`infrastructure.iap.yaml` that re-loads and re-validates green with the expected resources;
`--stdout` prints and touches no file; overwrite refused without `--force`, then overwrites
with it; stdin-sourced request; missing-required-field → exit 1, nothing written, question
shown; `--answers` supplies the value and commits; malformed `--answers` → exit 2;
unsupported request → exit 1 reporting capabilities, nothing written; `-o json` committed
payload (`formatVersion`, `outcome`, `resources`, `canonicalHash`, `provenanceCount`,
`document`) and needs-input payload (clarifications + unanswered ids); JSON byte-identical
across runs; no-request usage error; `create` present in help and `edit` still a phase-3
stub.

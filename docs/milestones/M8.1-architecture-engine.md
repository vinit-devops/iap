## Implemented

New package `@iap/architecture` 0.1.0 (`packages/architecture/`; deps: `@iap/model`, `@iap/graph`) — every diagram is a derived view of the canonical model (ch. 18 §18.1), never drawn:

- **`deriveView(model, view, options?)`** — the five standard views of ch. 18 §18.2 as pure functions of a `CanonicalModel`:
  - **architecture** — every semantic edge (all verbs except `dependsOn`); `Application` resources become containers, never nodes; a resource in more than one Application lands in the lexicographically first.
  - **dependency** — the ch. 9 ordering DAG from `deriveOrdering` (`@iap/graph`), drawn in provisioning direction (before → after), no labels, no containers.
  - **network** — nested exposure trust zones public ⊃ internal ⊃ private (all three always emitted); node zones read from the **materialized** `spec.exposure` (default `private`); `connectsTo`/`routesTo` edges only, labeled `protocol/port` (+ `path`/`host` for routes, verb omitted).
  - **security** — `Identity`/`Secret`/`Certificate` nodes plus every endpoint of an `authenticatedBy`/`protectedBy` or access-carrying edge; edges labeled with the derived access level (`connectsTo (read-write)`); textual encryption badges (`atRest:required`, `inTransit:required`) from the materialized spec.
  - **application** — one subgraph per `Application` (`options.application` required, error otherwise): its components grouped, boundary-crossing endpoints rendered as dashed external nodes.
- **`ViewGraph`** — the JSON graph output: stable node ids (resource ids), stable edge ids (`source--type--target`, ordinal suffix on collision), clickable provenance (`sourcePointer: /resources/<id>` on every node), and a `specHash` content fingerprint per node that powers diff change detection. Deterministic: nodes sorted lexicographically by id, edges by (source, verb enum order, target, label) per ch. 4 §4.7 / ch. 18 §18.3.
- **`toMermaid(g)` / `toDot(g)`** (M8.2) — textual exporters emitting byte-identical source for identical canonical input; groups as `subgraph`/cluster blocks, network zones nested, badges as `«atRest:required»` label suffixes. **SVG/PNG are renderer concerns**: per ch. 18 §18.3 the conformance artifact is the textual source (diffable, reviewable, hashable); rasterize with any Mermaid/Graphviz renderer.
- **`diffViews(before, after)`** (M8.3 change overlay) — union graph marking every node and edge `added`/`removed`/`changed`/`unchanged` in its `style` field; Mermaid renders `:::added`-style classes plus cosmetic `classDef` lines, DOT renders color/dash attributes. This is the before/after plan visualization; the Phase 14 **drift overlay is deferred** but will reuse exactly this mechanism.
- **Filtering** — `options.filter` by `kinds` and `labels`; edges survive only when both endpoints survive; exposure zones persist (§18.2.3), other emptied containers drop.
- **No layout hints in semantic output** — a `ViewGraph` carries no coordinates, sizes, or renderer directives (tested: no `x`/`y`/`position`/`layout`/`width`/`height` key anywhere). Cosmetic layout hints would be a separate overlay artifact; styling stripped depicts the identical graph (Extension Non-Interference corollary).

## Architecture decisions

- **`specHash` on nodes**: diffing operates on `ViewGraph`s, not models, so each node carries a SHA-256 of its canonical content (kind/labels/spec/extensions) — invisible spec changes (e.g. `size`, `scaling`) surface as `changed` without leaking spec data into labels. Deterministic, therefore compatible with byte-identical rendering.
- **Zone containers as flat groups + fixed nesting in exporters**: `ViewGroup` has no parent field; the public ⊃ internal ⊃ private nesting is normative and fixed (§18.2.3), so the exporters encode it for `view: network` — group ids are namespaced `zone-<name>` to avoid collision with resource ids.
- **Diff status in `style`**: cosmetic-only channel; a diff-annotated graph stripped of styles is the plain union graph, honoring §18.3's cosmetic-only rule.

## Specification gaps

Ch. 18 §18.3's edge-label template omits `host` while §18.2.3 requires it on route labels — resolved in favor of §18.2.3 for the network view (host emitted), template order elsewhere. Worth a one-line spec clarification in a future editorial pass.

## Security findings

None. No new runtime dependencies; exporters emit inert text.

## Known limitations

- Mermaid reserves the bare word `end`; a resource named `end` (a legal dns-label) is emitted as `end_` in Mermaid ids only (labels unaffected). DOT needs no such mapping.
- The roadmap's provider-realized, data-flow, and HA/DR views need Phase 6/7 artifacts (mappings, plans) as inputs and are out of scope for this milestone; the drift overlay is Phase 14 (mechanism ready via `diffViews`).
- SVG/PNG emission is intentionally absent (renderer concern, ch. 18 §18.3); an interactive graph component belongs to the designer app (Phase 15).

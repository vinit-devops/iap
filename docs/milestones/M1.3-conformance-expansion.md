# Milestone M1.3 — Conformance Coverage Expansion (Gap Report §4)

**Phase:** 1 — IaP v1 Stabilization
**Milestone:** M1.3
**Status:** Completed
**Date:** 2026-07-10

## Implemented

Sixteen new conformance cases in `spec/conformance/cases/invalid/` (07–22), all **semantic** (`# expected: IISnnn`): each is schema-valid by design and MUST be rejected only by a full IaP validator with the named code. Coverage added per validation phase:

- **Phase 1 cross-field (IAP104):** inert `deadLetter.maxReceives` with `enabled: false` (07); `scaling.min > max` (08); Database `engine: postgresql` with `class: document` (09).
- **Phase 2 references (IAP202–IAP205):** dangling Application component (10); dangling `outputs.*.resource` (11); dangling Gateway `tls.certificate` (12); profile `extends` cycle (13).
- **Phase 3 relationships (IAP301/IAP302):** `routesTo` targeting a `Volume` (14); `path` attribute on a `connectsTo` edge (15).
- **Phase 4 graph (IAP402):** rule-edge selector matching zero resources (16).
- **Phase 5 policy (IAP501/IAP502):** `deny` condition true on a public ObjectStore (17); `require` condition false on a weakened `encryption.atRest` (18).
- **Phase 6 security (IAP601/IAP602):** public ObjectStore that is a `storesDataIn` target (19); `DB_PASSWORD` key in `configuration` (20).
- **Phase 8 extensions (IAP803):** `extensions.aws` asserting `exposure: public` on a private Database — fails the chapter 11 deletion test (21).
- **Post-merge schema (IAP101):** profile override sets required `Database.spec.class` to `null`; RFC 7386 deletes it, so the document is pre-merge schema-valid but post-merge invalid — requires a merge-aware validator (22).

`spec/conformance/README.md` updated: layout tree, case-expectation table (16 new rows), and expected-ajv-results list (`invalid/07`–`invalid/22` → valid under plain schema validation).

## Tests added

16 conformance cases; the spec harness (`pnpm run test:spec`) now runs **36 checks** (20 pre-existing + 16 new). No harness code changes were needed — it already asserts that every `# expected: IISnnn` case passes schema validation and every `# expected: schema-invalid` case fails it.

## Conformance status

Green: `all 36 checks passed`. Every new case validated as schema-valid on the first run — no expectation had to be downgraded to `schema-invalid`.

None.

## Specification gaps

Gap report §4 is now fully covered as written: IAP104 (all three listed variants), IAP202/203/204/205, IAP301/302, IAP402, IAP5xx (deny + require), IAP601/602, IAP803, and post-merge IAP1xx. Codes deliberately **not** covered:

- **IAP303** (advisory relationship finding) and **IAP503** (`warn` finding) — warnings, not failures; the invalid-case format asserts rejection, so they belong in `valid/` cases with expected-warning annotations once a semantic validator exists to observe them.
- **IAP604** (isolation unenforceable) and **IAP505** (budget exceeded) — plan-time errors requiring a selected mapping and cost annotations respectively; not expressible as a standalone document case.
- **IAP7xx** (framework violations) — require the registered compliance policy bundles (framework → policy set), which are engine artifacts, not document content.

These need Phase 2+ engines (semantic validator, planner, compliance bundles) before machine-runnable cases are meaningful.

## Known limitations

Semantic cases assert schema-validity only until the Phase 2 semantic validator exists; the declared `IISnnn` codes are design assertions the future validator's test suite must discharge (the harness currently defers them). The M1.2 document cited 18 harness checks; the harness had since grown to 20 (error-code registry checks), hence 36 rather than 34 after this milestone.

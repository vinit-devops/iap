### `tests/conformance/providers.mjs` — shared provider conformance runner (M6.4)

Wired to the pre-declared root script `test:providers`; exact `run.mjs` house style (check lines, failure counter, summary, exit 0/1). Requires `pnpm build` first (imports the built SDK/model/parser and each package's `dist/index.js`, same pattern as run.mjs for the validator). Per discovered `providers/*/manifest.json` (directories without one are skipped with a mid-build note):

1. **PC-1 on every run** — trust store from the package's `keys/*.public.pem` (keyId = filename stem), allowlist = exactly the manifest name, `loadProviderPackage` must succeed.
2. **Negative checks by in-memory tampering** (no file is ever written): a modified manifest field and a flipped `integrity.digests` value must fail `verifyManifestSignature` (loader stage-3 refusal — the signature covers the digest table), and byte-flipped artifact content must break its pinned digest (stage-4 refusal).
3. **PC-2** — every `*.case.yaml` under the manifest's `conformanceCases` directory runs through `evaluateConformanceCase` with the attestation registry resolved from the package's built module: canonical export `createAttestationRegistry()`; any `create*Attestation*` factory or exported registry object is accepted (the aws package exports `createAwsAttestations()`).
4. **PC-3** — per corpus document referenced by each case (including per-assertion overrides): two full parse → canonicalize → map runs must produce byte-identical plans with equal `planHash` (or byte-identical diagnostics for rejected documents).
5. **Cross-target equivalence (phase 6 exit criterion 1)** — when both `providers/aws` and `providers/kubernetes` load: `spec/examples/basic-webapp.iap.yaml`, canonicalized with the production profile, must map through both packages with zero diagnostics and non-empty plans (currently 12 aws / 16 kubernetes plan resources). Skipped with a note while either is absent.

## Design decisions taken (lifecycle semantics and their spec basis)

1. **Substrate scope = one deployment target.** Managed objects absent from the executed plan classify `delete` (§14.2's delete row, with the plan standing in for the canonical document). Unmanaged (out-of-band) objects are invisible to the diff and to convergence checking — they are the import lifecycle's raw material.
2. **Diff taxonomy per §14.2.** `create` / `update` (all changed attributes mutable) / `replace` (an immutable-after-create attribute changed); attribute-equal ready objects are no-ops excluded from the execution graph; a `failed` object retries as `update` even when attribute-equal (§14.2: retry of the same intent, never no-op).
3. **Immutability tables.** Plans carry neutral-empty `lifecycle` in mapping v1 (M6.1 decision 4), so the mock defines `MOCK_REPLACE_ON` — the provider-level projection of ch. 3 per-kind lifecycle rules (`Store.engine` mirrors "Database.spec.engine is immutable-after-create") — and additionally honors any `lifecycle.replaceOn` a plan resource carries.
4. **Replace = successor identity.** §14.2's "create successor, rebind, delete predecessor" is modeled as a fresh sequence + freshly generated outputs (observably new `id`) with `generation` recording lineage; `update` keeps identity and increments `generation`.
5. **Stateful-kind replacement authorization is planner logic.** §14.2's "stateful kinds MUST NOT be auto-replaced" gates plan _construction_; the executor is the engine below that gate and applies the plan it is given. The boundary is documented rather than duplicated.
6. **Verify nodes.** All impacted dependents of changed nodes join as `verify` (readiness + attribute re-check) — a documented simplification of §14.3, which lets dependents of `update-in-place` verify and only promotes dependents of `create`/`replace` on bound-output changes.
7. **Wave schedule.** Longest-dependency-path waves, lexicographic within a wave (§14.4); ordering through excluded no-op nodes is preserved by transitive-edge insertion (§14.3). Delete waves run first, ordered in **reverse** among themselves using dependency edges stored at apply time (§14.3: a deleted resource is removed only after every deleted resource that depends on it).
8. **Halt-wave (§14.7).** Same-wave operations run to completion after a failure; not-yet-started transitive dependents are cancelled and stay pending; independent branches complete; the failed object is set `failed` and the outcome is `partial`. A failed create leaves a failed placeholder so the substrate "exactly describes reality"; **recovery is re-execution** — the next run's diff emits exactly the unfinished work (resume _is_ re-plan).
9. **Import.** `create` over an existing unmanaged object fails closed (`exists-unmanaged`); `importObject` adopts it (marks managed, reports drifted attribute _names_), after which the normal diff applies. Importing a missing or already-managed object fails closed.
10. **Secret hygiene (CE-6).** Sensitivity is the union of the plan resource's `sensitiveFields` (honored although neutral-empty in v1 plans) and the provider's own `MOCK_SENSITIVE_ATTRIBUTES` (SecretBox `value`). Every externalized surface — operation results, logs, read views, snapshots, convergence differences — substitutes `[REDACTED]`; convergence differences name attributes only, never values.
11. **Determinism.** The substrate's only "environment" is an injected sequence counter and the injected failure plan; the execution result (waves, operations, log, state hash) is byte-reproducible across substrates.
12. **`expect: unsupported` vs uncovered kinds.** Under the locked SDK evaluator, any mapping failure — including `unsupported-kind` — yields verdict `rejected`; `unsupported` is reserved for capabilities on successfully mapped resources with no registered attestation. The corpus therefore models the uncovered-kind exercise (`Cache`) as `expect: rejected` and the `unsupported` exercise as a deliberately unattested capability (`resilience.backup`).
13. **Corpus documents referenced under a case profile must define it.** The evaluator canonicalizes _every_ document a case names — per-assertion overrides included — against the case's profile, so `database-maximum.iap.yaml` carries a `production` profile (with an availability-neutral override).
14. **Attestation export convention.** Canonical: `createAttestationRegistry(): AttestationRegistry` from the package's built `dist/index.js`; the runner also accepts any `create*Attestation*` factory or a directly exported registry, duck-typed on `.lookup` (module-identity-safe).

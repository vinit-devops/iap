### `risk.ts` — risk rule table v1 (IEP-0011 risk scoring)

`riskRuleTableV1` replaces the M7.1 stub as the default `RiskAnnotator`: a pure function of the plan content (everything except `risk` itself), using ordinal integer weights only — no floating point anywhere in the hashed path. `ACTION_WEIGHTS` is total over action × reversibility (combinations the lifecycle table never emits still have defined weights, so the annotator is total over any schema-valid content): update-in-place 1 ≪ create 2 ≪ import 3 ≪ replace (replacement-based-recovery 15, reversible-with-data-risk 30) ≪ delete (replacement-based-recovery 20) ≪ manual-recovery-required 40 ≪ irreversible 60 — delete-of-stateful ≫ replace ≫ import/create ≫ update-in-place, weighted by the ch. 14 §14.6 reversibility class. One boundary factor (`security-boundary-change`, weight 5 per entry) covers every exposure/encryption-affecting change the plan surfaces in `deltas.security`. Factors are machine-readable (`id`, `weight` = unit × matched resources, sorted `resources`), sorted by id; the score is their integer sum; classification is a pure function of the score with explicit thresholds (`RISK_CLASS_THRESHOLDS`: medium ≥ 25, high ≥ 60, critical ≥ 100 — one irreversible delete is `high` on its own, two are `critical`).

### Rollback limitations + approval gates (`plan.ts`)

- `rollback.limitations` — one entry per **irreversible** action (delete of a stateful resource: no restore source is visible to the plan), with a deterministic reason string citing §14.6. Non-empty exactly when an irreversible action exists — unsupported rollback is explicitly reported (exit criterion 4), never discovered at rollback time.
- `approvalsRequired` — every destructive action carries exactly one gate from the closed vocabulary `stateful-replace | stateful-delete | destructive-replace | destructive-delete` (`approvalGateOf`, total over action × reversibility): the data-bearing reversibility classes map to the human gates of ch. 19 §19.6 (stateful replacement) and ch. 14 §14.6 (explicit data-loss acceptance); the replacement-recoverable classes map to the plain destructive gates. Sorted by (resource, gate).

### `envelope.ts` — plan envelope creation, signing, and PL-2 invalidation

- **`signPlan(artifact, {createdAt, expiresAt, privateKeyPem, keyId})`** — both timestamps are INJECTED RFC 3339 strings (the policy engine's injected `now` precedent); nothing reads a clock. Signing reuses the ed25519 + canonical-serialization semantics of `@iap/provider-sdk`'s manifest signing (node:crypto, PKCS#8 PEM, base64, one canonical byte form), lifted rather than imported because the SDK's signing surface is manifest-shaped. **Signed bytes:** the canonical serialization of `{ createdAt, expiresAt, planId }` (`planSigningBytes`) — planId is the SHA-256 of the canonical content serialization, so the signature binds the timestamps to the exact content bytes while making expiry tamper-evident (extending `expiresAt` breaks the signature). Fail-closed: refuses to sign a planId that does not match its content, a non-ed25519 key, or a non-increasing timestamp range. The envelope is never hashed — planId and content are byte-identical before and after signing (§14.5).
- **`verifyPlan(artifact, {desired, state, identities?, now, trustStore?, requireSignature?})` / `refuseIfInvalid`** — the engine-side PL-2 rule, enforced by construction for any future engine: recompute the identities of **all nine** determinism inputs from the CURRENT artifacts (via the same `deriveDeterminismInputs` the `plan()` facade uses, after verifying the current provider plan's `planHash` and the snapshot's `integrity` exactly as `buildPlan` does) and refuse on any mismatch. Closed refusal taxonomy (`PLAN_REFUSAL_CODES`): `schema-invalid`, `plan-id-mismatch` (content tampered after hashing), `inputs-hash-mismatch` (recorded identities don't fold to the recorded hash), `identity-mismatch` (per element, with the element named), `state-advanced` (revision moved past the plan), `expired` (injected `now` ≥ `expiresAt`), `unsigned`, `signature-invalid` (unknown keyId, wrong algorithm, bad signature, or tampered envelope). All refusals are collected in one pass — operators see the complete invalidation surface. Re-planning is always the remedy; there is no plan patching.

### M7.4 — CLI `iap plan` unlock (phase-7 design decision 9)

`iap plan --file <doc> [--profile <name>] --mapping <package-dir | artifact.iap-map.yaml> [--keys <dir>] [--state <snapshot.json>] [--out <plan.json|yaml>] [-o human|json]` — the `plan` stub is removed; every other engine-gated stub is untouched. Pipeline: `@iap/sdk` load + canonicalize → `applyMapping` → `plan()` (`emptySnapshot()` when `--state` omitted). A `--mapping` directory loads as a **verified provider package** (trust material from `--keys` or the package's `keys/*.public.pem`, keyId = filename stem — the harness convention; allowlist = the manifest's own name); a bare artifact is schema-validated and coverage-tiling-verified (the loader's checks minus signature/digests, which a bare artifact cannot carry). Human rendering shows waves with per-entry action/reversibility/destructive marks, `changedBy` (why the resource is scheduled) and per-field mapping sources (why each field has its value), destructive actions, unknown values, risk factors, honest deltas, rollback limitations, approval gates, and the planId; `-o json` emits the artifact verbatim (§22.1 — it carries its own `apiVersion`); `--out` writes canonical JSON bytes (or a YAML projection for `.yaml`/`.yml`). The CLI remains structurally unable to execute anything: `plan` imports no execution machinery and `deploy` stays phase-gated.

## Design decisions taken (spec ambiguities resolved)

1. **Risk class is a pure function of the score alone.** The schema says "deterministic classification of the score"; thresholds are explicit exported constants. Weights and thresholds are part of rule table v1: changing any of them requires a `PLANNER_VERSION` bump (decision 8 of M7.1, now exercised: 0.1.0 → 0.2.0).
2. **Approvals never derive from the risk class.** `RiskInput` includes `approvalsRequired`, so gating on the computed class would be circular. The deterministic rule is: every destructive action ⇒ exactly one gate (IEP-0011: every `destructiveActions` entry carries its required approval gate); non-destructive actions ⇒ none. The brief's "and/or risk class ≥ high" alternative is intentionally not taken.
3. **Statefulness in gates is read off the reversibility class.** Plan content does not carry statefulness directly; the reversibility classes encode it (M7.1 reversibility table). `reversible-with-data-risk`/`manual-recovery-required`/`irreversible` ⇒ the `stateful-*` human gates of ch. 19 §19.6 and ch. 14 §14.6; `replacement-based-recovery`/`fully-reversible` ⇒ the plain `destructive-*` gates.
4. **§14.2's "stateful replace MUST fail the plan" is realized as the `stateful-replace` gate, not a planner throw.** Statefulness is _derived_ fail-closed (M7.1 decision 3), so refusing outright on a derivation would refuse legitimate stateless plans misclassified toward the destructive extreme. The plan artifact is where authorization attaches (IEP-0011 non-goal: approval workflow itself is Phase 16); a conforming engine must satisfy the gate before execution, and `approvalsRequired` makes that machine-checkable.
5. **Rollback limitations are exactly the irreversible actions.** `reversible-with-data-risk` replaces gate on approval but are not rollback-limited (a restore path per §14.6 exists); only irreversible actions (stateful deletes with no restore source visible to the plan) can never be undone by re-planning. Non-empty ⇔ an irreversible action exists.
6. **Signed bytes are `{createdAt, expiresAt, planId}` canonical form.** IEP-0011 requires envelope integrity via signature; signing only `content` (the schema's previous wording) would leave `expiresAt` tamperable without breaking the signature. Signing the planId binds the exact content transitively while protecting expiry. The schema's two signature descriptions were updated accordingly (documentation-only change, no shape change; embedded copy re-synced, drift test green).
7. **Expiry refuses at the boundary.** `now ≥ expiresAt` ⇒ `expired` (IEP-0011: "the envelope is expired"); comparison is integer epoch milliseconds parsed from validated RFC 3339 strings — envelope checks are outside the hashed path, and no wall clock is ever read (`now` is injected).
8. **verifyPlan separates plan invalidation from caller error.** Corrupt CURRENT inputs (provider plan failing its own `planHash`, snapshot failing its `integrity`, malformed `now`) throw TypeError exactly like `buildPlan`'s input verification; only properties of the _plan under verification_ produce refusals. A present envelope is always fully verified; `requireSignature` additionally refuses unenveloped artifacts (`unsigned`).
9. **`state-advanced` is a distinct refusal code.** An advanced revision also mismatches identity 8a, but the operational meaning ("the world moved on; re-plan") differs from generic identity drift, so both are reported.
10. **multi-region's golden is a documented deterministic rejection** (see above): the honest `availability: maximum` rejection is kept exactly as the phase-7 design demands, the example has no profiles to pin around it, and PL-4's byte-exactness applies to the diagnostic bytes instead. All other eight examples commit plan goldens.
11. **Golden identity pins.** `profileHashes` (identity 2) = sha256 over the canonical serialization of the named profile definition in the authored document — deterministic, key-order independent, and computed identically by the golden runner and the CLI. `policyBundles`/`extensionVersions` pin `{}` and `discoverySnapshot`/`pricingSnapshot` pin `null` in `tests/determinism/manifest.json`: the plan pipeline loads no policy bundles or extension packages (document-declared `extensions` blocks are document content hashed into identity 1, not toolchain packages), and no snapshots exist before Phases 10/12.
12. **§24.4 perturbation runs in-process.** Run B mutates TZ/locale/env/cwd around a full pipeline re-run and restores them; a different machine/OS is CI's dimension, not the runner's. The key-shuffle run re-serializes the parsed document with every object key reverse-sorted and re-enters the pipeline from text, proving order-independence end to end (not merely at the canonicalizer).
13. **CLI exit codes** (ch. 22 §22.1 applied to plan): missing `--mapping`, missing `--profile` when the document declares profiles, and an unusable/unverifiable mapping argument are usage errors (exit 2 — the input artifact cannot be trusted or resolved); a non-conforming document (CP-4, including deny-gating policy findings per ch. 19 §19.6) and mapping diagnostics are findings (exit 1); planner input refusals (e.g. corrupt `--state` integrity) are operation failures (exit 3).
14. **Mock realize-rule order preserved.** The Database relational rule stays `realize[0]` (new timeseries/key-value rules follow it), keeping the PC-4 tamper fixtures and first-match-wins semantics of the existing corpus intact; `spec.class` is required with a constrained 3-value domain, so the three rules tile exactly without a default rule.

# Milestones M2.4 + M2.5 — Semantic Validators (Phases 1–4) and Graph Engine

**Phase:** 2 — Canonical Model and Reference SDK
**Milestones:** M2.4 (`@iap/validator`) + M2.5 (`@iap/graph`) — combined: reference/relationship/dependency validation is the graph engine's constraint surface (phase-2 design, wave 2)
**Status:** Completed
**Date:** 2026-07-10

## Implemented

**`@iap/graph` 0.1.0** (`packages/graph/`, depends on `@iap/model` only):

- `buildGraph(resources, edges)` — typed graph over the normalized edge set (ch. 4 §4.7 step 6 output): node → kind map, outgoing/incoming indexes, edges-by-verb index. Tolerates dangling endpoints (those are phase 2's concern).
- Normative constraint tables, encoded exactly per spec: `VERB_TARGET_KINDS` + `verbKindViolation()` (ch. 4 §4.3.1 — closed target lists for `routesTo`/`publishesTo`/`consumesFrom`/`storesDataIn`/`authenticatedBy`/`monitoredBy`, same-kind rule for `replicatesTo`, non-addressable exclusions for `connectsTo`, Application-only-on-`dependsOn` rule) and `EDGE_ATTRIBUTES_BY_VERB` + `attributeViolations()` (ch. 4 §4.4 — `port`/`protocol` on connectsTo+routesTo, `access` on connectsTo+storesDataIn, `path`/`host` on routesTo only, no attributes on dependsOn/publishesTo/consumesFrom).
- `deriveOrdering(graph)` — ch. 9 §9.2: _target before source_ for every verb except `replicatesTo` (no arc) with `dependsOn` as the pure ordering arc; same-pair constraints collapse to one arc (rule 4).
- `detectCycles(ordering)` — iterative Tarjan SCC; one full cycle path per non-trivial SCC plus one per self-loop, deterministic (lexicographic visit order, cycles anchored at their smallest member).
- `executionWaves(graph)` — Kahn topological layering with lexicographic tie-break (ch. 9 §9.4); throws on cyclic input (a planner must never receive a cyclic graph).
- Queries: `dependents(graph, id)` (transitive impact set), `dependencies(graph, id)`, `pathExists(graph, from, to)` over the derived ordering relation.

**`@iap/validator` 0.1.0** (`packages/validator/`, depends on `@iap/model`, `@iap/parser`, `@iap/graph`):

- `validateDocument(text | parsedDoc, {profile})` → `{findings, phases: {schema, reference, relationship, dependency}, ok}` implementing ch. 8 phases 1–4 with registry-coded findings, collect-all within each phase, and the between-phase gate (later phases run only when earlier phases are error-free; warnings never gate). Finding order is deterministic: phase, then path, then code (ch. 8 §8.6).
- **Phase 1 (IAP1xx):** pre-merge parse + schema validation reusing `@iap/parser` (`loadDocument`/`validateSchema`, shared compiled ajv instance) with the apiVersion gate; **post-merge re-validation** of the profile-merged document (ch. 8 §8.3 — findings carry a `post-merge:` path prefix and name the profile); the normative IAP104 cross-field rules (inert `deadLetter.maxReceives` with `enabled != true`, `scaling.min > scaling.max`, Database engine/class consistency per ch. 3); and the IAP105 banned provider-term scan over free-string core positions with the ch. 24 §24.3 whole-token rule and exemption list (`extensions`, `x-*`, `description`, `metadata.annotations`, `artifact.reference`).
- **Phase 2 (IAP2xx):** dangling inline and rule-edge targets (IAP201), dangling or self-referential Application components (IAP202), dangling output resources (IAP203), Gateway `tls.certificate` dangling or wrong-kind (IAP204), and profile reference errors (IAP205: unknown selected profile, unknown `extends` target, `extends` cycle — checked over _every_ declared profile so lint-only runs still reject a broken profile graph; the actual merge reuses `@iap/model`'s `mergeProfile`).
- **Phase 3 (IAP3xx):** the normalized edge set (reusing `@iap/model`'s `flattenEdges`) checked against the `@iap/graph` verb/kind (IAP301) and attribute/verb (IAP302) tables, plus the route-less-Gateway advisory warning (IAP303).
- **Phase 4 (IAP4xx):** ordering cycles with the full cycle path in the message (IAP401, via `deriveOrdering` + `detectCycles`), zero-match rule-edge selectors (IAP402, surfaced by `flattenEdges` during phase 3 and routed to phase 4 per the code's phase ownership), self-referential ordering edges reported distinctly (IAP403).

**Harness upgrade** (`tests/conformance/run.mjs`): semantic conformance cases are now **executed, not just declared**. A new `semantic validation (phases 1-4)` section dynamically imports the built `@iap/validator` (a missing build artifact fails one check with a "run pnpm build first" message) and asserts every phase-1–4 invalid case actually produces its expected code, plus that the three valid cases pass phases 1–4 with zero errors. Profile-relative cases carry a machine-readable `# profile:` header (added to case 22, read generically, documented in `spec/conformance/README.md`).

## Architecture decisions

1. **The constraint tables live in `@iap/graph`, the findings in `@iap/validator`** — the graph package encodes ch. 4's matrices as data + pure predicates (`verbKindViolation`, `attributeViolations`) and the validator turns violations into registry-coded findings, so planners/diagrammers can reuse the same tables without importing the validation pipeline.
2. **Phase-3/4 findings point into the normalized `edges` array** (`/edges/<i>/…`), matching ch. 4 §4.7's rule that phases 3+ read only the normalized graph; original declaration paths do not survive rule-edge expansion and dedupe.
3. **IAP402 is computed during flattening but reported in phase 4**, per the registry's phase ownership — flattening happens once, at the start of phase 3.
4. **Profile reference checking covers all declared profiles**, not just the selected one, so lint-only invocations (no profile) still reject `extends` cycles (conformance case 13 carries no profile header). Each cycle is reported once, anchored at its lexicographically smallest member.
5. **Reuse over reimplementation:** `mergeProfile`, `flattenEdges`, and `compareCodePoints` come from `@iap/model`; schema validation comes from `@iap/parser` with one shared compiled validator.

## Security findings

None. The validator introduces no new dependencies (both packages depend only on workspace packages; `yaml` is dev-only for tests), keeping the no-provider-SDK exit criterion intact.

## Known limitations

- **IAP203 checks resource existence only.** Validating `outputs.*.attribute` against each kind's abstract attribute set (ch. 3) needs machine-readable per-kind output tables, which don't exist yet; deferred until they are published with the schema.
- **IAP205 for `overrides` containing a `profiles` key** is surfaced from `mergeProfile` and gates phases 3–4 like any phase-2 error, though the merge itself proceeds (the key is ignored per ch. 6 §6.4).
- **Phases 5–8 are not implemented** (policy, security, compliance, version/extension); `validateDocument` documents its four-phase scope and the harness defers those cases explicitly. IAP801 reserved-kind warnings (phase 8) are therefore not yet emitted for `valid/02`'s Alert resource — permitted, since warnings do not affect conformance.

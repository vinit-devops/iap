# Milestone M2.6 — SDK Facade, Round-Trip Serializer, Plugin Registration

**Phase:** 2 — Canonical Model and Reference SDK
**Milestone:** M2.6 (`@iap/sdk`) — wave 3 of the phase-2 design: the facade over everything
**Status:** Completed
**Date:** 2026-07-10

## Implemented

**`@iap/sdk` 0.1.0** (`packages/sdk/`, depends on `@iap/model`, `@iap/parser`, `@iap/graph`, `@iap/validator`, `yaml`):

- `load(source, {profile, sourceMap})` — the single entry point (ch. 21 §21.4). Accepts YAML/JSON text or `{path}` (file read; filename decorates findings). Parsing and Phase 1 schema validation run eagerly; the returned `IaPWorkspaceResult` carries `{document, findings, ok, sourceMap?}` plus five memoized methods, each a pure delegation to a component engine (per §21.4 the facade adds no behavior a caller could not reproduce directly):
  - `validate()` → `@iap/validator`'s `validateDocument` (ch. 8 phases 1–4, profile-relative);
  - `canonical()` → `@iap/model`'s `canonicalize` (CIM, canonical JSON, SHA-256 hash, findings) — **independent of validity** (phase-2 design decision 3): an invalid-but-parseable document still canonicalizes; only total parse failure makes it unavailable (`IaPError`, documented);
  - `graph()` → `@iap/graph`'s `buildGraph` over the canonical model's resources and normalized edges;
  - `waves()` → `executionWaves(graph())` (ch. 9 §9.4);
  - `serialize(format)` — the round-trip serializer: `'canonical-json'` (default) returns the canonical byte projection (C5+C6); `'yaml'` re-emits the **original, profile-unmerged** parsed document via the `yaml` package with key order as authored (insertion order of the C1 lossless lift; aliases stay expanded, which is hash-neutral). Round-trip guarantee, tested over all 9 official examples: `load(ws.serialize('yaml')).canonical().hash === ws.canonical().hash`.
- **Findings, not exceptions (§21.1.3).** Document problems are always findings; `IaPError` is reserved for SDK misuse: unknown serialization format, `canonical()`/`serialize()` with no parsed document, duplicate/empty extension namespace registration.
- **Plugin registration (§21.5, ch. 11).** `registerExtension({namespace, version, schema?})` registers an extension package exactly once — a second package claiming the same namespace is rejected with `IaPError` (normative MUST of §21.5); `unregisterExtension(ns)` and `registeredExtensions()` (registration order) complete the registry surface. `validateExtensions(document)` consults the registry: (a) namespaces present in the document but unregistered produce **IAP802 warnings** — never errors (ch. 11 §11.5) — one per namespace, anchored at the document-level registration when present; (b) registered namespaces with a `schema` have every resource-level `extensions.<ns>` block's content validated against it (compiled through `@iap/parser`'s `createValidator`, so the SDK adds no direct ajv dependency), violations reported as **error-severity IAP802** findings scoped inside the extension block.
- **API compatibility surface test** (`test/api-surface.test.ts`) — pins the module's exact export names (`IaPError`, `load`, `registerExtension`, `registeredExtensions`, `unregisterExtension`, `validateExtensions`), the `IaPWorkspaceResult` property set with and without `sourceMap`, and the exact method surface (`validate`/`canonical`/`graph`/`waves`/`serialize`), so any accidental removal or rename fails the suite (§21.6 stability contract: minors may add, never change or remove).

## Architecture decisions

1. **`serialize('yaml')` emits the profile-unmerged original, not the canonical document.** The round-trip exit criterion is about lossless re-emission of what was authored; the canonical projection already has its own byte format (`'canonical-json'`). Reloading the YAML with the same profile selected reproduces the profile-relative hash (tested).
2. **`graph()` is built over the canonical model** (post-merge, post-flatten edges), so the facade's graph is by construction the same one validation phase 3–4 reasoned about, and key-order independence is inherited from canonicalization.
3. **Extension sub-schemas compile through `@iap/parser`'s `createValidator`** — one ajv configuration for the whole ecosystem (draft 2020-12, strict, `x-iap-*` vocabulary), and the SDK's own dependency list stays at workspace packages + `yaml`.
4. **Module-global extension registry** with explicit `unregisterExtension` — registration is process-wide state per ch. 21 §21.5 ("a host registers with the SDK before parsing"); the unregister escape hatch keeps tests and long-lived hosts (LSP) able to reset without process restarts.

## Specification gaps

One, documented as a deliberate judgment call: the error-code registry has **no code for registered-but-schema-invalid extension content**. IAP802 is defined as "unknown extension namespace (warning)" and IAP803 as the non-interference violation — neither matches content that a _registered_ package's own sub-schema rejects (ch. 21 §21.5 says extension sub-schemas validate `extensions.<ns>` blocks but ch. 8 assigns no code). `validateExtensions` emits **IAP802 with severity `error`** and a message naming the schema violation, keeping the finding inside the IAP8xx extension range without inventing an unregistered code. A future IEP should add a dedicated code (candidate **IAP806 — extension content schema violation, error**) and this implementation will migrate to it.

## Security findings

None. The SDK's runtime dependency closure (audited via `pnpm ls --prod`) is: workspace packages + `yaml` (direct) + `ajv` with its four utility transitive deps (via `@iap/parser`). No provider SDK, no network access, no ambient lookups — `load({path})` reads exactly the file the caller names.

## Known limitations

- `validate()` covers phases 1–4 (M2.4 scope); phases 5–8 arrive with the policy/security/compliance engines. `validateExtensions` is a standalone helper (a phase-8 subset), not yet folded into `validate()` — that happens when phase 8 is implemented in full (non-interference, IAP801/IAP804/IAP805).
- `serialize('yaml')` re-emits parsed content: YAML comments and anchor/alias _syntax_ are not preserved (the expanded content is, so canonical hashes round-trip). Comment-preserving rewriting is LSP/CLI formatting territory (Phase 4/5), not canonical serialization.
- The ch. 21 §21.4 sketch methods that need later-phase engines (`plan`, `diagram`, `cost`, `evidence`) are absent by design; the facade grows with the phases that implement them (§21.6 allows minor additions).

# Milestone M6.3 — AWS Reference Provider Package (core certification)

**Phase:** 6 — Provider Mapping and Plugin Framework
**Milestone:** M6.3 (wave 2, parallel with M6.2+M6.4 and M6.5)
**Status:** Completed
**Date:** 2026-07-11

## Implemented

**`@iap/provider-aws` 0.1.0** (`providers/aws/`) — the first real provider package: a complete, signed plugin claiming certification level **core** (mapping-only per phase-6 design decision 7; execution handlers arrive with Phase 14). Built entirely on `@iap/provider-sdk` M6.1: loaded by `loadProviderPackage`, mapped by `applyMapping`, evaluated by `evaluateConformanceCase`, signed by `tools/provider-packaging/sign-manifest.mjs`. The illustrative `spec/mappings/aws-reference.iap-map.yaml` remains the spec's teaching example; this package supersedes it in scope under the same ch. 12 rules.

### Signed manifest (`manifest.json`)

`plugin.iap.dev/v1`, name `iap-provider-aws`, namespace `aws`, version 0.1.0, `specCompat ">=1.0.0 <2.0.0"`, `sdkCompat ">=0.1.0 <1.0.0"`, `certificationLevel: core`, `capabilities.kinds` exactly equal to the eight covered kinds (PC-5 — no inflated or partial claims, enforced by the loader in both directions), 8 pinned `sha256:` digests over every mapping/schema/conformance artifact, ed25519 signature under keyId `aws-test-2026`. The committed keypair in `keys/` is **test material only** (see `keys/README.md`); the shared conformance runner builds its trust store from `providers/<name>/keys/*.public.pem` with keyId = filename stem. Any artifact edit requires re-signing — a stale digest refuses the whole package at load (PC-1), proven by test.

### Mapping (`mappings/core.iap-map.yaml`) — realization choices

One artifact, eight kinds: the six production-slice kinds plus `Identity` and `Application`, which the acceptance document declares and which zero-diagnostic mapping therefore requires.

| Kind          | Targets                                                                      | Why                                                                                                                                                                                                                                                                                                                                               |
| ------------- | ---------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Service`     | `aws:ecs:Service` + `aws:elasticloadbalancing:TargetGroup`                   | Fargate task set behind an ALB target group (ch. 12 §12.7's typical realization); Service Connect supplies the addressable `endpoint`; size→cpu/memory tiers, scaling→desired/max capacity + target utilization.                                                                                                                                  |
| `Database`    | `aws:rds:DBInstance` + `aws:rds:DBSubnetGroup` + `aws:secretsmanager:Secret` | Relational-only RDS instance in private subnets; credentials generated into Secrets Manager — the Secret target exists because `connectionSecret` must bind to a produced target (ch. 12 §12.6). `availability` → `multiAZ`, `encryption.atRest` → `storageEncrypted`, `resilience.backup` → retention days (required 30 / preferred 7 / none 0). |
| `Cache`       | `aws:elasticache:ReplicationGroup` + `aws:secretsmanager:Secret`             | Redis-compatible replication group; `availability: high` → automatic failover + 2 cache clusters; AUTH token in Secrets Manager backs `connectionSecret`.                                                                                                                                                                                         |
| `Gateway`     | `aws:elasticloadbalancing:LoadBalancer` + `aws:acm:Certificate`              | Application load balancer; `exposure` → scheme (internet-facing/internal), `tls.minimumVersion` → pinned `ELBSecurityPolicy-TLS13-1-{2,3}-2021-06` policies (constants, never "latest"), `observability.logs` → access logs.                                                                                                                      |
| `Queue`       | `aws:sqs:Queue`                                                              | First-match-wins pair of rules: `ordering: fifo` → FIFO queue with content-based deduplication, default rule → standard queue; SSE-SQS for at-rest, TLS-only queue policy for in-transit.                                                                                                                                                         |
| `ObjectStore` | `aws:s3:Bucket`                                                              | Versioning status, full public-access block, SSE (`required` → `aws:kms`, `preferred` → `AES256`), TLS-only bucket policy.                                                                                                                                                                                                                        |
| `Identity`    | `aws:iam:Role`                                                               | Workload identity as an IAM task role trusted by `ecs-tasks.amazonaws.com`, consistent with the Service realization.                                                                                                                                                                                                                              |
| `Application` | `aws:resourcegroups:Group`                                                   | Tag-query resource group as the grouping construct backing the Application `identifier`.                                                                                                                                                                                                                                                          |

Every derive map is total over its supported domain and every abstract output attribute of every covered kind is bound (loader-verified tiling; also asserted directly in tests).

### Deliberate rejection surface (fail-closed, ch. 12 §12.3)

Everything below is **outside** the supports matrices and rejects loudly with a diagnostic naming the kind/field/value — never a silent downgrade:

- `Database` — `availability: maximum` (a single-region Multi-AZ instance cannot attest a ≥99.99% floor; the ch. 12 worked example, kept verbatim); `class` other than `relational`; `engine` other than `postgresql`/`mysql`; `exposure: internal`; `capacity.throughput` (field not covered — no IOPS floor attested).
- `Service` — `artifact.type` other than `container-image`, `runtime` other than `container` (ECS on Fargate runs container images only); `exposure: public` (public entry is the Gateway's job); `availability: maximum`; `spec.resources` and `spec.configuration` (fields not covered).
- `Cache` — `engine: memcached-compatible`; `availability: maximum`; `exposure: internal`.
- `Queue` — `delivery: exactly-once`; `deadLetter.enabled: true` (this core slice wires no redrive policy, so declaring it must fail rather than deploy a queue without the redrive the document claims).
- `ObjectStore` — `exposure: public` (no public buckets); `spec.lifecycle` (field not covered).
- Uncovered kinds (e.g. `Function`) reject with `unsupported-kind`.

### Extension schema (`schema/extension.schema.json`)

Validates `extensions.aws` content: document-level `version`, per-resource refinements under `resources.<id>` — `backupWindow` (`^\d{2}:\d{2}-\d{2}:\d{2}$`, the roadmap's example), `instanceClassHint` (`^(db|cache)\.[a-z0-9]+\.[a-z0-9]+$`), and `tags`. `additionalProperties: false` at every level; compiles under ajv 2020-12 strict (loader-enforced).

### Attestations (`src/index.ts`)

`createAwsAttestations()` returns the SDK `AttestationRegistry`: eleven pure predicates per (capability, target type) — the package's auditable claim of _how_ AWS resources realize each floor. `encryption.atRest` = RDS `storageEncrypted` / ElastiCache `atRestEncryptionEnabled` / SQS `sqsManagedSseEnabled` / S3 `sseAlgorithm`; `encryption.inTransit` = `requireSecureTransport` / `transitEncryptionEnabled` / `enforceTlsInTransit` / `enforceTlsOnly`; `exposure.private` = RDS `publiclyAccessible === false`, S3 `blockPublicAccess === true`; `availability.zonesMinimum` = Multi-AZ realizes 2 zones (`min` above 2 fails — the mapping cannot attest more). Supplied programmatically per M6.1 design decision 9.

### Conformance corpus (`conformance/`)

Three cases over three corpus documents (all conforming IaP documents):

- `webapp-core` — basic-webapp-derived production slice (Service, Database `availability: high`, Cache, FIFO Queue, ObjectStore): eight `expect: satisfied` assertions (at-rest/in-transit encryption on database, cache, and queue; `exposure.private` on database and bucket; `availability.zonesMinimum min: 2`) plus one deliberate `expect: unsupported` (`resilience.backup` on the bucket — no attestation registered, proving the verdict is reachable).
- `database-availability-maximum` — `expect: rejected` with `unsupported-value` naming `spec.availability` (the milestone's named rejection case).
- `function-unsupported-kind` — a `Function` document; `expect: rejected` with `unsupported-kind`. Note: an uncovered kind is a mapping _failure_, and the shared evaluator maps any mapping failure to the `rejected` verdict — `unsupported` is reserved for mapped-but-unattested capabilities, so the reachable-`unsupported` proof lives in `webapp-core` instead.

## Acceptance surface

The exit-criterion document: `spec/examples/basic-webapp.iap.yaml` canonicalized with the **production** profile maps with **zero diagnostics** into 12 plan resources across all seven document resources — the exact document the shared runner uses for the cross-provider equivalence check against `providers/kubernetes`. The supports matrices were built against the canonical form (schema defaults materialized — e.g. Database `size: m`, `exposure: private`, full `observability` blocks), not the raw document.

## Tests added

29 vitest tests in `providers/aws/test/` (suite scope: `pnpm vitest run providers/aws`):

- `package.test.ts` (10) — signed-package happy path (manifest fields, single mapping artifact); `capabilities.kinds` = covered kinds exactly (PC-5); every abstract output bound (CM-4); refusals: empty trust store (`signature`), allowlist miss (`allowlist`), tampered mapping bytes (`integrity`, digest mismatch — the stale-digest discipline), smuggled unsigned corpus file (`integrity`, no digest); extension schema accepts the documented shape and rejects malformed `backupWindow` and unknown properties.
- `webapp-acceptance.test.ts` (8) — zero-diagnostic mapping of canonical basic-webapp (production); the exact 12 logical ids; **provenance record for every desiredAttributes entry of every resource** (exit criterion 3); profile-merged derive values (`multiAZ: true`, `cpu: 2048`, `desiredCount: 2`, TLS 1.3 policy); complete output bindings for all seven resources; edge-derived sorted `dependsOn`; double-run byte-identical plans + equal `planHash` (PC-3); non-interference — canonical hash unchanged by mapping and frozen-model mutation throws (CM-6).
- `conformance.test.ts` (11) — all three cases pass end-to-end (PC-2) with exact per-assertion verdicts; rejected cases carry `unsupported-value spec.availability` / `unsupported-kind` detail; **PC-4 tampered-plan fixture** — weakening the `storageEncrypted` derive map flips exactly `db-storage-encrypted` to `violated` while sibling assertions stay satisfied; engine-level fail-closed diagnostics for `availability: maximum`, the `Function` kind, and `deadLetter.enabled: true`.

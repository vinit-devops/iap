## Key design decisions

1. **The plan artifact is `plan.iap.dev/v1`** per the IEP-0011 sketch, exactly: `planId` = SHA-256 of canonical `content` serialization; timestamps/signature live only in the `envelope` (never hashed); the nine-element determinism input vector is enumerated in `content.inputs` and folded into `inputsHash`. `spec/schema/plan-v1.schema.json` is a normative-adjacent companion (embedded copy in the planner, synced + drift-tested; compiled by the spec harness), and ch. 14 gains a reference per the IEP's schema-impact clause.
2. **The planner consumes provider plans, not documents.** Input = the provider-resource model produced by `@iap/provider-sdk`'s `applyMapping` (desired), plus a state snapshot (actual), plus explicit snapshot identities. The planner never parses YAML documents itself and never invokes mappings — layering follows ch. 14: document → (mapping) → provider model → (planner) → execution plan.
3. **State snapshots are a minimal IEP-0010 subset**: `{ revision, integrity, objects: { <logicalId>: { type, attributes, managed } } }` — enough to compute create/update/replace/import/delete without the Phase 14 state engine. Empty snapshot (revision 0) ⇒ everything creates; this is the golden-plan baseline. The full state document engine remains Phase 14 scope.
4. **Lifecycle determination is a closed rule order**: absent in state → `create`; present but unmanaged → `import` required before management (mirrors the mock provider's M6.2 semantics); present + any `lifecycle.replaceOn` field differs → `replace` (destructive); present + other attribute differs → `update-in-place`; in state but not desired → `delete` (destructive). Deletes schedule in reverse dependency order after the forward waves, matching the mock provider's executor.
5. **Cost/security/compliance deltas are honest**: cost = `unavailable` unless a pricing snapshot id is supplied (Phase 10 wires real pricing); security delta derives from exposure/encryption-relevant attribute changes present in the plan itself; compliance delta stays empty with an explicit `deferred: phase-11` marker. No fabricated numbers.
6. **Golden plans (PL-4) require widening the mock provider** to cover every kind used by the 9 official examples (its `mock:core:*` targets and derives are trivial). The mock keeps an honest rejection surface via values constraints (e.g. `availability: maximum` stays rejected on a data kind) and re-targets its uncovered-kind conformance case to a genuinely uncovered kind. Goldens live in `tests/determinism/golden-plans/` as canonical plan JSON, one per official example × mock mapping × empty-state snapshot × pinned input identities; the determinism runner re-plans twice (second run with perturbed environment per §24.4) and byte-compares.
7. **Plan signing reuses the ed25519 + canonical-serialization primitives** already proven in Phase 6 (exported from `@iap/provider-sdk` or lifted into the planner with identical semantics). Envelope verification + the PL-2 invalidation rule (recompute all nine identities; refuse on any mismatch, advanced state revision, expiry, or signature failure) ship as `verifyPlan`/`refuseIfInvalid` APIs — the "execution cannot proceed after inputs change" exit criterion is enforced by construction for any future engine.
8. **Risk scoring is a versioned pure rule table** (`riskRules: 1`, part of `plannerVersion` identity per IEP-0011 open question 2 — resolved: folded into plannerVersion for now, noted for the IEP): ordinal weights per action class (delete-of-stateful ≫ replace ≫ create ≫ update), per reversibility class, plus boundary factors for exposure/access-affecting changes. Deterministic classification into `low`/`medium`/`high`/`critical`.
9. **CLI `iap plan` unlocks** in wave 2: `iap plan --file <doc> --mapping <artifact|package-dir> [--profile] [--state <snapshot>] [--out plan.yaml]` — human-readable rendering + machine-readable artifact, exit codes per ch. 22; remains impossible to execute (no deploy engine until Phase 14).

## Exit criteria mapping

Byte-equivalent plans for identical inputs → golden suite + double-run (env-perturbed) determinism tests. Independence from YAML ordering → plan over key-shuffled documents hashes identically (canonicalization already guarantees the document side; test proves it end-to-end). Destructive changes clearly marked → PL-3 test: every replace/delete appears in `destructiveActions` with reversibility. Unsupported rollback explicitly reported → rollback.limitations populated for irreversible actions (delete of stateful kind without backup); test asserts non-empty. Execution cannot proceed after inputs change → PL-2 invalidation tests (each of the nine identities perturbed individually ⇒ refusal; expiry ⇒ refusal; bad signature ⇒ refusal).

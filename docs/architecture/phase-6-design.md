## Key design decisions

1. **Companion schemas are normative-adjacent artifacts** — `spec/schema/plugin-manifest-v1.schema.json` and `spec/schema/conformance-case-v1.schema.json` (IEP-0012 schema impact). Embedded copies in `@iap/provider-sdk` are synced by `tools/schema-generation/sync-schemas.mjs` and drift-tested by byte equality, same as `@iap/model` (ADR-0002 no-second-source). Both compile in the spec harness.
2. **Signature model with zero new dependencies** — ed25519 via `node:crypto`. Signing input is the manifest with its `signature` member removed, canonically serialized (sorted keys, UTF-8, no insignificant whitespace); artifact digests are `sha256:<hex>` over exact file bytes. Trust store = explicit `keyId → PEM public key` map supplied to the loader; publisher allowlist = explicit set of package names. Any failure (signature, digest, compat, allowlist, coverage tiling) refuses the package outright — no degraded load (IEP-0012 PC-1). A signing helper lives at `tools/provider-packaging/sign-manifest.mjs`; test keypairs are committed fixtures marked test-only.
3. **Mapping-time diagnostics have their own reason taxonomy, not IaP codes.** The ch. 8 registry covers document-validation stages; ch. 12 requires loud rejection but assigns no codes. The engine emits structured diagnostics `{ reason, kind, resourceId, field?, value?, verb?, message }` with the closed reason set: `unsupported-kind`, `unsupported-field`, `unsupported-value`, `unsupported-relationship`, `no-realize-rule`, `derive-map-gap`, `unbound-output`, `spec-compat`, `newer-minor-construct`. Formalizing these as registry codes is deferred to a future IEP (candidate noted in the completion report).
4. **Plan artifact shape** (roadmap "Mapping output" list): `formatVersion: 1`, `provider`, `mappingVersion`, `specVersion`, `profile`, `documentHash`, `inputs` (explicit mapping inputs, part of the hashed identity), `resources[]` — each with `type`, `logicalId` (`<resourceId>.<targetType>`, deterministic), `desiredAttributes`, `dependsOn` (derived from canonical edges), `lifecycle` (`createOnly`/`updateInPlace`/`replaceOn`), `sensitiveFields`, `provenance` (per attribute: `constant` | `from` | `map` + source field + rule index) — plus `outputBindings` and `planHash` (SHA-256 over the canonical serialization with `planHash` excluded). Double-run hash equality is a conformance obligation (CM/PC-3).
5. **Purity is enforced structurally** — the engine takes `(canonical model, mapping, { profile, inputs })` and touches nothing else: no clock, no network, no environment. The input model is deep-frozen; a before/after canonical-hash test proves non-interference (CM-6, exit criterion 4). Resource iteration is sorted by resource ID; rule evaluation is first-match-wins in document order (ch. 12 §12.4).
6. **Static tiling verification at load time** — the loader verifies `supports` ⊆ realize coverage (every in-matrix resource shape matches some rule), every `derive` map is total over the supported values of its source field, and every core-declared abstract output attribute of each supported kind is bound (ch. 12 §12.5). Defective packages fail at install, not at mapping time.
7. **Certification levels at Phase 6** — mock claims **execution** (in-memory substrate, full lifecycle, injectable failures); AWS and Kubernetes claim **core** (mapping-only; execution handlers arrive with Phase 14). No partial claims (PC-5).
8. **Attestations are pure JS predicates** registered per `(capability, targetType)`, exercised against tampered-plan fixtures to prove they can fail (PC-4). The assertion evaluator lives in the SDK; each provider package's vitest suite runs it over that package's `conformance/` cases, and `tests/conformance/providers.mjs` runs every package's corpus as part of `verify`.

## Exit criteria mapping

Same model → ≥2 targets: one shared corpus document mapped by both `providers/aws` and `providers/kubernetes` with both plans asserted. Unsupported intent fails explicitly: `expect: rejected` cases per package. Every parameter traceable: test asserts each `desiredAttributes` entry has a provenance record. Packages cannot modify the model: frozen-input + hash-equality test. Compatibility and signature checks enforced: loader refusal tests (bad signature, tampered artifact, out-of-range specCompat, missing allowlist entry).

## Key design decisions

1. **Operations apply to the authored document, not the derived CIM.** The source of truth is the document; profiles/defaults are derived. `apply(document, batch) → { document', logEntries }` is a pure function; the dry run canonicalizes the mutated document and runs the full ch. 8 pipeline (all phases via `@iap/sdk`/`@iap/validator`/`@iap/policy`); only the round-trip serializer writes bytes. This preserves IEP-0008 immutability (copy-on-write) while keeping provenance attachable at canonicalization, where per-leaf provenance already exists.
2. **IEP-0009 open question 1 resolved: explicit set/unset.** `UpdateResource`/`UpdateRelationship`/`SetMetadata` carry `change.set` (map of document paths → values) and `change.unset` (path list) — no merge-patch ambiguity, schema-enforceable, cleanly diffable. Q2: no `RenameResource` in v1 (rename = remove+create, ch. 13). Q3: confidence threshold default **0.8**, configurable per apply session; policy-driven thresholds deferred.
3. **Operation errors are a package-level closed taxonomy, not IaP codes.** Same precedent as mapping diagnostics (Phase 6) and plan refusals (Phase 7): ch. 8 owns validation-stage codes only. The gate's own refusals (`invalid-operation-type`, `dangling-target`, `id-grammar`, `batch-conflict`, `unconfirmed-assumptions`, `below-confidence-threshold`, `extension-namespace-violation`, …) are exported constants; registry registration goes to the standing candidate diagnostics IEP. Validation findings raised by the dry run keep their IaP codes untouched.
4. **The gate is airtight by construction (OP-1).** The only exported path that produces serializable bytes takes a validated, confirmed batch. Proposals (from any adapter or UI) are data; `apply` re-validates structurally (ajv strict against the companion schema), resolves targets, dry-runs, and refuses unconfirmed assumption-bearing or sub-threshold operations (OP-3). Batches are atomic — one failure aborts all (fail closed).
5. **Replayable log (OP-2).** Append-only entries record operation, envelope, confirmation (who/what accepted), and resulting document hash; replaying a confirmed batch against the same base document yields byte-identical output. Model-independence is a test: two different adapters emitting the same confirmed batch produce the same IaP.
6. **Provenance (§3.4/OP-4) is a closed vocabulary**: `explicit-user`, `confirmed-clarification`, `organization-profile`, `policy`, `iap-default`, `mapping-default` (reserved — mapping defaults exist right of the boundary and never flow back; recorded for completeness), `accepted-recommendation`. Every field written by an operation gets a record citing the operation id; defaults/profile materialization keep their existing canonicalization provenance.
7. **Clarification engine is deterministic rules, not model output.** The §3.3 trigger list is a closed rule set evaluated over proposals + the current document; each question is machine-readable (id, prompt, options with recommended default, impact note, the operation/field it blocks) so UIs and text flows render the same object. Answers become confirmations with channel `confirmed-clarification`.
8. **Destructive preview per ch. 14 §14.2**: `RemoveResource` targeting a stateful kind (Database, Volume, ObjectStore, Queue, Topic, Secret), or an update touching a field named in the resource's declared `lifecycle.replaceOn`, flags `destructive: true` in the preview diff and requires explicit acknowledgment — the authoring-side mirror of the planner's gates, computed without invoking the planner.
9. **No vendor SDKs in-tree; verify stays hermetic.** `@iap/intent-compiler` ships the `ModelAdapter` interface plus two deterministic adapters: **fixture** (replays recorded proposal batches — evaluation + OP-2 tests) and **rules** (a real, deterministic NL→operations extractor over the §3.2 facet vocabulary — the offline authoring path). Real LLM adapters are out-of-tree packages implementing the interface (documented); no network I/O exists anywhere in package code or tests. Prompt registry entries are versioned data artifacts consumed by out-of-tree adapters; redaction hooks and token/cost limits are middleware on the adapter boundary.
10. **The compiler structurally cannot deploy (exit criterion 5).** `@iap/intent-compiler` may depend only on `@iap/model`, `@iap/parser`, `@iap/validator`, `@iap/policy`, `@iap/sdk` (+ ajv/yaml). Never on `@iap/provider-sdk`, `@iap/planner`, or any execution surface. A test asserts the dependency set and that no module imports them.
11. **Exit criterion 1 is proven through the benchmark**: every official example gets a natural-language benchmark case whose confirmed pipeline output is a _valid_ document (full ch. 8 pipeline), with semantic-equivalence scoring against the example where the rules adapter's vocabulary allows and fixture-recorded proposals where it does not — both paths cross the identical gate. Metrics per roadmap §3.7 (extraction correctness, clarification precision, unsupported-feature detection, false assumptions, validity, equivalence, deterministic re-serialization).
12. **`iap create` (M5.3, ch. 22)**: interactive flow — NL request → adapter → clarifications as prompts → preview diff → explicit confirmation → serialize; non-interactive mode takes `--answers`/`--yes-defaults` for automation and tests; `--adapter` selects fixture/rules (default rules). Exit codes per ch. 22; the command writes only through the operation gate.

## Exit criteria mapping

NL→valid IaP for all official examples → benchmark suite (decision 11). No unvalidated LLM YAML to disk → OP-1 gate test: no exported byte-producing path bypasses `apply` (decision 4). Low-confidence requires confirmation → OP-3 tests (threshold + assumptions + clarifications). Same confirmed operations ⇒ same IaP regardless of model → OP-2 replay + two-adapter test (decision 5). Compiler cannot call deployment APIs → dependency-set test (decision 10). Every generated field has provenance → OP-4 per-field records citing operation ids (decision 6).
